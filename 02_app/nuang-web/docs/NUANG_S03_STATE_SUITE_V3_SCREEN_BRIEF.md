## 0. 한 문장 목표

검사가 잠시 멈추거나 마지막 단계에 도달해도 화면 구조와 신뢰를 흔들지 않고, 사용자가 현재 상태와 다음 행동을 즉시 이해하게 한다.

## 1. 공통 shell 불변 조건

- app bar, 연속 progress, 본문 stage, 하단 dock의 기준 위치를 상태 전환 전후 유지한다.
- 상태별 본문은 같은 최소 stage 높이를 사용해 phone shell이 위아래로 튀지 않는다.
- 오류 안내는 본문을 밀어내지 않고 dock 위 overlay card로 나타난다.
- 로딩·결과 준비에서는 runner dock과 같은 높이의 status dock을 사용한다.
- 정상 상태에서는 저장 성공·자동 저장 문구를 표시하지 않는다.
- 운영체제 기본 이모지와 반복 spinner를 사용하지 않는다.

## 2. 상태별 화면 계약

| 상태                          | 상단                                         | 본문                                                                      | 하단                                                                     | 주 행동                                 |
| ----------------------------- | -------------------------------------------- | ------------------------------------------------------------------------- | ------------------------------------------------------------------------ | --------------------------------------- |
| 초기 로딩                     | `빠른 코어` / `검사를 준비하는 중` / `— / —` | 뉴앙 5-node motif, `검사를 준비하고 있어요`, `곧 첫 질문을 보여드릴게요.` | `준비가 끝나면 바로 시작할게요.`                                         | 준비 완료 시 첫 문항으로 자동 전환      |
| 원격 저장 실패·로컬 보관 성공 | 현재 문항과 진행률 유지                      | 선택 응답 유지                                                            | dock 위 `응답을 바로 저장하지 못했어요`, `다시 저장`; `다음`은 활성 유지 | 재시도 또는 다음 문항                   |
| 마지막 문항                   | `20 / 20` 예시와 100% 진행선                 | `마지막 질문` badge, 상황·질문·1~5                                        | `결과 보기`                                                              | 마지막 응답 확정 후 완료 요청           |
| 결과 준비                     | `빠른 코어 완료` / `결과를 만드는 중`        | `응답 완료`, `내 성향을 정리하고 있어요`, 5-node motif                    | `결과가 준비되면 바로 보여드릴게요.`                                     | 결과 준비 완료 시 간단 결과로 자동 이동 |

빠른·정밀 문항 수는 실제 `measurement_release`에서 읽으며 `20 / 20`은 화면 검토용 예시다.

## 3. 로딩 상태

- 진행률을 모르면 가짜 퍼센트나 무한 회전 spinner를 보여주지 않는다.
- 5-node motif는 진입 때 한 번만 제자리로 모이고 멈춘다.
- app shell을 먼저 보여 화면 전환이 끝났음을 즉시 알린다.
- 3초를 넘기면 같은 위치에 `조금만 더 기다려주세요.`를 추가할 수 있다.
- 10초를 넘기거나 실패가 확인되면 `다시 불러오기`와 안전한 이전 화면 이동을 제공한다.
- 로딩 중에는 응답 입력·이전·완료를 비활성화한다.

## 4. 저장 실패 상태

### 4.1 이번 시안: 로컬 보관 성공

- 문항·선택 응답·진행률·dock 위치를 바꾸지 않는다.
- dock 위 overlay card에 다음을 표시한다.
  - `응답을 바로 저장하지 못했어요`
  - `이 기기에 임시 보관했어요. 연결되면 다시 저장할 수 있어요.`
  - `다시 저장`
- 로컬에 응답이 확인되어 있으므로 사용자는 다음 문항으로 계속 진행할 수 있다.
- 재시도 성공 시 `다시 저장했어요`를 짧게 확인하고 card를 닫는다.

### 4.2 별도 필수 상태: 로컬 보관도 실패

- `이 기기에 임시 보관했어요`라고 거짓 안내하지 않는다.
- `응답을 안전하게 보관하지 못했어요`와 재시도를 blocking 상태로 제공한다.
- 보관 성공 전에는 다음 문항·완료 요청을 진행하지 않는다.

### 4.3 구현 판정값

화면은 최소 다음 상태를 구분해야 한다.

| 필드                    | 값 예시                                  | 화면 사용                                   |
| ----------------------- | ---------------------------------------- | ------------------------------------------- |
| `local_persist_status`  | `pending · saved · failed`               | 계속 진행 가능 여부                         |
| `remote_sync_status`    | `idle · syncing · synced · failed`       | overlay error와 재시도                      |
| `completion_status`     | `idle · submitting · completed · failed` | 결과 준비·중복 완료 방지                    |
| `completion_request_id` | idempotency key                          | 완료 중복 요청 방지                         |
| `last_saved_at`         | device timestamp                         | 복구 판단용이며 정상 화면에는 노출하지 않음 |

필드명과 저장 위치는 `RBL-07` versioned migration에서 확정한다.

## 5. 마지막 문항

- progress는 한 개의 연결된 선으로 100%를 표시한다.
- `마지막 질문` badge는 축하 장식이 아니라 현재 위치를 알려주는 정보로 사용한다.
- 답을 선택하기 전 `결과 보기`는 비활성화한다.
- 이전 문항으로 돌아가도 선택 응답과 판단 어려움 이유를 복원한다.
- `결과 보기` 연속 탭은 첫 요청만 인정하고 즉시 결과 준비 상태로 전환한다.
- confetti, 점수 preview, 과장된 분석 파형은 사용하지 않는다.

## 6. 결과 준비 상태

- `S05`, 내부 route, 점수 계산식, 임의 퍼센트를 사용자에게 노출하지 않는다.
- 5-node motif가 한 번 모이는 움직임으로 여러 응답을 종합한다는 인상만 전달한다.
- 완료 응답이 오면 간단 결과 화면으로 자동 이동한다.
- 10초 이상 지연되면 `결과를 만드는 데 시간이 걸리고 있어요`와 `다시 시도`를 제공한다.
- 완료 요청은 idempotent해야 하며 뒤로가기·재진입이 결과를 중복 생성하지 않는다.
- 앱을 벗어날 수 있는 정책을 열 경우 결과 요청·로컬 응답을 먼저 보존하고 재진입 경로를 제공한다.

## 7. 시각·모션 규칙

- 승인된 soft gradient, orbit line, rounded response row, blur dock을 그대로 사용한다.
- 상태 진입은 본문 8px settle, node 5개의 320~480ms 순차 settle, progress 너비 변화로 구성한다.
- 모든 동작은 한 번만 실행하고 멈춘다.
- 저장 실패 card는 240ms 이내로 dock 위에 올라오며 본문을 재배치하지 않는다.
- 기능 아이콘은 같은 open-source line icon set을 사용한다.
- 브랜드 의미가 필요한 캐릭터·보상 그래픽은 뉴앙 전용 asset만 사용한다.
- `prefers-reduced-motion`에서는 모든 위치 이동을 제거한다.

## 8. 접근성·복구

- loading·preparing 문구는 `aria-live=polite`, 저장 실패는 `role=alert`로 알린다.
- progress는 로딩 중 미확정 값과 문항 진행 중 확정 값을 구분한다.
- 상태 전환 뒤 의미 있는 제목으로 읽기 흐름을 이동하되 native tab order를 유지한다.
- 오류 card의 재시도는 44px 이상이며 상태 설명을 색만으로 구분하지 않는다.
- 320px와 200% 확대에서 오류 card·dock·본문이 겹치지 않는다.
- 네 상태 모두 light·dark theme에서 같은 의미 대비를 유지한다.

## 9. 이번 승인 범위

1. 로딩의 무한 spinner 없는 5-node settle 방식
2. 로컬 보관 성공 시 non-blocking 저장 실패 card
3. 마지막 문항의 절제된 완료감과 `결과 보기`
4. 결과 준비의 무가짜 퍼센트·자동 이동 원칙
5. 네 상태가 같은 높이와 같은 visual language를 쓰는 구조

승인 뒤 저장 실패의 로컬 보관 실패·장기 지연 상태를 추가하고, 430px·520px 반응형과 실제 동작 prototype으로 확장한다.
