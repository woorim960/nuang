# NUANG 성향지도·관계 비교 지식 아키텍처

작성일: 2026-07-20 KST  
계약: `nuang-trait-map-knowledge.v0.1`, `nuang-comparison-knowledge.v0.1`

## 1. 목표

뉴앙의 성향지도는 재미있는 유형 소개문을 모아 둔 페이지가 아니다. 사용자가 자기 생각과 실제 반응을 이해하고, 가족·친구·연인·마음 가는 사람·동료와의 관계를 더 안전하게 조율하도록 돕는 기준 지식이다.

따라서 32개 코드는 한 개씩 독립된 연구·집필 단위로 다룬다. 코드 하나의 승인 원문은 공백 제외 약 5만 자를 목표로 하고, 원문에서 검토된 주장만 결과 리포트·공개 프로필·1:1 비교에 재사용한다.

이 문서는 아직 32개 성향 연구가 완료되었다는 뜻이 아니다. 긴 원문을 빠르게 채우기보다, 앞으로 작성할 내용이 근거·버전·검토 상태를 잃지 않도록 데이터 구조와 승인 절차를 먼저 고정한다.

## 2. 규모와 작성 전략

- 대표 성향지도: `32개 × 약 50,000자 = 약 160만 자`
- 방향이 있는 두 사람 조합: `32 × 32 = 1,024개`
- 순서를 무시한 조합(동일 코드 포함): `528개`
- 6개 기본 관계 맥락을 적용한 방향성 조합: `1,024 × 6 = 6,144개`

6,144개의 완성 리포트를 따로 집필하면 같은 의미가 반복되고, 수정 시 서로 모순될 위험이 커진다. 뉴앙은 아래처럼 작은 근거 단위를 조합한다.

1. 연구·측정 근거
2. 5개 코드 자리와 세부 신호 정의
3. 32개 대표 성향의 통합 해석
4. 사람·상황별 생각과 실제 반응
5. 두 성향이 만날 때의 강점·마찰·오해·대화 규칙
6. 화면별 길이에 맞춘 표현

## 3. 지식 데이터 계층

### L0. Evidence

모든 중요한 주장의 출처다.

- 메타분석·체계적 문헌고찰
- 동료평가를 거친 원 연구와 도구 타당화 연구
- 공식 검사 매뉴얼과 측정 연구
- 뉴앙 문항의 신뢰도·구조·준거 분석
- 심리학·심리측정 전문가 검토
- 출시 후 익명화된 사용자 이해도·유용성 관찰

출처가 있다는 이유만으로 뉴앙 문장에 바로 쓰지 않는다. 연구 대상, 측정 구성개념, 효과의 크기, 재현 가능성, 문화·연령 범위, 뉴앙 축과의 대응 수준을 함께 기록한다.

### L1. Claim

사용자에게 전달할 수 있는 가장 작은 의미 단위다. 각 claim은 다음을 가진다.

- `claimId`: 변경되지 않는 식별자
- `contentKey`: 의미 중복을 막는 키
- `claimKind`: 처음 드는 생각, 실제 나타나는 반응, 강점, 마찰, 오해 가능성, 편한 지원, 대화 질문 등
- `contexts`: 전체, 가족, 친구, 연인, 마음 가는 사람, 업무
- `evidenceRefs`: 근거 연결
- `confidence`: 이론 기반, 예비, 지지됨, 검증됨
- `shortCopy`, `longCopy`: 같은 의미의 화면 길이별 표현

한 claim을 여러 화면에서 쓸 수는 있지만, 한 리포트 안에서 같은 `contentKey`를 두 번 보여주지 않는다.

### L2. 32개 성향지도 원문

각 코드의 원문은 claim을 단순히 이어 붙인 문서가 아니다. 다섯 코드 자리가 한 사람 안에서 어떻게 함께 나타날 수 있는지 통합해 설명한다. 축을 독립적으로 직역한 문장만으로 역할형 성향을 단정하지 않는다.

### L3. 관계·상황 모듈

같은 코드라도 가족, 친구, 연인, 마음 가는 사람, 일에서 중요하게 느끼는 신호와 행동은 달라질 수 있다. 각 모듈은 다음을 분리한다.

- 처음 드는 생각
- 실제 나타나는 반응
- 편안할 때와 부담이 클 때
- 상대가 오해하기 쉬운 지점
- 상대에게 바라는 도움
- 스스로 선택할 수 있는 조율 방법

### L4. Pair interaction rule

`viewerCode → targetCode`의 방향을 보존한다. 내가 상대를 이해하는 설명과 상대가 나를 이해하는 설명은 같지 않을 수 있기 때문이다.

규칙은 새 성격 사실을 만들어내지 않는다. 승인된 두 성향지도 claim과 관계 모듈을 근거로 다음을 연결한다.

- 자연스럽게 맞을 수 있는 지점
- 같은 방향이지만 필요한 정도가 다른 지점
- 다르게 느끼거나 행동할 수 있는 지점
- 서로의 의도를 오해하기 쉬운 순간
- 바로 써볼 수 있는 질문과 조율 방법

### L5. Report composition

화면은 깊이를 단계적으로 연다.

1. 한눈에 보기: 항목 이름만
2. 5개 코드 자리: 수치, 각자 편한 방향, 세부 신호
3. 관계 장면: 그 맥락에서만 필요한 오해 가능성과 질문
4. 더 깊이 보기: 해당 성향지도 원문으로 이동

합성기는 `claimId`와 `contentKey`를 추적하여 한 화면에서 같은 의미를 다시 보여주지 않는다.

## 4. 코드 한 개당 약 5만 자 집필 골격

분량은 품질의 증거가 아니라 누락 방지용 목표다. 필요 없는 반복으로 5만 자를 채우지 않는다.

| 영역 | 목표 분량 | 핵심 내용 |
|---|---:|---|
| 한눈에 이해하기 | 2,000~3,000자 | 이름의 의미, 핵심 동기, 단정하지 않는 읽는 법 |
| 5개 코드 자리의 통합 | 6,000~8,000자 | 각 자리와 자리 간 상호작용, 가까운 경계 |
| 처음 드는 생각과 실제 반응 | 4,000~5,000자 | 내적 선호와 행동 조절을 구분 |
| 일상과 의사결정 | 3,000~4,000자 | 선택, 휴식, 일정, 변화, 갈등 |
| 가족 | 3,000~4,000자 | 돌봄, 독립, 연락, 역할, 갈등 회복 |
| 친구 | 3,000~4,000자 | 친밀감, 약속, 대화, 거리 조절 |
| 연인·마음 가는 사람 | 5,000~6,000자 | 표현, 연락, 기대, 오해, 회복; 연애 전용으로 단정 금지 |
| 업무·학업 | 4,000~5,000자 | 협업, 피드백, 계획, 리더십, 집중 |
| 스트레스와 회복 | 3,000~4,000자 | 부담 상황의 경향, 회복 조건; 임상 진단 금지 |
| 강점과 성장 선택지 | 3,000~4,000자 | 자원, 과사용 위험, 작은 실험 |
| 자주 생기는 오해 | 2,000~3,000자 | 고정관념과 반례 |
| 대화 가이드 | 3,000~4,000자 | 상황별 질문과 피해야 할 단정 |
| 근거·한계 | 2,000~3,000자 | 어떤 근거를 썼고 무엇을 말할 수 없는지 |

## 5. 코드 한 개의 제작·승인 절차

한 번에 32개를 생성하지 않는다. 코드 하나가 아래 문을 모두 통과한 뒤 다음 코드로 간다.

1. 구성개념 명세: 다섯 자리와 세부 신호가 이 코드에서 어떻게 결합되는지 가설을 적는다.
2. 문헌 검토: 검색식, 포함·제외 기준, 핵심 연구, 반대 결과와 한계를 기록한다.
3. 근거표 작성: 모든 주요 claim을 evidence에 연결한다.
4. 초안 집필: 사람·상황별 처음 드는 생각과 실제 반응을 분리한다.
5. 심리학 검토: 과도한 일반화, 임상적 오해, 낙인 위험을 점검한다.
6. 심리측정 검토: 뉴앙 점수로 말할 수 있는 범위를 넘지 않는지 확인한다.
7. 쉬운 문장 검토: 고등학생도 한 번에 이해할 수 있는 표현인지 확인한다.
8. 반례·모순 감사: 같은 코드 안과 기존 코드 사이의 충돌을 찾는다.
9. 중복 감사: `contentKey`와 의미 유사도를 함께 확인한다.
10. 제품 검수: 지도, 개인 리포트, 비교 리포트에서 단계적으로 읽히는지 확인한다.
11. 사용자 인지 인터뷰와 파일럿: 사용자가 자신을 단정당한다고 느끼지 않는지 확인한다.
12. 승인·버전 발행: 여섯 전문 검토 상태가 모두 `passed`일 때만 공개한다.

## 6. ‘잘 맞는 유형’과 ‘힘들 수 있는 유형’ 제공 원칙

사용자 니즈는 충족하되, 관계의 성공을 코드만으로 예측하지 않는다.

제공:

- 편하게 맞을 가능성이 있는 방식과 그 조건
- 힘들 수 있는 장면과 그 이유
- 같은 코드여도 정도 차이로 생길 수 있는 마찰
- 가족·친구·연인·업무별로 달라지는 상호작용
- 서로에게 유용한 질문과 행동 선택지
- 실제 두 사람 점수를 반영한 개인화

제공하지 않음:

- 궁합 점수와 순위
- 관계 성공·이별 확률
- 좋은 유형·나쁜 유형
- 위험 인물 판정
- 비공개 응답이나 민감 정보의 추정

사용자에게는 `잘 맞는다`보다 `이 조건에서 편하게 맞을 수 있어요`, `이 장면에서는 힘들 수 있어요`처럼 상황과 조건을 함께 말한다.

## 7. 비교 리포트 중복 방지 규칙

- 동일 코드 방향은 두 사람 설명을 두 번 쓰지 않고 `두 사람에게 공통으로 나타난 방향`으로 한 번만 보여준다.
- 같은 방향이지만 점수 차이가 큰 경우 공통 설명 뒤에 `나타나는 정도가 다름`만 추가한다.
- 5개 코드 자리에서 보여준 성향 설명은 관계 장면에서 반복하지 않는다.
- 관계 장면은 `상황 → 오해 가능성 → 확인 질문`만 담당한다.
- 한눈에 보기에서는 상세 문장을 보여주지 않는다.
- 한 질문을 관계 장면과 페이지 하단에 반복하지 않는다.
- 렌더 직전 `claimId`, `contentKey` 중복을 모두 차단한다.

## 8. 버전과 변경 관리

- 성향지도 원문, claim, evidence, 상호작용 규칙, 사용자 문구의 버전을 분리한다.
- 연구 근거가 바뀌면 영향을 받는 claim과 1:1 비교 조합을 역추적한다.
- 과거 리포트는 생성 당시 지도·규칙 버전을 보존한다.
- 중대한 오류가 확인되면 영향 범위를 기록하고 관련 리포트를 재생성한다.
- 사용자 피드백은 근거가 아니라 개선 후보로 저장하며, 분석과 검토 없이 일반화하지 않는다.

## 9. 구현 순서

1. 현재 비교 화면의 동일 의미 반복 제거
2. 지식 계약과 중복 검증 테스트 도입
3. 32개 중 첫 코드의 구성개념 명세·연구 프로토콜 승인
4. 첫 코드 원문과 claim/evidence 데이터 제작
5. 첫 코드의 성향지도 상세 화면 연결
6. 두 번째 코드 제작 후 최초 pair interaction rule 검증
7. 승인된 코드가 늘어날 때만 비교 리포트 지식 범위를 확장
8. 32개 완료 후 전체 1,024개 방향성 조합의 누락·모순·중복 자동 감사

다음 콘텐츠 단계는 첫 코드를 임의로 대량 생성하는 것이 아니라, `첫 코드 연구 프로토콜과 5만 자 목차·근거표`를 먼저 승인받는 것이다.
