## 0. 화면 ID 정리

`S04-R`은 측정 재기준화 뒤의 압축된 첫 가치 여정에서 사용하는 빠른 결과 화면 ID다. 기존 Master Blueprint의 S04 중단·이어하기와 S05 완료 전환은 S03-R 상태·복구 suite로 흡수됐고, 기존 S06 빠른 결과 역할을 S04-R이 이어받는다.

## 1. 목표

빠른 검사를 끝낸 사용자가 첫 화면에서 아래 세 가지를 이해하게 한다.

1. 지금 보이는 5글자는 짧은 검사에서 먼저 보인 `예비 뉴앙 코드`다.
2. 코드는 능력·도덕성·진단이 아니라 다섯 생활 영역에서 반복된 방향을 기억하기 쉽게 요약한 것이다.
3. 더 다양한 상황을 확인하는 정밀 검사와, 결과를 보관한 채 홈으로 돌아가는 두 경로를 모두 선택할 수 있다.

## 2. 첫 읽기 정보 순서

```text
결과 리포트
→ 빠른 결과·예비 코드
→ 코드와 가장 분명한 한 문장
→ 다섯 코드 자리 탐색
→ 이번 답에서 가장 분명한 두 모습
→ 비슷한 자리 또는 응답 부족 안내
→ 왜 예비 결과인지
→ 정밀 검사 / 홈
```

빠른 결과에서는 원점수, 백분율, 정확도 수치, 순위, 레이더 차트를 보여주지 않는다. 문항 수가 적은 상태에서 숫자와 그래프가 측정 정밀도를 과장하기 쉽기 때문이다.

## 3. 기본 결과 계약

- `예비 뉴앙 코드`를 5글자 전체로 같은 크기와 간격으로 보여준다.
- v1 시뮬레이션의 `ERGKC`는 레이아웃 검토용 연구 fixture다. 승인된 `measurement_release_id`와 scoring release가 생기기 전에는 운영 결과로 사용하지 않는다.
- 첫 문장은 다섯 자리를 모두 길게 나열하지 않고 이번 답에서 가장 반복된 두 방향만 생활 언어로 요약한다.
- 사용자가 글자를 누르면 한 번에 한 자리의 `영역 · 쉬운 방향명 · 짧은 설명`만 연다.
- 상세 facet, 과정 화살표, `처음 드는 생각 → 실제 나타나는 반응`, 본인 전용 상세 신호는 빠른 결과에서 발급하지 않는다.
- 결과 캐릭터는 감정적 초점으로 한 번 등장하고, 코드 reveal은 한 번만 움직인다. 결과를 읽는 동안 의미 없는 반복 motion을 두지 않는다.

## 4. 다섯 자리 고객 문구

| 자리 | 글자 | 영역                  | 빠른 결과 방향명            | 빠른 결과 설명 원칙                                                       |
| ---- | ---- | --------------------- | --------------------------- | ------------------------------------------------------------------------- |
| 1    | E    | 사람 사이 에너지      | 함께하며 먼저 표현          | 교류 활력과 먼저 표현하는 경향만 말하고 인기·사교 능력을 추론하지 않는다. |
| 2    | R    | 생각과 탐색           | 익숙하고 구체적인 것에 관심 | 현실 능력이나 창의성의 높고 낮음으로 설명하지 않는다.                     |
| 3    | G    | 관계에서 먼저 보는 것 | 원인과 해결이 먼저 보임     | 공감 능력·착함·존중 수준을 추론하지 않는다.                               |
| 4    | K    | 일상을 꾸리는 방식    | 꾸준히 이어감               | 성과·책임감·J/P·유연성을 추론하지 않는다.                                 |
| 5    | C    | 걱정과 감정 반응      | 비교적 차분하게 반응        | 건강·회복력·정신질환을 추론하지 않는다.                                   |

반대 방향 글자의 의미는 같은 자리에서 함께 설명할 수 있지만, 기본 첫 읽기에서는 선택된 글자의 쉬운 의미를 우선한다.

## 5. 비슷한 자리 상태

- 고객 표현은 `경계`, `혼합형`, `확신도 낮음`보다 `두 방향이 비슷하게 나타났어요`를 우선한다.
- 예비 코드는 현재 조금 더 가까운 글자를 유지하되, 해당 자리의 두 방향 차이가 크지 않다는 사실을 바로 붙인다.
- 비슷함을 오류나 실패로 표현하지 않는다.
- `정밀 검사에서 여러 상황을 더 살펴보면 더 선명해질 수 있어요`로 다음 행동의 이유를 설명한다.
- 실제 코드 발급 가능 여부와 비슷함 표시 기준은 M08 scoring release에서 확정한다. UI가 임의 임계값을 계산하지 않는다.

## 6. 응답 부족 상태

- `판단하기 어려워요`가 많거나 유효 증거가 부족하면 빈 글자, 중립 글자, 임의 대체 점수로 5글자를 완성하지 않는다.
- 제목은 `아직 코드로 묶기 어려워요`를 사용한다.
- 답한 내용이 사라진 것이 아니라 보관됐다는 사실을 먼저 알린다.
- 판단하기 어려웠던 답을 고객의 의사에 반해 3점이나 중간 점수로 바꾸지 않는다.
- 주 행동은 `정밀 검사로 확인하기`, 보조 행동은 `홈으로 돌아가기`다.
- 정밀 검사에서도 판단하기 어려운 문항을 그대로 선택할 수 있어야 한다.

## 7. 정밀 검사와 홈 복귀

### 정밀 검사

- 주 CTA는 `정밀 검사로 더 알아보기`다.
- 응답 부족 상태에서는 `정밀 검사로 확인하기`로 바꾼다.
- 다음 화면은 빠른 결과를 부정하지 않고 `더 많은 상황을 통해 코드 자리와 세부 성향을 구체화한다`고 설명한다.
- 정밀 검사에서 빠른 응답을 재사용할 수 있는지는 같은 measurement release·문항 버전·채점 계약에 따라 결정한다. UI가 재사용을 약속하지 않는다.

### 홈

- 상단과 하단 모두 `홈`이라는 목적지를 텍스트로 보여준다.
- 홈으로 이동하기 전에 결과 저장 위치를 고객에게 설명하지 않는다. 결과는 하나의 `내 결과`로 다시 열려야 한다.
- 홈에는 `빠른 결과 완료` 상태와 가장 적절한 다음 행동 하나를 보여준다. 정밀 검사를 시작하지 않았다고 실패처럼 표현하지 않는다.

## 8. 신뢰 문구

`왜 예비 결과인가요?`를 사용자가 열 수 있게 제공한다.

표준 문구:

> 빠른 결과는 짧은 문항에서 보인 첫 방향이에요. 능력·도덕성·건강을 판단하지 않으며, 더 다양한 상황에 답하면 코드와 설명이 달라질 수 있어요.

금지:

- `당신은 확실히 ERGKC입니다`
- `정확도 95%`
- `과학적으로 증명된 성격`
- `K가 낮으면 게으름`
- `Q는 예민하거나 불안함`
- `G는 공감이 부족함`

## 9. 데이터 계약

| 필드                      | 역할                                                   |
| ------------------------- | ------------------------------------------------------ |
| `result_id`               | 같은 결과를 홈·마이·리포트에서 다시 여는 불투명 식별자 |
| `assessment_attempt_id`   | 완료 응답과 결과를 연결하는 식별자                     |
| `measurement_release_id`  | 문항·채점·카피 release 일치 확인                       |
| `result_resolution`       | `QUICK` 또는 `FULL`                                    |
| `code_status`             | `PROVISIONAL_AVAILABLE · INSUFFICIENT_EVIDENCE`        |
| `provisional_code`        | 발급 가능할 때만 저장하는 5글자 코드                   |
| `position_summaries`      | 자리별 글자, 쉬운 방향명, 허용된 짧은 설명             |
| `position_clarity_state`  | `CLEAR · NEAR_BALANCE · INSUFFICIENT`                  |
| `headline_signal_ids`     | 첫 화면에 보여줄 검증된 최대 두 개의 요약 신호         |
| `recommended_next_action` | 정밀 검사·홈 중 상태 기반 주 행동                      |
| `scored_at`               | 결과 생성 시점                                         |

`position_summaries`에는 빠른 결과에서 발급하지 않는 facet 점수, 과정 코드, 직접 응답, 이유 probe를 넣지 않는다. 원점수와 내부 error·scoring key는 고객 화면과 공유 payload에 노출하지 않는다.

## 10. 상태와 이동 계약

```text
result_generation_status=completed
├─ code_status=PROVISIONAL_AVAILABLE
│  ├─ 모든 자리 CLEAR → 기본 결과
│  └─ 한 자리 이상 NEAR_BALANCE → 비슷한 자리 안내 포함
└─ code_status=INSUFFICIENT_EVIDENCE → 코드 미발급·정밀 검사 안내
```

- 결과 route 재진입은 같은 `result_id`를 복원하고 새 결과를 만들지 않는다.
- 정밀 검사 CTA 연속 탭은 한 번의 navigation만 수행한다.
- 홈 이동은 결과 보관 완료 후 수행한다.
- 오프라인에서 결과가 이미 로컬에 완성됐다면 같은 읽기 화면을 제공하고, 계정 연결·공유는 필요한 시점에만 별도 처리한다.

## 11. 접근성·반응형·motion

- 390px light와 320px dark에서 코드, 다섯 자리 버튼, 카드, CTA가 수평 overflow 없이 표시돼야 한다.
- 코드 자리는 글자만 보여도 버튼의 접근성 이름에 순서·영역·방향을 포함한다.
- 선택된 자리는 `aria-pressed`, 바뀐 설명은 `aria-live=polite`로 전달한다.
- 색만으로 비슷한 자리와 응답 부족을 구분하지 않는다.
- `prefers-reduced-motion`에서는 코드와 캐릭터 reveal을 즉시 완료한다.
- 주 CTA와 홈 버튼은 모두 손가락으로 누르기 쉬운 전체 폭 행동 영역을 사용한다.

## 12. 제품 지표

- 빠른 결과 첫 코드 노출률
- 코드 자리 1개 이상 열어본 비율
- `왜 예비 결과인가요?` 열기 비율
- 정밀 검사 CTA 선택률
- 홈 복귀율
- 응답 부족 상태 뒤 정밀 검사 진입률
- 결과 화면 이탈·재진입 성공률

지표는 체류 시간을 늘리기 위한 과장에 사용하지 않는다. `정밀 검사 CTA 선택률`보다 결과 이해도·재진입 성공·신뢰 문구 이해를 함께 본다.
