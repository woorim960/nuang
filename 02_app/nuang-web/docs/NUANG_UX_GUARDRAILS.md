# NUANG UX Guardrails

작성일: 2026-07-09 KST

이 문서는 뉴앙 화면을 만들 때 반드시 지켜야 하는 UI/UX 규칙이다. 구현 전후에 이 문서와 실제 화면을 대조한다.

## 기본 방향

뉴앙은 모바일 우선 SNS 앱처럼 느껴져야 한다. AI가 만든 임시 데모처럼 보이면 안 된다.

최상위 UX 원칙:

- `/feed`에서 잡은 인스타그램/스레드형 경험을 전 앱의 기준으로 삼는다.
- 사용자는 기능 설명을 읽기보다 자연스럽게 스크롤하고, 보고, 반응하고, 다음 행동으로 이어져야 한다.
- 각 메뉴는 자기 역할을 지키되, 카드와 문단을 쌓는 관리툴 화면이 아니라 실제 SNS 앱의 밀도와 리듬을 가져야 한다.
- 운영/기획 관점에서는 고객 만족, 흥미, 재방문, 체류 시간을 높이는 방향을 우선한다.
- 흥미를 높이더라도 민감 정보 보호, 진단/치료 오인 방지, 비교 기능의 비낙인 원칙은 절대 약화하지 않는다.

필수:

- Tailwind 기반 구현
- 모바일 우선 레이아웃
- 여백과 타이포 중심
- 실제 서비스처럼 보이는 밀도와 상태
- 감각적이지만 대중적인 직관성
- lucide-react 등 검증된 오픈소스 아이콘 사용
- 필요한 이미지 자산은 생성 이미지 또는 실제 자산 사용

금지:

- 이모지와 데스크탑 이모티콘
- 과한 그라데이션
- 한 화면에 너무 많은 선택지
- 바텀 nav와 같은 이동 경로 반복 카드
- 피드와 커뮤니티의 중복 표기
- 카드 안의 카드
- 기능 설명 문구로 화면을 채우는 방식
- 아이콘을 컬러 박스로 감싸 기능 목록처럼 보이게 하는 방식
- 둥근 외곽선 사각형 태그를 반복해서 AI 데모처럼 보이게 하는 방식
- CSS만으로 캐릭터나 프로필 이미지를 직접 그리는 방식
- 대표 성향 코드 외의 내부 영어 축약어를 사용자 화면에 노출하는 방식

## 선택지 밀도

한 화면에서 사용자가 즉시 선택해야 하는 행동은 적어야 한다.

기준:

- 화면 상단 primary action은 1개를 기본값으로 한다.
- 같은 섹션 안에서 primary action 1개, secondary action 1개를 넘기지 않는다.
- 3개 이상의 선택지가 필요하면 탭, segmented control, 접힘 영역, 단계형 흐름 중 하나로 정리한다.
- 사용자가 매번 읽어야 하는 설명문은 줄이고, 상태와 행동을 먼저 보여준다.

## 하단 메뉴 중복 방지

하단 메뉴는 이미 최상위 이동 수단이다. 각 화면에서 모든 하단 메뉴로 다시 이동시키는 카드 목록을 만들지 않는다.

허용:

- 현재 작업을 끝낸 뒤 자연스럽게 필요한 다음 행동 1개를 제안
- 결과 리포트에서 성향지도 보기처럼 맥락상 필요한 이동
- 홈에서 피드 preview를 보고 `/feed` 더보기로 이동

금지:

- 홈, 검사, 성향지도, 함께, 마이 전체를 카드로 반복
- 함께 탭에서 피드로 이동시키는 route card
- 검사 탭에서 함께/마이/홈 이동 카드를 과하게 배치

## 메뉴별 화면 규칙

홈:

- 오늘의 시작점이다.
- 첫 화면에서 검사 이어하기, 내 성향 요약, 피드 preview 중 가장 강한 다음 행동 하나를 보여준다.
- 피드 preview는 2~3개 카드만 보여주고 더보기는 `/feed`로 보낸다.

검사:

- 검사 시작과 이어하기가 가장 쉬워야 한다.
- 코어 검사와 별난 성향 연구소를 분리한다.
- 민감/치료 주제는 검사처럼 열지 말고 도움 허브와 안전 안내로 연결한다.

성향지도:

- 뉴앙 코드와 32개 코드 탐색이 핵심이다.
- 개인 결과의 시각화는 `코드 지도`와 `세부 신호`로 설명한다.
- 결과가 없을 때는 검사 시작 하나만 제안한다.
- 수치가 많을 때도 먼저 시각 지표, 그다음 짧은 해석, 마지막 세부 목록 순서로 둔다.

함께:

- 나와 비교하기와 그룹 검사가 핵심이다.
- 피드, 커뮤니티, 게시글 composer를 넣지 않는다.
- 비교 리포트는 두 사람의 뉴앙 코드, 같은 코드 자리, 다른 코드 자리, 오해 장면, 대화 가이드, 조율 가이드 중심으로 구성한다.

마이:

- 계정 연결, 공개 범위, 데이터 관리가 핵심이다.
- 같은 결과의 여러 저장 기록은 한 행으로 합치고 저장 위치는 표시하지 않는다.
- 사용자는 모든 결과를 같은 결과 리포트로 다시 열 수 있어야 한다.
- 결과 삭제는 하나의 동작으로 제공하고 연결된 공유·공개·비교 기록은 내부에서 함께 정리한다.
- `계정 결과`, `로컬 결과`, `기기 사본`, `계정 저장`을 사용자-facing 상태나 CTA로 쓰지 않는다.
- 공개 범위 기본값을 분명히 보여준다.
- 민감 항목은 기본 비공개이며 공개 UI를 열지 않는다.

피드:

- 하단 메뉴에는 없는 별도 페이지다.
- 홈 preview의 더보기 목적지다.
- 초기에는 오늘의 기분, 오늘의 질문, 내 성향 기반 카드, 공식 주제 카드 중심으로 구성한다.
- 인스타그램/스레드처럼 단일 스트림, 스토리 영역, 텍스트 피드, 반응 버튼의 익숙한 리듬을 유지한다.
- 사용자에게 내부 분류, 안전 등급, 운영 상태를 직접 노출하지 않는다.
- 다른 메뉴를 개선할 때도 피드의 여백, 타이포, 경계선, 단순한 액션 배치를 우선 참고한다.

## 시각 규칙

- 색상은 보라 계열만으로 지배되지 않게 한다.
- 과한 glow, orb, blur 장식을 쓰지 않는다.
- 카드 radius는 8px 이하를 기본값으로 한다.
- 버튼과 고정 포맷 요소는 stable dimension을 가진다.
- 텍스트는 버튼과 카드 안에서 줄바꿈 가능해야 하고 겹치면 안 된다.
- 큰 제목은 진짜 hero 또는 화면 제목에만 쓴다.

## 아이콘과 이미지

아이콘:

- 메뉴, 검색, 공유, 저장, 뒤로가기 등은 lucide-react를 우선 사용한다.
- 이모지는 쓰지 않는다.

이미지:

- 캐릭터, 앱 아이콘, 프로필 이미지가 필요하면 생성 이미지 또는 실제 이미지 asset을 사용한다.
- CSS shape로 캐릭터를 만들지 않는다.
- 이미지는 모바일에서 잘리지 않고 핵심 피사체가 보여야 한다.

## 구현 전 체크

새 화면 또는 큰 UI 변경 전 확인:

1. 이 화면의 메뉴 역할이 `NUANG_PRODUCT_CANON.md`와 일치하는가.
2. 사용자가 바로 누를 primary action이 하나인가.
3. 하단 메뉴와 중복되는 이동 카드가 없는가.
4. 피드가 함께 탭에 섞이지 않았는가.
5. 이모지, 과한 그라데이션, 데모 같은 copy가 없는가.
6. 모바일 360px 폭에서 텍스트가 겹치지 않는가.

## 구현 후 체크

가능하면 아래를 확인한다.

```bash
npm run harness:check
npm run typecheck
npm run lint
npm run test
```

큰 UI 변경 또는 출시 후보에서는 build까지 실행한다.

```bash
npm run qa:mvp
```
