## 0. 한 문장 목표

사용자가 앱을 처음 써도 상황과 질문을 한 번에 이해하고, 한 손 엄지로 답을 고른 뒤 다음 문항으로 자연스럽게 이동하는 신뢰감 있는 집중 화면을 만든다.

## 1. 진입 맥락과 사용자 질문

| 항목                  | 결정                                                                 |
| --------------------- | -------------------------------------------------------------------- |
| 주요 진입             | 온보딩 G04의 빠른 코어 시작, 홈·검사 허브의 이어하기, 정밀 코어 시작 |
| 사용자의 첫 질문      | “무엇을 기준으로 답하면 되지?”                                       |
| 사용자의 두 번째 질문 | “얼마나 남았고, 선택한 답이 유지되나?”                               |
| 화면의 주 행동        | 응답을 고르고 `다음`으로 이동                                        |
| 보조 행동             | 이전 문항, 판단하기 어려움, 닫기                                     |
| 성공 기준             | 첫 문항 진입 후 5초 안에 첫 응답을 선택할 수 있음                    |

## 2. 390px 정보 위계

```text
닫기      빠른 코어      3 / 20
━━━━━━━━ 연속 진행선 ━━━━━━━━

상황 라벨
가까운 사람이 계획한 일이 뜻대로 되지 않았다고
메시지를 보낼 때

질문
처음에는 그 사람이 지금 어떤 마음일지가
궁금해진다.

이럴 때 내 모습은?
○ 1  거의 그렇지 않아요
○ 2  드문 편이에요
○ 3  반반이에요
● 4  자주 그래요
○ 5  거의 항상 그래요

이 상황은 답하기 어려워요

[이전 icon]  [             다음             ]
```

우선순위는 `상황 → 질문 → 응답 → 다음`이다. 진행 정보와 닫기는 작게 유지하고 질문보다 먼저 경쟁하지 않는다.

## 3. 시각 방향

- 온보딩과 같은 뉴앙 퍼플 계열을 상황 라벨·선택·진행선에 절제해 사용한다.
- 검사 본문에는 캐릭터·streak·배지·보상·점수 preview를 넣지 않는다.
- 큰 종이 설문 카드처럼 보이지 않도록 전체 배경·연속 진행선·즉각적인 선택 반응으로 앱의 동적 감각을 만든다.
- 질문 전체를 무거운 카드에 가두지 않고 여백과 타이포로 위계를 만든다.
- 응답은 각 행 전체가 최소 48px 이상의 터치 영역이며 한 방향만 미리 강조하지 않는다.
- 선택 상태는 부드러운 퍼플 surface, radio/check, 굵기 변화로 함께 구분한다.
- 하단 dock은 얇은 경계·blur·safe area를 사용하고 이전보다 다음을 크게 만든다.
- black/white 직접 지정 대신 Foundation token을 사용한다.

## 5. 상태 계약

| 상태             | 화면 변화                                               | CTA 계약                                |
| ---------------- | ------------------------------------------------------- | --------------------------------------- |
| 미응답           | 모든 응답 중립                                          | `다음` 비활성                           |
| 1~5 선택         | 선택 행 즉시 강조, 다른 선택 해제                       | `다음` 활성                             |
| 판단 어려움 선택 | 1~5 해제, dock 위 adaptive bottom sheet로 이유 4개 표시 | 이유 선택 후 sheet가 닫히고 `다음` 활성 |
| 저장 중          | 선택 상태 유지, 정상 문구 없음                          | 중복 선택·완료만 잠금                   |
| 저장 실패        | dock 위 compact 오류와 `다시 저장`                      | 메모리 응답이 있으면 계속 진행 가능     |
| 이전 복원        | 저장된 1~5 또는 판단 어려움·이유 복원                   | 응답이 있으면 활성                      |
| 마지막 문항      | 진행선 100%, CTA 문구 변경                              | `결과 보기`, 중복 탭 잠금               |
| 로딩             | shell·진행선 위치를 유지한 skeleton                     | 이동 불가                               |
| reduced motion   | 수평 이동 제거                                          | 즉시 교체·focus 이동 유지               |

판단 어려움 이유:

- `비슷한 경험이 거의 없어요`
- `상황에 따라 많이 달라요`
- `문장이 이해되지 않아요`
- `답하고 싶지 않아요`

이유는 정식 응답 상태로 저장·복원하며 3점으로 바꾸지 않는다.

### 5.1 판단 어려움의 non-reflow 원칙

- 이유 선택 UI는 본문 안에서 펼치지 않는다. 문항·응답·dock 위치를 그대로 둔 채 phone viewport 안의 bottom sheet로 나타난다.
- sheet는 dock 바로 위에서 올라오고, 닫기·바깥 영역·Escape로 닫을 수 있다.
- 이유를 고르면 sheet를 닫고 `판단 어려움 · 선택한 이유` 한 줄 요약과 `변경` 행동만 남긴다.
- 320px·200% 확대에서 이유 문구가 길어지면 sheet 높이만 유연하게 늘어나며, 뒤 화면의 정보 위계는 재배치하지 않는다.
- 실제 구현에서는 dialog focus 이동·focus trap·복귀 위치를 보장한다.

## 6. 상호작용·모션

- 응답 선택: 160~200ms surface·border·radio 동시 변화
- 진행선: 다음 문항 진입 시 200~240ms 너비 변화
- 다음 문항: 8~12px 이내 짧은 수평 이동과 focus 갱신
- 이전 문항: 반대 방향의 같은 규칙
- reduced motion: 위치 이동 제거
- 자동 다음 이동은 사용하지 않는다. 답을 바꾸고 확인할 시간을 남긴다.
- 마지막 문항에서도 confetti·분석 파형·과장된 기다림을 사용하지 않는다.

### 6.1 동적·시각적 품질 원칙

- 다음 문항 전환에는 본문 8px 이동, 진행선 증가, 배경의 뉴앙 orbit motif 이동을 한 번만 함께 적용한다.
- 응답 선택은 행 전체의 soft surface·radio·check icon·4px 이동으로 즉시 보이되 한쪽 답을 미리 강조하지 않는다.
- 상황 라벨에는 주제에 맞는 line icon을 연결해 긴 문장을 읽기 전에 맥락을 빠르게 잡도록 돕는다.
- 장식은 질문 판독과 신뢰를 방해하지 않는 낮은 대비의 뉴앙 전용 orbit line motif로 제한한다.
- 운영체제 기본 이모지는 사용하지 않는다. 기능 아이콘은 일관된 open-source line icon set을 쓰고, 캐릭터·보상처럼 브랜드 의미가 필요한 그래픽만 뉴앙 전용 asset으로 제작한다.
- 반복·자동 재생 애니메이션은 쓰지 않고 `prefers-reduced-motion`에서는 위치 이동을 제거한다.

## 7. 접근성·엄지 동선

- progress는 `role=progressbar`와 현재/전체 값을 함께 제공한다.
- 상황 라벨과 질문을 문항 전환 시 하나의 atomic 안내로 읽는다.
- 질문은 화면의 `h1` 하나, 1~5는 native radio group이다.
- 최소 터치 영역 44×44px, 응답 행과 주 CTA는 48px 이상을 목표로 한다.
- 이전은 44px icon action, 다음은 남은 폭 전체를 쓴다.
- 200% 확대와 320px에서 마지막 응답이 dock 뒤로 숨지 않는다.
- 선택·비활성·오류는 색만으로 구분하지 않는다.
- 정상 저장 성공을 화면 읽기 도구로 반복 낭독하지 않는다.

## 8. 현재 구현과의 차이

| 현재 runtime drift                       | v3 처리                                |
| ---------------------------------------- | -------------------------------------- |
| `N개 응답 · 자동 저장됨` 노출            | 제거; 실패·복구 시에만 저장 정보 표시  |
| `contextLabel` 미표시                    | 별도 필드·별도 시각 위계로 표시        |
| `전혀 내 모습이 아니다` 계열 legacy 응답 | 조건부 빈도 1~5 문구로 교체            |
| `판단하기 어렵다` 단일 radio             | 정식 상태명과 이유 선택·복원 추가      |
| 이전·다음 동일 폭                        | 44px 이전 icon + 확장형 다음 CTA       |
| 상단 뒤로가기 즉시 이동                  | S04 이탈·이어하기 sheet trigger로 변경 |
| white·black 직접 색 지정                 | Foundation theme token으로 교체        |
| 저장·완료 오류 복구 없음                 | 오류·재시도·중복 완료 잠금 추가        |

## 9. 이번 검토의 승인 범위

이번 승인에서는 다음만 결정한다.

1. 상황 라벨·질문·응답의 정보 위계
2. 세로형 1~5 응답 행
3. 판단 어려움 이유의 non-reflow bottom sheet와 선택 후 한 줄 요약
4. 44px 이전 icon과 확장형 다음 CTA
5. 정상·판단 어려움·저장 실패·마지막·로딩 상태 구조

이번 수정본에서 390px motion·색·아이콘 방향까지 함께 검토한다. 승인 뒤 430px·520px 반응형과 저장 실패·마지막·로딩 상태를 같은 visual language로 확장한다. 실제 production 코드·DB migration·최종 문항 binding은 별도 구현·검증 단계다.
