# NUANG 문항 설계 완료 후 UI/UX 복귀 계획

작성일: 2026-07-17 KST  
상태: `ITEM_DESIGN_GATE_COMPLETE_S03_R_NEXT`  
목적: 상황 라벨·질문 설계 때문에 중단된 앱 UI/UX 작업으로 정확한 시점에 복귀하고, 화면을 한 개씩 승인·구현한다.

## 0. 핵심 결정

문항 연구가 앱 UI/UX 설계를 무기한 막지 않는다.

```text
전 코어 문항 내부 설계·세트 감사 완료
→ 원래 중단된 S03 검사 진행 화면으로 즉시 복귀
→ 빠른 검사부터 홈까지 골든 사용자 여정을 화면별 설계
→ 완성된 모바일 프로토타입으로 M05 인지 인터뷰
→ 피드·지도·마이·공유·비교까지 전반 UI/UX 개선
→ 정량 검증된 scoring release를 최종 UI와 DB에 연결
```

전문가 검토·인지 인터뷰·정량 파일럿은 계속 필요하지만, 모든 연구가 끝날 때까지 화면 설계를 멈추지는 않는다. UI 설계와 측정 검증은 문항 내부 설계가 끝난 뒤 두 개의 연결된 작업선으로 진행한다.

## 1. UI/UX 복귀를 여는 문항 완료 게이트

다음 항목을 모두 완료해 `ITEM_DESIGN_GATE_COMPLETE`로 처리했다.

| Gate         | 작업                                         | 현재 상태 |
| ------------ | -------------------------------------------- | --------- |
| `RBL-01A`    | SE 두 facet 후보 세트 감사                   | 완료      |
| `RBL-01B`    | RO 두 facet 후보 은행·세트 감사              | 완료      |
| `RBL-01C`    | SM 후보 은행·2/3-facet 세트 감사             | 완료      |
| `RBL-01D`    | ER 두 facet 후보 은행·안전 감사              | 완료      |
| `RBL-01E`    | SE·OE·RO·SM·ER 전체 seam·상황·문구 효과 감사 | 완료      |
| `RBL-01E-R1` | RO G/A 주점수 후보 3G·3A 보강·재감사         | 완료      |

게이트 완료의 의미:

- 각 문항이 `contextLabel + promptText`로 분리돼 있다.
- 목표 facet·HIGH/LOW·단일 반응·상황 버킷·오염 위험이 정의돼 있다.
- 빠른·정밀 후보를 UI에 넣어도 데이터 필드와 화면 문법이 바뀌지 않는다.
- 아직 최종 점수 문항이나 공개 코드로 확정됐다는 뜻은 아니다.

통합 감사에서 RO-EC의 기존 12개 문장 중 대표 G/A를 직접 측정하는 `처음 드는 생각`은 6개뿐임을 확인했다. 실제 나타나는 반응 6개는 과정 프로필 보조층으로 유지하고, `RBL-01E-R1`에서 승인된 처음 생각 3G·3A를 보강했다. 공식 은행 v0.5와 150개 통합 재감사를 통과했으므로 다음 화면은 `S03-R`이다.

## 2. 게이트 이후 두 작업선

### A. UI/UX 설계·프로토타입

- 실제 모바일 화면을 한 개씩 설계하고 사용자 승인을 받는다.
- 후보 문항을 사용하되 화면에는 내부 facet·HIGH/LOW·연구 상태를 노출하지 않는다.
- 문항·결과 copy가 바뀌어도 레이아웃·상태·접근성 계약을 재사용할 수 있게 한다.
- UI가 승인되면 실제 동작 프로토타입까지 구현한다.

### B. 측정 검증

- M04 전문가는 문항 target·방향·오염을 독립 검토한다.
- M05 사용자는 A 작업선의 실제 모바일 runner에서 상황 라벨·질문·응답을 이해하는지 검증한다.
- M06·M07은 승인된 UI에서 수집한 응답으로 문항·구조·신뢰도를 검증한다.
- M08·M09에서 최종 scoring·경계·quick/full·리포트 문구 release를 만든다.

두 작업선은 최종 production binding에서 합쳐진다. 연구 전 후보 copy는 시뮬레이션·연구용이고, 고객 공개 결과는 승인된 `measurement_release_id`만 사용한다.

## 4. 화면별 고정 작업 방식

화면 하나마다 다음 순서를 지킨다.

1. 진입 맥락·사용자 질문·주 행동 하나를 brief로 확정한다.
2. 기본·로딩·오프라인·오류·권한·복구 상태를 함께 설계한다.
3. 360px wireframe으로 정보와 엄지 동선을 확인한다.
4. 390px high-fidelity 시안을 만든다.
5. 430px·520px에서 줄바꿈·safe area·고정 dock을 검증한다.
6. motion·pressed·loading·reduced-motion 상태를 정한다.
7. 사용자 문구와 개인정보 공개 등급을 검수한다.
8. 사용자가 승인하면 실제 동작 프로토타입에 반영한다.
9. 브라우저 smoke와 접근성 검증 뒤 다음 화면으로 이동한다.

여러 화면을 한꺼번에 high-fidelity로 만들지 않는다. 단, 한 화면의 정상·오류·로딩처럼 같은 기능의 필수 상태는 함께 승인한다.

## 5. S03 복귀 시 보존할 결정

- 상황 라벨은 질문에서 분리된 데이터 필드다.
- 상황 라벨은 작은 뉴앙 퍼플 텍스트, 질문은 주 제목이다.
- 진행률은 끊어진 segment가 아니라 하나의 이어진 bar다.
- 정상 상태의 `자동 저장` 문구는 보이지 않는다.
- `판단하기 어려워요`는 실제 저장·복원되는 정식 응답이다.
- 이전과 다음은 safe area를 포함한 하단 엄지 영역에 둔다.
- 검사 중 하단 탭·점수·streak·보상·정답 암시는 노출하지 않는다.
- 마지막 문항 뒤 개발용 S05 안내 문구를 보여주지 않는다.
- 빠른·정밀 문항 수는 UI에 하드코딩하지 않고 활성 release에서 받는다.

## 6. 결과·홈 설계에서 보류할 내용

UI 구조는 먼저 설계하되 아래 내용은 측정 release 전 확정하지 않는다.

- 대표 코드 확정 임계값
- 균형 구간과 대안 코드 규칙
- 빠른 결과에서 공개할 세부 신호 범위
- 정밀 결과의 숫자·그래프 정밀도
- 32개 코드 이름과 motif 최종 매핑
- 무료 주제 결과가 대표 코드를 바꾸는 조건
- 결과 리포트의 확정적 장점·주의점·관계 팁

이 영역은 실제 화면에서 `candidate content`로 추적하고, M08·M09 승인 후 production copy로 교체한다.

## 7. 완료 정의

UI/UX 복귀 계획이 지켜졌다고 판단하려면:

- 문항 내부 설계 완료 뒤 S03부터 작업이 재개된다.
- G01~G04부터 홈까지 실제로 조작 가능한 첫 가치 여정이 있다.
- M05가 별도 종이 문항이 아니라 실제 모바일 runner를 사용한다.
- 모든 화면은 승인된 온보딩 이미지와 같은 색·타이포·캐릭터 언어를 사용한다.
- 사용자 승인 없이 다음 high-fidelity 화면이나 production DB 연결로 넘어가지 않는다.
- 최종 결과·공유·비교는 승인된 measurement release만 사용한다.
