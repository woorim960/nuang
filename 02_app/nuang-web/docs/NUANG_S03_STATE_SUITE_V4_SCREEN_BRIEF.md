## 0. 한 문장 목표

검사가 잠시 멈추거나 마지막 단계에 도달해도 화면 구조와 신뢰를 흔들지 않고, 사용자가 현재 상태와 다음 행동을 즉시 이해하게 한다.

## 1. 공통 shell 불변 조건

- app bar, 연속 progress, 본문 stage, 하단 dock의 기준 위치를 상태 전환 전후 유지한다.
- 상태별 본문은 같은 최소 stage 높이를 사용해 phone shell이 위아래로 튀지 않는다.
- 오류 안내는 본문을 밀어내지 않고 dock 위 overlay card로 나타난다.
- 로딩·결과 준비에서는 runner dock과 같은 높이의 status dock을 사용한다.
- 정상 상태에서는 저장 성공·자동 저장 문구를 표시하지 않는다.
- 운영체제 기본 이모지, 의미가 불분명한 궤도 장식, 범용 회전 spinner를 사용하지 않는다.

## 2. 상태별 화면 계약

| 상태                          | 상단                                         | 본문                                                                   | 하단                                                                     | 주 행동                                 |
| ----------------------------- | -------------------------------------------- | ---------------------------------------------------------------------- | ------------------------------------------------------------------------ | --------------------------------------- |
| 초기 로딩                     | `빠른 코어` / `검사를 준비하는 중` / `— / —` | 뉴앙 전용 결과 조합 일러스트, `검사를 준비하고 있어요`, 처리 단계 문구 | `준비가 끝나면 바로 시작할게요.`                                         | 준비 완료 시 첫 문항으로 자동 전환      |
| 원격 저장 실패·로컬 보관 성공 | 현재 문항과 진행률 유지                      | 선택 응답 유지                                                         | dock 위 `응답을 바로 저장하지 못했어요`, `다시 저장`; `다음`은 활성 유지 | 재시도 또는 다음 문항                   |
| 마지막 문항                   | `20 / 20` 예시와 100% 진행선                 | `마지막 질문` 태그, 상황·질문·1~5                                      | `결과 보기`                                                              | 마지막 응답 확정 후 완료 요청           |
| 결과 준비                     | `빠른 코어 완료` / `결과를 만드는 중`        | `응답 완료`, `내 성향을 정리하고 있어요`, 뉴앙 전용 결과 조합 일러스트 | `결과가 준비되면 바로 보여드릴게요.`                                     | 결과 준비 완료 시 간단 결과로 자동 이동 |

빠른·정밀 문항 수는 실제 `measurement_release`에서 읽으며 `20 / 20`은 화면 검토용 예시다.

## 3. 로딩 상태

- 진행률을 모르면 가짜 퍼센트를 표시하지 않는다.
- 뉴앙 전용 일러스트의 미세한 호흡, 상태 점, 현재 처리 단계의 pulse가 로딩이 끝날 때까지 이어진다.
- 반복 움직임은 조급함을 만드는 회전 spinner가 아니라 1.6~2.8초의 낮은 진폭으로 유지한다.
- 처리 문구는 `질문을 불러오고 있어요` → `화면을 정리하고 있어요` → `곧 첫 질문을 보여드릴게요` 순서로 바뀐다.
- 시안의 시간 기반 문구 전환은 시뮬레이션 전용이다. 실제 앱은 `question_load_status`, `runner_prepare_status` 등 확인된 lifecycle 상태에 연결하며 처리되지 않은 단계를 완료처럼 표시하지 않는다.
- app shell을 먼저 보여 화면 전환이 끝났음을 즉시 알린다.
- 10초를 넘기거나 실패가 확인되면 `다시 불러오기`와 안전한 이전 화면 이동을 제공한다.
- 로딩 중에는 응답 입력·이전·완료를 비활성화한다.

## 4. 저장 실패 상태

### 4.1 이번 시안: 로컬 보관 성공

- 문항·선택 응답·진행률·dock 위치를 바꾸지 않는다.
- dock 위 overlay card에 다음을 표시한다.
  - `응답을 바로 저장하지 못했어요`
  - `이 기기에 임시 보관했어요. 연결되면 다시 저장할 수 있어요.`
  - `다시 저장`
- 로컬에 응답이 확인되어 있으므로 사용자는 다음 문항으로 계속 진행할 수 있다.
- 재시도 성공 시 `다시 저장했어요`를 짧게 확인하고 card를 닫는다.

### 4.2 별도 필수 상태: 로컬 보관도 실패

- `이 기기에 임시 보관했어요`라고 거짓 안내하지 않는다.
- `응답을 안전하게 보관하지 못했어요`와 재시도를 blocking 상태로 제공한다.
- 보관 성공 전에는 다음 문항·완료 요청을 진행하지 않는다.

### 4.3 구현 판정값

화면은 최소 다음 상태를 구분해야 한다.

| 필드                    | 값 예시                                  | 화면 사용                                   |
| ----------------------- | ---------------------------------------- | ------------------------------------------- |
| `local_persist_status`  | `pending · saved · failed`               | 계속 진행 가능 여부                         |
| `remote_sync_status`    | `idle · syncing · synced · failed`       | overlay error와 재시도                      |
| `completion_status`     | `idle · submitting · completed · failed` | 결과 준비·중복 완료 방지                    |
| `completion_request_id` | idempotency key                          | 완료 중복 요청 방지                         |
| `last_saved_at`         | device timestamp                         | 복구 판단용이며 정상 화면에는 노출하지 않음 |

필드명과 저장 위치는 `RBL-07` versioned migration에서 확정한다.

## 5. 태그 정렬 계약

- `마지막 질문`, `상황`, `응답 완료`는 공통 태그 component를 사용한다.
- 태그는 최소 높이 28px의 `inline-flex`이며 icon과 text를 수직 중앙 정렬한다.
- icon box는 16px 고정 폭으로 두고 글자 기준선과 시각 중심이 흔들리지 않게 한다.
- 마지막 문항의 `마지막 질문`과 `상황`은 같은 본문 column과 같은 왼쪽 기준선을 사용한다.
- `응답 완료`는 결과 준비 본문의 중앙축에 맞춘다.
- 태그가 추가되거나 빠져도 질문·응답 영역의 고정 기준 위치를 바꾸지 않는다.

## 6. 마지막 문항

- progress는 한 개의 연결된 선으로 100%를 표시한다.
- `마지막 질문` 태그는 축하 장식이 아니라 현재 위치를 알려주는 정보로 사용한다.
- 답을 선택하기 전 `결과 보기`는 비활성화한다.
- 이전 문항으로 돌아가도 선택 응답과 판단 어려움 이유를 복원한다.
- `결과 보기` 연속 탭은 첫 요청만 인정하고 즉시 결과 준비 상태로 전환한다.
- confetti, 점수 preview, 과장된 분석 파형은 사용하지 않는다.

## 7. 결과 준비 상태

- `S05`, 내부 route, 점수 계산식, 임의 퍼센트를 사용자에게 노출하지 않는다.
- 일러스트의 미세한 호흡과 단계 pulse를 유지해 처리가 계속되고 있음을 보여준다.
- 처리 문구는 `응답을 확인하고 있어요` → `성향 신호를 연결하고 있어요` → `곧 결과를 보여드릴게요` 순서로 바뀐다.
- 실제 앱에서는 완료 요청·채점·리포트 생성 lifecycle에 문구를 연결한다.
- 본문 설명은 의미 단위로 두 줄을 고정한다: `지금까지의 답을 종합해` / `간단한 결과를 만들고 있어요.`
- 320px에서도 각 줄의 끝 글자가 혼자 다음 줄로 떨어지지 않아야 한다.
- 완료 응답이 오면 간단 결과 화면으로 자동 이동한다.
- 10초 이상 지연되면 `결과를 만드는 데 시간이 걸리고 있어요`와 `다시 시도`를 제공한다.
- 완료 요청은 idempotent해야 하며 뒤로가기·재진입이 결과를 중복 생성하지 않는다.

## 8. 전용 로딩 일러스트

- 운영 asset: `public/assets/assessment/nuang-loading-result-v1.webp`
- 투명 PNG 원본: `public/assets/assessment/nuang-loading-result-v1.png`
- 생성 원본: `public/assets/assessment/nuang-loading-result-v1-source.png`
- 의미: 서로 다른 다섯 성향 신호가 중앙의 따뜻한 결과로 모이는 모습을 표현한다.
- 스타일: 뉴앙 팔레트의 premium clay/gel, 귀엽지만 유아적이지 않은 질감, 텍스트·로고·운영체제 이모지 없음.
- decorative SVG 궤도선은 제거한다. 브랜드 의미가 필요한 영역은 이 전용 asset을 사용한다.

## 9. 시각·모션 규칙

- 승인된 soft gradient, rounded response row, blur dock을 유지한다.
- 의미가 불분명했던 우측 상단 이중 타원 장식은 모든 상태에서 제거한다.
- 상태 진입 settle은 180~~320ms, 로딩 유지 motion은 낮은 진폭의 1.6~~2.8초 주기로 구성한다.
- 저장 실패 card는 240ms 이내로 dock 위에 올라오며 본문을 재배치하지 않는다.
- 기능 아이콘은 같은 open-source line icon set을 사용한다.
- 브랜드 의미가 필요한 캐릭터·보상·로딩 그래픽은 뉴앙 전용 asset만 사용한다.
- `prefers-reduced-motion`에서는 위치·크기 pulse를 제거하고 상태 문구만 갱신한다.

## 10. 접근성·복구

- loading·preparing 문구는 `aria-live=polite`, 저장 실패는 `role=alert`로 알린다.
- 이미지에는 `다섯 가지 성향 신호가 하나의 결과로 모이는 뉴앙 일러스트` 대체 텍스트를 제공한다.
- progress는 로딩 중 미확정 값과 문항 진행 중 확정 값을 구분한다.
- 상태 전환 뒤 의미 있는 제목으로 읽기 흐름을 이동하되 native tab order를 유지한다.
- 오류 card의 재시도는 44px 이상이며 상태 설명을 색만으로 구분하지 않는다.
- 320px와 200% 확대에서 오류 card·dock·본문이 겹치지 않는다.
- 네 상태 모두 light·dark theme에서 같은 의미 대비를 유지한다.

## 11. 이번 검토 범위

1. 의미 없는 우측 상단 이중 타원 제거
2. 뉴앙 전용 생성 일러스트 적용
3. 로딩·결과 준비에서 동작과 `곧` 문구가 지속되는 구조
4. `마지막 질문`·`상황`·`응답 완료` 태그의 공통 정렬 규칙
5. 결과 준비 설명의 의도된 두 줄 처리
6. 로컬 보관 성공 시 non-blocking 저장 실패 card와 무가짜 퍼센트 원칙

승인 뒤 로컬 보관 실패·장기 지연 상태, 430px·520px 반응형을 추가하고 실제 동작 prototype으로 확장한다.
