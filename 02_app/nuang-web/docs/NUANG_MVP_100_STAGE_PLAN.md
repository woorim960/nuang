# NUANG MVP 100 Stage Plan

작성일: 2026-07-09 KST

이 문서는 하네스 구축 이후 MVP 출시까지의 실행 계획이다. 각 단계 시작 전 `docs/NUANG_STAGE_HARNESS.md`의 작업 시작 순서를 따른다.

## Phase 1. Information Architecture and UX Cleanup

| Stage | 목적                    | 상세 작업                                                                         | 완료 조건                                                    |
| ----- | ----------------------- | --------------------------------------------------------------------------------- | ------------------------------------------------------------ |
| 001   | 하단 메뉴 역할 고정     | 홈, 검사, 피드, 성향지도, 마이의 역할을 `NUANG_PRODUCT_CANON.md`와 대조한다.      | 메뉴별 포함/제외 목록이 코드와 문서에 반영된다.              |
| 002   | 중복 이동 제거          | 각 화면에서 하단 메뉴 전체를 반복하는 route card를 줄인다.                        | 화면마다 맥락상 필요한 다음 행동만 남는다.                   |
| 003   | 홈 첫 화면 단순화       | 첫 검사, 진행 중 검사, 최근 결과 중 하나를 가장 강한 primary action으로 보여준다. | 첫 viewport에서 사용자가 해야 할 행동이 명확하다.            |
| 004   | 검사 홈 단순화          | 빠른 코어, 정밀 코어, 무료 주제 검사, 별난 연구소를 분리하고 선택지를 줄인다.     | 검사 시작까지 1~2번 탭으로 가능하다.                         |
| 005   | 성향지도 빈 상태 정리   | 결과가 없을 때 검사 시작 하나만 제안한다.                                         | 빈 상태가 설명 과잉 없이 행동을 유도한다.                    |
| 006   | 피드 역할 정리          | 피드 탭은 성향 기반 SNS 체류와 추천순 게시글에 집중한다.                          | 검사 진행, 리포트 관리, 비교 생성 UI가 피드에 섞이지 않는다. |
| 007   | 마이 역할 정리          | 마이는 대표 성향, 내 리포트, 설정에 집중한다.                                     | 검사/피드 탐색 UI가 마이에 섞이지 않는다.                    |
| 008   | 사용자-facing 명칭 정리 | 커뮤니티 병기를 제거하고 피드, 내 결과, 공개 범위 용어를 통일한다.                | 주요 화면에서 금지 병기 문구가 사라진다.                     |
| 009   | 화면별 선택지 밀도 점검 | 한 섹션 primary 1개, secondary 1개 원칙을 적용한다.                               | 모바일에서 선택 피로가 줄어든다.                             |
| 010   | IA 회귀 테스트 추가     | 홈/검사/피드/마이의 핵심 문구와 순서를 테스트한다.                                | IA 위반이 테스트에서 잡힌다.                                 |

## Phase 2. Visual System and Mobile Polish

| Stage | 목적                  | 상세 작업                                                         | 완료 조건                                       |
| ----- | --------------------- | ----------------------------------------------------------------- | ----------------------------------------------- |
| 011   | 디자인 토큰 정리      | 색상, radius, shadow, text scale을 현재 앱 기준으로 정리한다.     | Tailwind class 사용이 일관된다.                 |
| 012   | 과한 효과 제거        | 과한 gradient, glow, 장식 요소를 줄인다.                          | 화면이 실제 SNS 앱처럼 차분해진다.              |
| 013   | 타이포 계층 정리      | hero급 제목과 섹션 제목 크기를 분리한다.                          | 작은 카드 안 텍스트가 과하지 않다.              |
| 014   | 모바일 360px QA       | 버튼, 카드, 탭, 배지가 좁은 폭에서 겹치지 않게 조정한다.          | 360px viewport에서 레이아웃 깨짐이 없다.        |
| 015   | 이미지 자산 규칙 적용 | 캐릭터/프로필/앱 아이콘은 생성 이미지 또는 실제 asset을 사용한다. | CSS 캐릭터 신규 추가가 없다.                    |
| 016   | 5종 motif asset 정리  | 불꽃, 햇살, 물결, 숲, 별빛 계열 asset 사용 위치를 정한다.         | 32성향 확장 전 motif가 안정화된다.              |
| 017   | 빈 상태 visual 개선   | 결과 없음, 로그인 필요, 서버 준비 중 상태를 감각적으로 정리한다.  | 빈 화면이 임시 데모처럼 보이지 않는다.          |
| 018   | icon 사용 점검        | lucide-react 아이콘으로 메뉴/검색/공유/저장/뒤로가기를 통일한다.  | 이모지 사용이 없다.                             |
| 019   | 공통 컴포넌트 정리    | Button, StatusPill, 주요 카드 패턴을 정리한다.                    | 반복 UI가 같은 밀도와 행동을 가진다.            |
| 020   | 시각 회귀 확인        | 주요 viewport screenshot QA를 수행한다.                           | 홈/검사/피드/지도/마이가 모바일에서 안정적이다. |

## Phase 3. Assessment Experience

| Stage | 목적                  | 상세 작업                                                                                               | 완료 조건                                                        |
| ----- | --------------------- | ------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------- |
| 021   | 빠른 코어 시작 UX     | 20문항 시작 전 설명을 줄이고 즉시 진입을 강화한다.                                                      | 신규 사용자가 빠르게 시작한다.                                   |
| 022   | 정밀 코어 진입 UX     | 빠른 코어 완료 후 영구 무료 정밀 코어와 공식 대표 성향 기준 동선을 만든다.                              | 정밀 결과 획득 흐름이 자연스럽다.                                |
| 023   | 검사 진행 화면 polish | progress, unsure, back/next, 저장 상태, 재진입 상태를 정리한다.                                         | 검사 중 이탈/재진입이 안정적이다.                                |
| 024   | 검사 완료 전환        | 완료 직후 결과 리포트로 부드럽게 연결한다.                                                              | 완료 후 다음 행동이 모호하지 않다.                               |
| 025   | 동적 성향 증거 모델   | 검사별 관찰 점수, 유효 가중치, 대표 코드 변경 조건, 과거 리포트 보존 방식을 설계한다.                   | 무료 검사가 체류와 대표 성향 구체화의 핵심 시스템이 된다.        |
| 026   | 무료 주제 검사 모델   | 관계, 일상, 감정, 사회적 에너지, 취향, 성장 카테고리와 축·세부 성향 매핑을 설계한다.                    | 무료 검사가 세부 해석과 현재 대표 코드 갱신에 안전하게 기여한다. |
| 027   | 별난 연구소 분리      | 재미형 연구소가 기본적으로 코드 갱신에 영향을 주지 않고, 승인된 일부만 낮은 가중치로 반영되게 표시한다. | 연구소 결과가 코어 성향처럼 오해되지 않는다.                     |
| 028   | 민감 주제 안전선      | 자살, 자해, 우울, ADHD, 중독, 트라우마 등은 검사로 열지 않는다.                                         | 도움 허브 연결만 제공한다.                                       |
| 029   | 검사 seed와 저장 QA   | 문항 문구, 역문항, 축 배치, 불확실 응답, 진행 중/완료 저장을 검토한다.                                  | scoring과 재진입 test가 통과한다.                                |
| 030   | 검사 회귀 테스트      | quick/full/lab 주요 경로 테스트를 강화한다.                                                             | 검사 핵심 흐름이 자동 검증된다.                                  |

| 060   | 비교 리포트 화면        | 공통점, 차이점, 대화 질문, 조율 가이드를 보여준다.                                                                                                                    | 궁합 점수 없이 유용한 리포트가 된다.            |

## Current Next Stage

기존 Phase 1~10은 제품 셸 구현 이력으로 유지한다. 2026-07-17 점검에서 구현된 legacy 코드 체계와 승인된 `E/I · R/N · G/A · K/M · C/Q` 측정 설계의 불일치가 확인됐으므로 Stage 098 release candidate보다 [측정·제품 재기준화 게이트](./NUANG_MVP_MEASUREMENT_PRODUCT_REBASELINE.md)를 우선한다.

`RBL-01A~01D` 영역별 후보 감사, `RBL-01E` 전 5영역 통합 감사, `RBL-01E-R1` RO G/A 3G·3A 보강·정식 병합·150개 재감사를 모두 완료했다. `ITEM_DESIGN_GATE_COMPLETE`에 따라 현재 다음 단계는 `UIX-00` 상태 계약 정렬과 S03-R 빠른·정밀 공통 검사 runner v3 UI/UX다. 통합 M04·M05는 UI 작업선과 병행한다. 이후 최소 정량 파일럿, scoring/copy release, versioned migration 순으로 진행한다. 법률·OAuth·RLS·접근성 준비는 병렬 가능하지만 기존 provisional 결과를 고객 공개 MVP로 출시하지 않는다.
