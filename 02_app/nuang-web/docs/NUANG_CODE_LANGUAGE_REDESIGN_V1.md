# NUANG Code Language Redesign v1

작성일: 2026-07-17 KST  
상태: `SUPERSEDED_BY_V2`  
적용 범위: 뉴앙 코드 5자리의 글자 체계, 고객 설명 문법, 기존 코드 migration 원칙  
이번 단계에서 하지 않는 일: 실제 코드·DB·URL·32개 코드 이름 변경, 측정 경계 확정

> 후속 검토에서 이 문서의 뉴앙 F·P가 MBTI의 F·P와 다른 뜻으로 읽히는 충돌이 확인됐다. 이 문서의 알파벳 안은 사용하지 않는다. v2도 고객 표시 순서와 RO·ER·SM 의미 검토로 대체됐으며, 현재 작업 범례는 [NUANG_CODE_ORDER_AND_CORE_AXIS_REVIEW_V3.md](./NUANG_CODE_ORDER_AND_CORE_AXIS_REVIEW_V3.md)의 `E/I · R/N · G/A · K/M · C/Q`다. 아래 내용은 역사적 검토 기록으로만 남긴다.

## 0. 해결해야 하는 문제

현재 첫 번째 자리의 `S / T`는 설명을 읽으면 의미를 알 수 있지만, 글자만 보고는 어느 쪽이 사람 사이에서 에너지를 얻는 방향인지 바로 판단하기 어렵다.

MBTI의 `E / I`가 강하게 기억되는 이유는 한 글자가 이미 `Extraversion / Introversion`이라는 방향 이름과 직접 연결되기 때문이다. 뉴앙도 코드 글자가 설명문의 임의 약자가 아니라, 한 글자 자체로 방향을 회상시키는 기억 장치가 되어야 한다.

현재 체계의 구체적 문제:

- `S`가 Solo인지 Silent인지, `T`가 Together인지 먼저 배워야 한다.
- 고객 설명은 `조용히 살피는 에너지 / 먼저 움직이는 에너지`인데 S/T와 직접 연결되지 않는다.
- `S형`, `T형`만 말하면 상대가 뜻을 추론하기 어렵다.
- 코드 자체보다 별도 해설을 외워야 하므로 소셜 대화 언어로 퍼지기 어렵다.
- 첫 자리만 `E / I`로 바꾸면 현재 다섯 번째 자리의 `E`와 충돌한다. 같은 E가 `사람 에너지`와 `아이디어 탐색`이라는 두 뜻을 갖게 된다.

## 1. 설계 목표

1. 각 알파벳은 뉴앙 전체에서 한 가지 뜻만 가진다.
2. 첫 글자는 별도 설명 없이도 방향을 가장 쉽게 짐작할 수 있어야 한다.
3. 한 글자 아래의 짧은 한국어만 읽으면 생활 속 차이를 이해할 수 있어야 한다.
4. 5글자 코드는 간단한 소셜 언어로 유지한다.
5. 연속 점수, 세부 성향, 불확실성을 버리지 않는다.
6. 기존 사용자 결과와 공유 링크를 안전하게 새 코드로 변환할 수 있어야 한다.

## 2. 검토한 선택지

### A. 기존 S/T를 유지하고 Solo/Together를 교육

장점:

- 개발 migration이 없다.
- 기존 32개 코드와 motif를 그대로 쓸 수 있다.

탈락 이유:

- 사용자가 지적한 문제를 근본적으로 해결하지 못한다.
- `S/T`를 볼 때마다 영어 원뜻과 한국어 설명을 다시 연결해야 한다.
- S와 T는 성향 분야에서 이미 다른 의미로 자주 쓰여 즉시성이 낮다.

### B. 첫 자리만 E/I로 변경

장점:

- 외향·내향 방향을 가장 빠르게 이해할 수 있다.
- 사람과 상호작용하며 에너지를 얻는 경향과 먼저 표현하는 경향 모두 E/I 설명과 상당 부분 맞닿아 있다.

탈락 이유:

- 다섯 번째 자리의 기존 `E`와 글자가 중복된다.
- 예를 들어 `EVOAE`에서 첫 E와 마지막 E가 서로 다른 뜻이 되어, 글자 하나만 떼어 말할 수 없게 된다.

### C. 첫 자리 E/I + 다섯 번째 E를 N으로 변경

권장안이다.

- 첫 자리: `E / I`
- 다섯 번째 자리: `N / P`
- 나머지 세 자리는 현재 의미와 글자를 유지한다.
- 열 개 글자가 서로 겹치지 않는다.
- 기존 점수와 32개 조합은 그대로 두고 표시 코드만 일대일 변환할 수 있다.

## 3. 권장 코드 알파벳 v1

| 자리 | 고객 질문 | 방향 1 | 방향 2 | 글자 기억 단서 |
|---|---|---|---|---|
| 1 | 에너지는 어디에서 살아날까? | `E` 사람들과 움직이며 충전 | `I` 혼자 생각하며 충전 | E = Externally, I = Internally |
| 2 | 마음은 어떻게 반응할까? | `V` 감정 신호가 선명 | `C` 차분하게 정리 | V = Vivid, C = Calm |
| 3 | 일상은 어떻게 꾸릴까? | `O` 정돈해서 완성 | `F` 상황에 맞춰 유연하게 | O = Organized, F = Flexible |
| 4 | 관계에서는 무엇을 먼저 볼까? | `A` 상대 마음을 살핌 | `D` 각자의 거리와 선택을 존중 | A = Attentive, D = Distance |
| 5 | 생각은 어디로 넓어질까? | `N` 새로운 가능성을 탐색 | `P` 현실에 적용 | N = New, P = Practical |

고객 화면에는 영어 단어를 외우게 하지 않는다. 영어 단서는 글자 선정 근거와 초기 학습 보조일 뿐이며, 실제 UI에서는 아래의 짧은 한국어를 반복한다.

```text
E  함께하며 충전
V  선명한 반응
O  정돈한 일상
A  마음 살핌
N  가능성 탐색
```

반대 방향 예시:

```text
I  혼자서 충전
C  차분한 반응
F  유연한 일상
D  거리 존중
P  현실 적용
```

## 4. 첫 번째 자리의 정확한 의미

### E — 함께하며 충전

- 사람과 대화하거나 함께 움직일 때 에너지가 살아나는 편이다.
- 필요한 의견·요청·선택지를 비교적 먼저 밖으로 꺼내는 편이다.
- `사람을 더 좋아한다`, `친구가 더 많다`, `관계를 더 잘한다`는 뜻이 아니다.

### I — 혼자서 충전

- 혼자 생각하고 정리하는 시간에서 에너지가 더 잘 회복되는 편이다.
- 상황과 자신의 생각을 살핀 뒤 의견을 보태는 편이다.
- `사람을 싫어한다`, `소통을 못한다`, `소극적이다`는 뜻이 아니다.

고객은 일상 언어로 E를 외향 쪽, I를 내향 쪽으로 이해할 수 있다. 다만 뉴앙 결과 화면에서는 MBTI와 같은 검사라고 오해하지 않도록 `뉴앙의 E/I는 사람 사이의 활력과 표현 시작 방식을 함께 본 결과`라고 최초 1회 안내한다.

공식 MBTI 설명도 E/I를 에너지와 주의가 바깥 또는 안쪽으로 향하는 선호로 설명하며, E 쪽에 상호작용을 통한 활성화와 주도적 시작을 함께 포함한다. 이 점은 뉴앙 SE의 두 세부 성향과 내용상 상당 부분 겹친다. 그러나 뉴앙은 독자 문항과 연속 점수를 사용하는 별도 모델이며, MBTI 결과의 복제나 변환이 아니다.

## 5. 한 글자와 정확성을 함께 유지하는 방법

한 글자는 빠른 기억을 위한 요약이고, 측정값 전체를 대체하지 않는다.

```text
대표 글자: E
방향 강도: E 쪽으로 비교적 선명
세부 설명: 함께할 때의 에너지 높음 · 먼저 표현하기 낮음
```

필수 원칙:

- 프로필과 피드에서는 5글자 코드로 빠르게 소통한다.
- 결과 화면에서는 각 글자 아래에 짧은 한국어를 함께 보여준다.
- 글자를 누르면 연속 방향과 두 세부 설명을 펼친다.
- 두 세부 성향이 갈리면 `E인데도 먼저 나서기보다 상황을 보는 편`처럼 혼합형으로 설명한다.
- 가운데에 가까우면 글자를 보여주되 `두 모습이 비슷함`을 함께 표시한다.
- 빠른 검사의 글자는 `예비 코드`, 정밀 검사의 글자는 `현재 공식 코드`로 권위를 구분한다.

이 구조를 사용하면 소셜 표면에서는 단순한 한 글자를 유지하면서도 검사 리포트에서는 과도한 단정을 막을 수 있다.

## 6. 결과 화면의 표준 표시

### 6.1 처음 결과를 보는 화면

```text
내 뉴앙 코드
E V O A N
함께  선명  정돈  살핌  탐색

첫 글자 E
사람들과 움직일 때 에너지가 살아나는 편이에요.
```

### 6.2 프로필·피드의 축약 표시

```text
EVOAN · 불꽃의 온기 탐험가
```

코드가 아직 익숙하지 않은 초기 출시 기간에는 한 줄 보조 설명을 허용한다.

```text
EVOAN · 함께하며 충전하고, 마음 신호가 선명한 편
```

### 6.3 한 자리 상세

```text
E  함께하며 충전
I  혼자서 충전

현재는 E 쪽에 더 가까워요.
두 방향 중 어느 쪽이 더 좋다는 뜻은 아니에요.
```

### 6.4 비교 화면

`E형`, `I형`을 우열이나 고정 집단처럼 쓰지 않는다.

허용:

- `나는 E, 상대는 I에 가까워요.`
- `나는 대화하며 생각이 선명해지고, 상대는 생각을 정리한 뒤 말하는 편일 수 있어요.`

금지:

- `E가 I보다 관계에 유리해요.`
- `I는 소통이 부족해요.`
- `둘은 맞지 않아요.`

## 7. 기존 코드와 v1 코드의 변환

점수나 성향을 다시 계산하지 않고 코드 표기만 바꿀 수 있다.

| 위치 | 기존 | v1 |
|---|---|---|
| 1 낮은 방향 | S | I |
| 1 높은 방향 | T | E |
| 2 | C / V | C / V 유지 |
| 3 | F / O | F / O 유지 |
| 4 | D / A | D / A 유지 |
| 5 낮은 방향 | P | P 유지 |
| 5 높은 방향 | E | N |

예시:

| 기존 코드 | v1 코드 | 기존 코드 이름 |
|---|---|---|
| `SVODE` | `IVODN` | 이름과 캐릭터 의미는 유지 |
| `TVOAE` | `EVOAN` | 이름과 캐릭터 의미는 유지 |
| `SCFAP` | `ICFAP` | 이름과 캐릭터 의미는 유지 |

Motif key 변환:

- `TV → EV`
- `TC → EC`
- `SV → IV`
- `SC → IC`

32개 성향 조합과 코드 이름의 의미는 바꾸지 않는다. 코드 문자열 key만 변환한다.

## 8. 구현·DB migration 요구사항

코드 체계를 승인하면 한 번에 문자열 치환하지 않고 버전 migration으로 처리한다.

필수 필드·정책:

- `code_scheme_version`: 결과가 어느 알파벳 체계로 생성됐는지 기록
- `domain_scores`: 코드 문자열과 별도로 5개 연속 점수 보존
- `facet_scores`: 허용된 세부 점수 보존
- `display_code`: 요청 시 현재 체계로 계산한 표시 코드
- 구버전 `S/T...E/P` 코드는 읽기 시 v1으로 변환
- 공유 snapshot에는 생성 당시 코드와 scheme version을 함께 보존
- 기존 공유 URL과 비교 리포트는 alias resolver로 계속 열리게 함
- 코드별 피드 통계는 이전 코드 key와 새 코드 key를 같은 canonical profile로 집계
- 32개 코드 이름과 캐릭터는 canonical profile ID에 연결하고 문자열 코드에 직접 종속시키지 않음

변경 대상:

- 코드 타입과 validator
- 빠른·정밀 검사 domain symbol 정의
- 32개 코드 dictionary key
- motif key
- 결과·공유·프로필·비교·피드 통계
- URL parameter와 snapshot parser
- 테스트 fixture와 기획 예시
- 운영 DB의 코드 snapshot 및 scheme version

실제 migration은 측정 정의와 새 코드 체계가 함께 승인된 뒤 수행한다.

## 9. 출시 전 이해도 검증

코드의 목표는 설명을 읽은 뒤 이해하는 것이 아니라, 짧게 본 뒤 기억하고 말할 수 있는 것이다.

2030 사용자 대상으로 다음을 비교한다.

### 비교안

- 기존 `S/T, P/E`
- 권장 `E/I, N/P`

### 핵심 과제

1. 10초 노출 뒤 `E와 I 중 어느 쪽이 함께할 때 에너지가 살아나는가`를 맞힐 수 있는가
2. 자신의 첫 글자를 한 번 보고 10분 뒤 기억하는가
3. 다른 사람에게 `내 첫 글자는 무엇이고 어떤 뜻인지` 한 문장으로 설명할 수 있는가
4. `E가 더 좋은 성향`이라고 오해하지 않는가
5. 마지막 N과 첫 E를 서로 다른 의미로 정확히 구분하는가
6. 5글자 전체를 보고 각 자리의 짧은 한국어를 얼마나 회상하는가

### 승인 기준 초안

- 방향 선택 정확도: 90% 이상
- 첫 글자 지연 회상: 85% 이상
- 한 문장 설명 성공: 80% 이상
- 우열 오해: 10% 미만
- 첫 자리와 다섯 번째 자리 의미 혼동: 10% 미만

수치는 제품 이해도 기준 초안이며, 모집 규모와 신뢰구간을 포함한 UX 리서치 계획에서 최종 확정한다.

## 10. 승인 결정

권장 결정:

- [ ] 첫 번째 자리 `S / T`를 `I / E`로 변경한다.
- [ ] 다섯 번째 자리 `P / E`를 `P / N`으로 변경해 E 중복을 제거한다.
- [ ] 나머지 `C / V`, `F / O`, `D / A`는 의미 검증 전까지 유지한다.
- [ ] 새 코드는 한 글자당 하나의 한국어 설명을 항상 연결한다.
- [ ] 5글자 코드는 빠른 기억층, 연속 점수와 세부 설명은 정확성층으로 분리한다.
- [ ] 기존 코드는 scheme version과 canonical profile ID를 이용해 무손실 migration한다.

승인 후에는 먼저 디자인 시뮬레이션에서 `EVOAN / IVODN` 표기와 글자별 한국어 설명을 검증한 뒤 실제 코드와 DB를 변경한다.

## 11. 근거

- [The Myers & Briggs Foundation — E/I preference](https://www.myersbriggs.org/my-mbti-personality-type/the-mbti-preferences/) — E/I를 에너지와 주의가 바깥 또는 안쪽으로 향하는 상반된 선호로 설명하며, E 쪽에 상호작용을 통한 활성화와 주도적 시작을 포함한다.
- [DeYoung, Quilty, & Peterson (2007)](https://pubmed.ncbi.nlm.nih.gov/17983306/) — 외향성 영역의 Enthusiasm과 Assertiveness라는 두 aspect는 뉴앙 SE의 교류 활력·주도적 표현 가설과 연결되는 이론적 선례다.
- [Standards for Educational and Psychological Testing](https://www.testingstandards.net/) — 코드가 단순하더라도 점수 해석과 사용 범위는 타당성 근거에 의해 제한되어야 한다.
