# 신규 뉴앙 32코드 이름 설계·검증 계획

작성일: 2026-07-18 KST  
대상 코드: `E/I · R/N · G/A · K/M · C/Q`  
상태: `NAME_01_OWNER_APPROVED_APP_IMPLEMENTED_REMOTE_DB_PENDING`

## 1. 현재 문제

`불꽃의 온기 탐험가` 같은 현재 이름은 기존 코드 체계와 기존 motif 조합에서 만든 이름이다. 신규 코드 옆에 이를 그대로 표시하면 서로 다른 측정 release가 섞이고, 이름이 실제 다섯 글자의 의미를 설명하지 못한다.

따라서 다음 원칙을 즉시 적용한다.

- legacy 결과는 당시 코드와 당시 이름을 함께 보존한다.
- 신규 코드에는 legacy 이름을 자동 변환하거나 재사용하지 않는다.
- 신규 이름 release가 승인되기 전에는 `연구 중인 예비 조합`과 후보 이름을 함께 표시하되, 공유·피드·비교에는 노출하지 않는다.
- motif는 시각적 기억 장치로만 사용하고, 이름의 핵심 의미를 대신하지 않는다.

## 2. 이름이 담당할 범위

32개 이름은 사용자가 코드를 기억하고 대화를 시작하게 하는 짧은 제목이다. 이름 하나가 세부 facet, 능력, 도덕성, 관계 성공 가능성까지 설명해서는 안 된다.

결과는 세 층으로 나눈다.

1. `5글자 코드`: 공유하고 기억하는 식별자
2. `구체적인 성향 이름 + 한 줄 설명`: 다섯 방향의 대표 조합을 쉬운 한국어로 요약
3. `개인별 상세 리포트`: 실제 facet 점수, 분리 프로필, 경계와 불확실성을 설명

같은 코드를 가진 사람도 facet 점수와 경계 위치가 다를 수 있으므로 가장 구체적인 개인 설명은 정적 32개 이름이 아니라 상세 리포트가 담당한다.

## 3. 이름 작성 원칙

- 시적인 비유만으로 끝내지 않고 실제 성향 방향을 떠올릴 수 있어야 한다.
- 다섯 글자 중 일부만 설명하는 이름을 피한다.
- `리더`, `천재`, `치유자`, `완벽주의자`처럼 능력·우열·직업·도덕성을 암시하지 않는다.
- C/Q를 건강·불안장애·멘탈 강약처럼 표현하지 않는다.
- G/A를 이성적/감정적, 착함/차가움의 대립으로 표현하지 않는다.
- K/M을 성실함/무책임함 또는 계획적/즉흥적 능력으로 과장하지 않는다.
- R/N을 현실적 능력/상상력 능력으로 단정하지 않는다.
- E/I를 사교 능력이나 사회적 성공으로 해석하지 않는다.
- 2030 사용자가 한 번 읽고 뜻을 추측할 수 있는 일상 한국어를 쓴다.
- 32개 이름끼리 발음, 첫 단어, 핵심 의미가 충분히 구분돼야 한다.

## 4. 작업 시점

### NAME-00. legacy 이름 즉시 분리

시점: 신규 런타임 연결 전에 수행

- 신규 코드가 표시되는 화면에서 기존 32개 이름 lookup을 차단한다.
- 승인 전 fallback은 `연구 중인 예비 성향`으로 통일한다.
- legacy 결과는 기존 이름을 계속 재현한다.

### NAME-01. 32개 후보 이름·설명 설계

시점: 실제 결과 리포트 UI 구현과 병행

- 각 글자의 의미 블록을 먼저 고정한다.
- 32개 조합마다 다섯 방향의 공통 핵심과 내부 긴장을 정리한다.
- 조합당 이름 후보 2~3개와 한 줄 설명 후보를 만든다.
- 이름이 근거로 삼은 코드 자리와 말하면 안 되는 해석을 함께 기록한다.
- 결과 리포트 prototype에서 실제 줄바꿈·발음·기억성을 확인한다.

이 단계의 이름은 `candidate`이며 production 이름이 아니다.

### NAME-02. 사용자 언어 검토

시점: M05 인지 인터뷰와 함께 수행

- 이름만 보고 예상한 성향을 말하게 한다.
- 한 줄 설명을 읽은 뒤 다섯 방향을 어떻게 이해했는지 확인한다.
- 낙인, 우열, 정신건강, 도덕성, MBTI 동일 코드 오해를 점검한다.
- 비슷한 이름끼리 혼동되는지, 본인 코드 이름을 다시 기억하는지 확인한다.

### NAME-03. 데이터 기반 의미 정합성 검토

시점: M06·M07 문항·요인구조 검증 후, RBL-06 scoring/copy release에서 수행

- 실제로 유지된 domain·facet 구조와 이름의 주장이 일치하는지 재검토한다.
- 경계·분리 프로필에서 이름이 과도하게 단정하지 않는지 확인한다.
- 점수→코드→이름→한 줄 설명의 추적표를 승인한다.
- `profile_name_release_id`를 발행하고 32개 이름을 잠근다.

### NAME-04. 앱 전반 적용

시점: RBL-07~RBL-08 versioned migration과 production binding

- 결과 리포트, 코드 지도, 마이, 공유, 피드, 비교가 같은 이름 release를 사용한다.
- 이전 결과는 당시 이름 release를 유지한다.
- 이름만 조용히 수정하지 않고 변경 시 새 release를 발행한다.

## 7. NAME-01 정정 결정

초기 구현의 `의미 직결형 3박자 이름`은 역할형 프로필 이름이 아니라 코드의 뜻을 풀어 쓴 `직역 설명`으로 재분류한다. 사용자가 의도한 이름은 `조용히 흐름을 이끄는 지휘자`처럼 조합 전체를 기억하게 하는 역할형 별칭이다.

결과의 표시 순서는 다음으로 정정한다.

1. `ENAKQ` 같은 5글자 코드
2. 조합 전체를 기억하게 하는 역할형 프로필 이름
3. 다섯 방향을 생활 모습으로 연결한 세 문장의 `한눈에 보는 성향 설명`
4. 각 글자의 뜻을 빠르게 훑는 자리별 요약 태그
5. 자리별 상세 설명과 경계·보조 반응

자리별 요약 태그의 사용자 언어는 아래처럼 고친다.

- E `함께` / I `혼자`
- R `구체` / N `탐색`
- G `원인과 해결 살피기` / A `상대 마음 살피기`
- K `꾸준` / M `상황 따라`
- C `차분한 반응` / Q `빠른 걱정·감정 반응`

예: `ENAKQ`의 자리별 요약 → `함께·탐색, 상대 마음 살피기·꾸준, 빠른 걱정·감정 반응`

이 짧은 나열은 전체 프로필 설명으로 사용하지 않는다. 메인 설명은 `에너지와 관심 → 관계와 일상 → 걱정과 감정 반응` 순서의 세 문장으로 작성한다.

역할형 이름은 능력이나 직업을 판정하는 말이 아니라 기억과 대화를 돕는 브랜드 별칭이다. `압도하는`, `천재`, `치유자`, `완벽주의자`처럼 우열·능력·도덕성을 직접 주장하는 말은 사용하지 않는다. 이름만으로 측정 의미를 축소하지 않도록 아래 두 층을 항상 함께 제공한다.

1. 기억하기 쉬운 역할형 후보 이름
2. 생활 모습을 연결한 세 문장의 한눈에 보는 설명
3. `함께 활력·표현을 시작함`, `걱정·감정이 빨리 커짐`처럼 측정 정의에 가까운 자리별 설명과 우열 금지 문구

32개 역할형 이름 1차안은 [NUANG_32_ROLE_PROFILE_NAME_REVIEW_V1.md](./NUANG_32_ROLE_PROFILE_NAME_REVIEW_V1.md)에서 사용자 검토를 받는다. 승인 전에는 앱·DB의 이름 release를 역할형 이름으로 바꾸지 않는다. 코드 구현은 legacy 사전과 분리한 후보 release를 사용하며, DB 역시 `report.profile_name_release`와 `report.profile_name_definition`에서 별도 관리한다. 언어 검토·측정 정합성·모바일 사용성 게이트와 32개 정의가 모두 확인되기 전에는 production에서 활성화할 수 없다.
