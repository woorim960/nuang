작성일: 2026-07-17 KST  
상태: 사용 중지 · 상황과 질문 중복으로 측정 재설계 필요  
대상 release 후보: `NUANG-CORE-QUICK-1.0`  
관련 문서: `NUANG_S03_CORE_ASSESSMENT_RUNNER_DESIGN_SPEC.md`, `NUANG_S03_CONTEXT_LABEL_DB_MIGRATION_PLAN.md`

> 이 문서는 상황 라벨을 기존 질문에 덧붙이는 방식의 한계를 기록하기 위해 보존한다. 문구를 코드·DB에 반영하지 않는다. 후속 작업은 `NUANG_CORE_MEASUREMENT_VALIDATION_PLAN.md`의 M01 구성개념 명세부터 다시 진행한다.

## 0. 작성 원칙

- 기존 질문 문장은 이번 단계에서 변경하지 않는다.
- 상황 라벨은 질문이 성립하는 장면만 짧게 알려준다.
- 바람직한 행동, 좋은 성향, 정답 방향을 암시하지 않는다.
- `domainId`, `facetId`, 역채점 여부 같은 내부 기준을 노출하지 않는다.
- 질문에 없는 사람·장소·기간·빈도를 새로 만들어 넣지 않는다.
- 상황 라벨과 질문을 함께 읽었을 때 어색한 중복이 없도록 한다.
- 모든 문구는 고객 승인과 문항 콘텐츠 검수를 마친 뒤에만 release seed에 넣는다.

```ts
{
  itemId: "NU-C17-SMEP-01",
  domainId: "SM",
  facetId: "SM-EP",

  text: "해야 할 일이 생기면 미루기보다 시작하는 편이다.",
  isReverse: false,
```

## 4. 승인 게이트

빠른 코어 release seed를 만들기 전에 아래를 확정한다.

- 20개 상황 라벨 문구 승인
- 기존 질문 원문 유지 여부 승인
- 유도 표현·가치 판단·문항 간 중복 검수
- 모바일 320px에서 상황 라벨 최대 두 줄 확인
- 화면 읽기 도구에서 라벨과 질문의 연속 낭독 확인
- release ID와 content hash 생성 규칙 확인

문구 승인이 끝나기 전에는 TypeScript seed, DB migration seed, 운영 DB를 변경하지 않는다.
