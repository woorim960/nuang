## 0. 이번 개정의 결론

S04-R의 정보 구조와 시각 언어는 앱 화면에 적합하므로 전면 재설계하지 않는다. 대신 첫 결과 화면의 중복 홈 동선, 리포트 공유 진입, 반복되는 코드 설명을 아래처럼 고친다.

1. 검사를 마친 직후에는 상단 왼쪽 `홈`을 제거하고 하단 `홈으로 돌아가기`만 유지한다.
2. 상단 오른쪽에는 이 화면의 맥락 행동인 `공유`를 둔다.
3. 사용자가 마이·리포트 목록에서 결과를 다시 열었다면 왼쪽에는 `홈`이 아니라 실제 이전 화면으로 돌아가는 뒤로가기를 둘 수 있다.
4. 분명한 자리는 `~하는 편이에요`, 두 방향이 비슷한 자리만 `조금 더 가까웠지만 차이가 크지 않아요`를 사용한다.
5. 공유는 한 개의 action sheet에서 시작하되, 외부 전송과 뉴앙 피드 공유의 수명·목적을 분리한다.

## 1. 앱 내비게이션 계약

| 진입 맥락                   | 상단 왼쪽                          | 상단 중앙   | 상단 오른쪽             | 하단 행동                              |
| --------------------------- | ---------------------------------- | ----------- | ----------------------- | -------------------------------------- |
| 검사 완료 직후 첫 결과      | 비움                               | 결과 리포트 | 공유                    | 정밀 검사, 홈으로 돌아가기             |
| 마이·리포트 목록에서 재진입 | 이전 화면으로                      | 결과 리포트 | 공유                    | 상태에 맞는 다음 검사, 홈으로 돌아가기 |
| 외부 공개 공유 리포트       | 닫기 또는 브라우저 기본 내비게이션 | 공유 리포트 | 별도 재공유 정책에 따름 | 나도 같은 검사 해보기                  |

- 같은 화면에 목적지가 같은 `홈`을 두 번 노출하지 않는다.
- 첫 결과의 하단 CTA는 읽기를 마친 다음 선택하는 여정 종료 행동이라 화면 하단이 적합하다.
- 상단 오른쪽 공유는 결과를 읽는 중에도 바로 찾을 수 있는 리포트 맥락 행동이다.
- 왼쪽 슬롯을 시각적 대칭만을 위해 임의 아이콘으로 채우지 않는다.

## 2. 코드 한 자리씩 보기 문구

### 분명한 자리

| 자리 | 제목                        | 설명                                                                                   |
| ---- | --------------------------- | -------------------------------------------------------------------------------------- |
| E    | 함께하며 먼저 표현          | 사람들과 함께할 때 활력이 오르고, 필요한 말도 먼저 꺼내는 편이에요.                    |

`~하는 편이에요`는 관찰된 방향을 이해하기 쉽게 설명하되 사람 전체를 단정하지 않는 문장이다. 모든 자리에 같은 문장 골격을 기계적으로 반복하지 않고, 영역에 맞는 동사와 리듬을 사용한다.

### 두 방향이 비슷한 자리

표준 구조:

> 일을 이어가는 모습이 상황에 따라 달랐어요. 지금은 K 쪽에 조금 더 가까웠지만 M과 차이가 크지 않아요.

- `조금 더 가까웠어요`는 `NEAR_BALANCE`에서만 쓴다.
- 비슷한 상태에서는 양쪽 글자를 함께 표시하고, 현재 코드에 한 글자가 들어간 이유와 차이가 작다는 사실을 같은 설명 안에 둔다.
- `혼합형`, `애매함`, `낮은 확신도`처럼 고객이 오류로 받아들일 표현을 쓰지 않는다.

## 3. 모든 리포트의 공유 진입

공유할 결과가 있는 모든 완료 리포트는 상단 오른쪽 `공유`를 제공한다. `INSUFFICIENT_EVIDENCE`는 아직 공유할 코드·요약이 없는 복구 상태이므로 공유 버튼을 숨기고 정밀 검사와 홈만 제공한다.

공유를 누르면 하나의 action sheet를 연다.

```text
공유
├─ 카카오톡으로 보내기 · 가장 눈에 띄는 주 행동
├─ 링크 복사 · 외부 전송 보조 행동
├─ 다른 앱으로 · 기기의 기본 공유 메뉴
└─ 피드에 공유 · 뉴앙 안의 별도 수명과 작성 흐름
```

카카오톡, 링크 복사, 다른 앱으로는 모두 같은 30일 공개 리포트 URL을 사용한다. 피드 공유는 외부 URL 전달과 목적이 다르므로 구분선 아래 별도 행으로 두고 `뉴앙 안에서 반응과 댓글을 받아요`라고 설명한다.

## 4. 공유 action sheet 정보 순서

1. 제목: `이 결과를 어떻게 공유할까요?`
2. 공개 미리보기: `예비 코드 ERGKC · 가장 분명한 두 모습`
3. 개인정보 안내: `직접 답변, 원점수, 비공개 상세는 포함하지 않아요.`
4. 카카오톡으로 보내기
5. 링크 복사, 다른 앱으로
6. 외부 공유 주소의 30일 유효기간
7. 피드에 공유와 피드 수명 설명
8. 처리 결과 또는 실패·복구 안내

`ERGKC`는 화면 검토용 fixture이며 승인된 measurement release 없이 운영 결과로 쓰지 않는다. 빠른 결과를 공유할 때는 카드와 공개 페이지 모두 `예비 결과`임을 제거하지 않는다.

## 5. 공유 수명과 개인정보 계약

### 외부 공유

- `카카오톡으로 보내기`, `링크 복사`, `다른 앱으로`는 같은 `share_token`의 30일 공개 리포트 URL을 재사용한다.
- 공개 payload는 `provisional_code`, `result_resolution`, 검증된 공개용 headline, 공개용 이름·캐릭터, 만료 시점만 포함한다.
- 직접 응답, 이유 probe, 원점수, facet 점수, 과정 신호, RO-RN 등 본인 전용 상세, OAuth·이메일·연락처는 포함하지 않는다.
- 공개 공유 페이지는 noindex이며 받은 사람에게 `나도 같은 검사 해보기`를 제공한다.

### 피드 공유

- 리포트 화면의 `피드에 공유`에서 시작하며 일반 피드 작성기의 콘텐츠 유형 선택지에 중복 노출하지 않는다.
- 게시 전 공개 미리보기와 함께 공개 범위를 다시 확인한다.
- 게시한 preview와 상세 연결은 작성자가 해당 피드 글을 삭제할 때까지 유지한다.
- 공개용 projection만 저장하고 직접 응답·원점수·본인 전용 상세는 feed payload에 복제하지 않는다.

## 6. 인증과 실패 복구

- 로컬 결과를 읽는 것에는 로그인을 강제하지 않는다.
- 공유 URL 생성이나 피드 게시에 계정 연결이 필요하면 사용자가 고른 방법을 보존한 채 로그인으로 이동한다.
- 로그인 뒤 같은 결과와 action sheet로 돌아와 선택한 방법을 이어간다.
- 필요한 intent 필드는 `pending_share_method`, `return_to_result_id`다.
- 클립보드 자동 복사가 막히면 생성된 URL을 화면에 보여주고 길게 눌러 복사할 수 있게 한다.
- 기기 공유 메뉴가 지원되지 않으면 `링크 복사`를 바로 대안으로 제안한다.
- 카카오톡 앱 전환이 취소되면 실패로 낙인찍지 않고 action sheet를 유지한다.

공유 처리 상태:

```text
idle
→ auth_required
→ creating_link
→ ready
├─ handoff
├─ copied
├─ posting_feed → posted
└─ failed → 같은 방법 다시 시도 / 링크 복사
```

연속 탭은 같은 결과의 공유 URL을 중복 생성하거나 피드 글을 중복 게시하지 않아야 한다.

## 7. 구현 연결 원칙

- 카카오톡 공유는 공식 Kakao Talk Share JavaScript SDK의 기본 또는 사용자 정의 템플릿을 사용한다.
- 운영 공유 도메인을 Kakao Developers의 Product Link에 등록한 뒤 활성화한다.
- `navigator.share()`는 지원 여부를 실행 시 확인하고, 미지원·거절·실패에서는 링크 복사로 복구한다.
- action sheet는 포커스가 열린 대화상자 안에 머물고, 닫기·바깥 영역·Escape로 닫을 수 있어야 한다.
- 공유 버튼과 방법 버튼은 텍스트 라벨과 44px 이상의 터치 영역을 갖는다.
- safe area와 작은 화면을 고려해 sheet 내용은 잘리지 않아야 하며, 배경 리포트는 sheet가 열린 동안 조작되지 않아야 한다.

## 8. 앱 최적화 판단

현재 S04-R은 다음 이유로 네이티브 앱 또는 앱형 PWA에 적합한 방향이다.

- 결과 핵심을 첫 화면의 코드·한 문장으로 압축하고 세부 자리는 눌러서 여는 progressive disclosure 구조다.
- 주 CTA와 홈 복귀가 하단 전체 폭 영역에 있어 한 손 조작이 쉽다.
- 카드가 정보 단위를 나누지만 과도한 중첩과 끝없는 대시보드 탐색을 요구하지 않는다.
- motion은 코드 reveal과 상태 전환의 의미를 돕고, 신뢰를 해칠 수 있는 장식성 점수 애니메이션을 만들지 않는다.
- 시스템 공유와 앱 내부 피드 공유를 한 진입점에서 구분해 사용자의 목적 선택을 단순화한다.

따라서 현재 시각 방향은 유지하고, 이후 실제 구현에서 safe area, 화면 복귀, 키보드·스크린리더, 네트워크 실패, 분석 이벤트를 production 수준으로 연결한다.

## 9. 검토 지표

- 공유 action sheet 열기 비율
- 방법별 선택률: 카카오톡·링크·다른 앱·피드
- 공유 URL 생성 성공률과 소요 시간
- 복사 실패 뒤 수동 복사 성공률
- 공유받은 리포트 열기율과 `나도 같은 검사 해보기` 전환율
- 피드 공유 작성 완료율과 취소율
- 코드 자리 설명을 한 개 이상 열어본 비율
- 비슷한 자리 설명 이해도

공유 횟수만 높이기 위해 민감한 결과 공개를 유도하거나 기본 공개 범위를 넓히지 않는다.
