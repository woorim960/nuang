# NUANG Code Understanding Strategy

작성일: 2026-07-10 KST

상태: Stage 1 기획 초안. 이 문서는 구현 전 사용자 승인을 받기 위한 분석 문서다. 승인 전에는 비교 리포트 생성 로직, 화면 문구, DB 구조를 변경하지 않는다.

## 목적

뉴앙의 핵심 소통 단위는 `SVODE`, `TVOAE`, `SCFAP` 같은 5글자 코드다. 앞으로 사용자 화면에서는 이 코드를 `뉴앙 코드`라고 부른다.

이 문서의 목적은 사용자가 뉴앙 코드를 MBTI처럼 기억하고 말할 수 있으면서도, 단순 유형 놀이가 아니라 자신의 생활, 관계, 감정, 선택 방식을 더 구체적으로 이해하게 만드는 설명 체계를 설계하는 것이다.

## 현재 점검 결과

이미 구현된 기반:

- 정밀 코어는 공식 뉴앙 코드를 만드는 기준 검사다.
- 뉴앙 코드는 다섯 자리의 알파벳 조합으로 만들어진다.
- 현재 코드 조합은 32개 대표 성향 이름으로 연결된다.
- 10개 세부 신호는 뉴앙 코드를 더 자세히 설명하는 자료로 존재한다.
- 무료 주제 검사는 뉴앙 코드를 즉시 흔들지 않고 성향 해상도를 높이는 증거로 누적된다.
- MVP의 1:1 비교는 서로의 현재 뉴앙 코드를 기반으로 생성된다. 현재 뉴앙 코드는 정밀 코어와 이후 누적된 승인 검사 증거를 바탕으로 계산한 대표 코드다.

현재 문제:

- 화면과 문서에 `5축`, `10축` 같은 표현이 남아 있어 사용자가 뉴앙 코드를 브랜드 언어로 받아들이기 어렵다.
- 비교 리포트가 각 코드 자리별 고유 해석보다 동일한 문장 구조를 반복한다.
- 사용자는 `SVODE`가 어떤 삶의 장면과 관계되는지 직관적으로 배우기 어렵다.
- 뉴앙 코드, 성향 이름, 캐릭터, 그래프, 세부 신호가 하나의 이해 흐름으로 엮여 있지 않다.
- 32개 성향 이름은 존재하지만, 코드 조합이 왜 그 이름으로 이어지는지 설명하는 중간 레이어가 부족하다.
- `FOAMT`는 현재 32개 뉴앙 코드에 포함되지 않는 잘못된 예시로 확인됐다. 기획 문서에서는 삭제하고, 테스트 fixture와 예전 예시는 구현 정리 단계에서 유효 코드로 교체한다.

현재 확인된 용어 마이그레이션 대상:

- `src/features/together/PublicComparisonReportView.tsx`: 비교 리포트 섹션 제목
- `src/features/together/PublicComparisonReportPreview.tsx`: 비교 리포트 미리보기 섹션 제목
- `src/features/result/LocalResultView.tsx`: 결과 리포트 그래프 중심 라벨과 안내 문구
- `src/features/result/AccountResultView.tsx`: 저장 결과 리포트 그래프 중심 라벨
- `src/features/account/MyTraitDetailView.tsx`: 마이 성향 상세 제목과 안내 문구
- `src/features/map/LocalMapView.tsx`: 성향지도 탭 지표 문구
- `src/app/share/[token]/page.tsx`: 공유 리포트 그래프 중심 라벨
- `src/features/assessment/free-topic-assessments.ts`: 무료 주제 검사 결과의 사용자-facing area copy
- `src/features/navigation/nuang-next-action-flow.ts`: 다음 행동 안내 문구
- `src/features/feed/feed-seed.ts`: 공식 피드 seed 문구

## 핵심 결정

사용자-facing 표준어:

- `뉴앙 코드`: `SVODE` 같은 5글자 대표 성향 코드
- `코드 이름`: `물결의 새길 개척가` 같은 대표 성향 이름
- `코드 지도`: 뉴앙 코드를 시각적으로 보여주는 그래프 또는 요약
- `코드 자리`: 뉴앙 코드를 구성하는 각 위치
- `세부 신호`: 코드를 더 자세히 설명하는 한국어 해석 단위
- `최근 더 선명해진 부분`: 추가 검사로 더 분명해진 성향 신호
- `두 모습이 비슷한 자리`: 한쪽으로 아주 선명하게 기울기보다 두 방향이 모두 나타날 수 있는 코드 자리

사용자-facing 금지어:

- `5축`
- `10축`
- `도메인`
- `facet`
- `construct`
- `RO-EC`, `SE-AI` 같은 내부 영어 키
- `원점수`
- `raw score`
- `궁합 점수`
- `순위`
- `진단`
- `치료 필요`
- 설명 없는 `경계`

내부에서는 계산과 테스트를 위해 기존 domain/facet 구조를 유지할 수 있다. 단, 화면과 공유 문구에는 내부 구조명이 직접 노출되지 않아야 한다.

## 경계 표현 개선

기존의 `경계에 가까운 자리`는 의미가 추상적이다. 사용자에게는 `두 모습이 비슷한 자리` 또는 `두 방향이 함께 보이는 자리`로 설명한다.

내부 의미:

- 뉴앙 코드의 각 자리는 양쪽 방향 중 더 가까운 글자로 정해진다.
- 점수가 가운데 근처에 있으면 한쪽 모습만 강하게 나타난다고 보기 어렵다.
- v0.1 기준으로는 45~55 구간을 가운데 근처로 본다.
- 이 구간에서는 코드 글자가 작게 바뀌어도 사용자의 실제 모습이 완전히 달라졌다고 말하지 않는다.

사용자 설명 예시:

- `이 자리는 한쪽으로 아주 강하게 기울기보다 두 모습이 함께 나타날 수 있어요.`
- `상황에 따라 조용히 지켜보는 모습과 먼저 움직이는 모습이 번갈아 보일 수 있어요.`
- `지금은 두 방향이 비슷해서, 이름보다 실제 설명을 함께 보는 편이 좋아요.`

피해야 할 표현:

- `경계입니다.`
- `애매한 유형입니다.`
- `진짜 유형이 아직 정해지지 않았습니다.`
- `불안정합니다.`

## 사용자가 이해해야 하는 것

사용자는 뉴앙 코드를 다음 순서로 이해해야 한다.

1. 내 코드는 무엇인가
2. 이 코드는 어떤 이름과 캐릭터로 기억되는가
3. 각 코드 자리는 내 어떤 생활 장면과 연결되는가
4. 세부 신호는 내 코드를 어떻게 더 구체화하는가
5. 검사를 더 하면 무엇이 더 선명해지는가
6. 다른 사람과 비교할 때 같은 자리와 다른 자리는 어떤 오해와 편안함을 만드는가

이 흐름을 벗어나면 사용자는 점수표를 읽는 느낌을 받는다. 뉴앙은 점수표보다 `내 코드가 생활에서 드러나는 장면`을 먼저 보여줘야 한다.

## 뉴앙 코드의 5단계 이해 레이어

### 1. 코드 배지

역할:

- 사용자가 가장 먼저 기억하는 단위다.
- 프로필, 마이, 피드, 공유 리포트, 비교 리포트에서 일관되게 노출한다.
- 글자 자체가 소셜 대화 소재가 되어야 한다.

표현:

- `SVODE`
- `내 뉴앙 코드`
- `현재 뉴앙 코드`

피해야 할 표현:

- `대표 성향 코드`를 화면의 주 용어로 반복
- `5글자 코드`
- `성향 코드`
- `테스트 코드`

### 2. 코드 이름

역할:

- 코드가 차갑게 느껴지지 않도록 정서적 이름을 붙인다.
- 사용자가 자신을 소개할 때 코드와 함께 말할 수 있는 별칭이다.

표현:

- `물결의 새길 개척가`
- `불꽃의 온기 탐험가`
- `숲의 편안한 조율가`

규칙:

- `32개 성향 중` 같은 수식어를 붙이지 않는다.
- 코드 이름은 코드 아래 또는 옆에 바로 둔다.
- 이름만 단독으로 쓰기보다 뉴앙 코드와 함께 보여준다.

### 3. 코드 지도

역할:

- 뉴앙 코드가 어떤 균형으로 만들어졌는지 시각적으로 보여준다.
- 사용자가 한눈에 `어디가 선명하고 어디가 두 방향 모두 보이는지` 이해하게 한다.

표현:

- `코드 지도`
- `내 코드가 만들어지는 방식`
- `코드 자리별 흐름`
- `내 코드에서 선명한 부분`

그래프 원칙:

- 오각형 그래프는 유지할 수 있다.
- 그래프 중심 라벨은 `뉴앙 코드` 또는 `코드 지도`를 사용한다.
- 그래프 설명에서 `5축`이라는 말을 쓰지 않는다.
- 수치보다 장면형 해석이 먼저 나와야 한다.

### 4. 세부 신호

역할:

- 뉴앙 코드를 더 자세하게 이해하게 만든다.
- 무료 주제 검사와 정밀 코어의 세부 결과를 사용자 언어로 연결한다.

표현:

- `상대 마음 살피기`
- `먼저 말 꺼내기`
- `걱정과 망설임`
- `정리와 계획`
- `감각과 상상`

규칙:

- 세부 신호는 한국어 이름으로만 노출한다.
- 내부 영어 키는 절대 표시하지 않는다.
- 세부 신호는 뉴앙 코드를 바꾸는 즉시 판정이 아니라, 코드를 더 자세히 설명하는 근거로 보여준다.

### 5. 변화와 누적 근거

역할:

- 사용자가 검사를 더 할 이유를 만든다.
- 코드는 고정 운명이 아니라 현재까지의 근거로 보는 설명을 제공한다.

표현:

- `최근 검사에서도 같은 방향이 반복 확인되어 더 선명해졌어요.`
- `이 자리는 두 모습이 비슷하게 함께 나타날 수 있어요.`
- `여러 검사에서 같은 흐름이 쌓여 현재 코드 해석이 더 구체화됐어요.`

피해야 할 표현:

- `진짜 유형이 바뀌었습니다.`
- `이전 결과가 틀렸습니다.`
- `정확도가 상승했습니다`처럼 차갑고 기술적인 표현
- `유료 검사를 해야 더 정확합니다`

## 뉴앙 코드 자리 설계

뉴앙 코드는 다섯 개의 코드 자리로 구성된다. 사용자에게는 `첫 번째 자리`, `두 번째 자리`처럼 말하되, 필요할 때만 전체 코드 안에서 해당 글자를 강조한다.

원칙:

- 글자는 항상 전체 뉴앙 코드 안에서 보여준다.
- 단독으로 `S형`, `T형`처럼 분리해 사용자 라벨로 만들지 않는다.
- 내부 계산 ID는 노출하지 않는다.
- 각 자리는 점수가 아니라 생활 장면으로 설명한다.

| 코드 자리 | 사용자 설명 방향 | 생활 장면 예시 |
|---|---|---|
| 첫 번째 자리 | 사람 사이에서 에너지를 쓰는 방식 | 모임, 대화 시작, 관계 속 활력 |
| 두 번째 자리 | 마음이 흔들리거나 회복되는 방식 | 걱정, 감정 반응, 안정감 회복 |
| 세 번째 자리 | 일상을 움직이는 방식 | 실행, 마무리, 정리, 계획 |
| 네 번째 자리 | 가까운 관계에서 거리를 맞추는 방식 | 배려, 선택 존중, 부탁과 거절 |
| 다섯 번째 자리 | 감각과 생각을 받아들이는 방식 | 상상, 취향, 아이디어, 새로운 관점 |

## 32개 성향 이름 구조

현재 32개 코드 이름은 아래 구조로 이해할 수 있다.

첫 두 글자는 대표 motif를 만든다. 사용자에게 보여줄 때는 motif와 이름 사이에 `의`를 붙여 더 자연스럽게 읽히도록 한다.

| 코드 시작 | motif | 사용자 인상 | 코드 이름 예시 |
|---|---|---|---|
| TV | 불꽃 | 선명함, 온도, 추진력 | 불꽃의 온기 탐험가 |
| TC | 햇살 | 밝음, 안정, 친근함 | 햇살의 세심한 실천가 |
| SV | 물결 | 유연함, 감수성, 흐름 | 물결의 새길 개척가 |
| SC | 숲 | 차분함, 안정감, 깊이 | 숲의 편안한 조율가 |

뒤 세 글자는 성향 이름의 방향을 만든다.

| 코드 끝 | 이름 조합 | 사용자 인상 |
|---|---|---|
| OAE | 온기 탐험가 | 따뜻하게 다가가고 새로움을 탐색 |
| OAP | 세심한 실천가 | 관계를 살피며 현실적으로 실행 |
| ODE | 새길 개척가 | 자기 리듬으로 새로운 길을 찾음 |
| ODP | 현실 추진가 | 분명한 기준으로 현실을 움직임 |
| FAE | 영감 동행자 | 감성과 아이디어로 함께 움직임 |
| FAP | 편안한 조율가 | 부드럽게 맞추고 안정감을 만듦 |
| FDE | 자유 발견가 | 자유롭게 탐색하고 자기 방식을 찾음 |
| FDP | 담백한 해결가 | 과장 없이 필요한 해결을 찾음 |

이 구조는 사용자에게 처음부터 표로 모두 노출하지 않는다. 결과 리포트와 코드 상세 화면에서 `내 코드 이름이 어떻게 만들어졌는지`를 단계적으로 보여줄 때 사용한다.

코드 이름 표기 원칙:

- 사용자-facing 이름은 `모티프의 이름` 형태를 기본으로 한다.
- 예: `불꽃의 온기 탐험가`, `물결의 새길 개척가`, `숲의 편안한 조율가`
- 기존 데이터와 테스트 fixture에 남아 있는 `불꽃 온기 탐험가` 형태는 승인 후 구현 단계에서 일괄 교체한다.

## 사용자 경험 설계 원칙

### 마이

마이 상단은 사용자의 뉴앙 코드가 주인공이어야 한다.

권장 구조:

1. 프로필 이미지
2. 뉴앙 코드
3. 코드 이름
4. 짧은 한 문장 해석
5. 코드 상세로 이동하는 자연스러운 터치 영역

마이에서 피해야 할 것:

- 코드 자리별 긴 설명
- 모든 리포트 요약을 한 화면에 펼치기
- 설정성 기능을 상단 성향 영역에 섞기

### 결과 리포트

결과 리포트는 `내 뉴앙 코드가 무엇이고, 생활에서 어떻게 보이는가`를 설명해야 한다.

권장 구조:

1. 뉴앙 코드와 코드 이름
2. 한 문장 핵심 해석
3. 코드 지도
4. 코드 자리별 해석
5. 세부 신호
6. 최근 더 선명해진 부분
7. 다음에 해볼 검사 또는 작은 실험

### 성향지도

성향지도 탭은 32개 뉴앙 코드를 둘러보는 정보성 탭으로 정리한다.

권장 구조:

1. 뉴앙 코드란 무엇인지 아주 짧게 소개
2. 32개 코드 탐색
3. motif별 또는 코드 이름별 탐색
4. 내 코드와 가까운 코드 보기

개인 결과 상세는 마이 상단 뉴앙 코드 영역에서 진입한다.

성향지도 신뢰 설명:

- 뉴앙 코드는 성격을 한 단어로 단정하기보다, 사람이 상황에 따라 여러 모습을 가질 수 있다는 전제에서 설계한다.
- 널리 알려진 성격 연구의 다섯 가지 큰 관점과 성향 검사 설계 원칙을 참고하되, 뉴앙 서비스에 맞게 관계, 일상, 감정, 취향, 대화 장면으로 다시 풀어낸다.
- MBTI처럼 기억하기 쉬운 코드 경험은 살리되, 사람이 한쪽으로만 나뉜다고 단정하지 않는다.
- 정밀 코어 이후에도 다양한 주제별 검사를 통해 내 코드 해석이 더 구체화될 수 있다.
- 뉴앙 코드는 진단이나 등급이 아니라, 나와 다른 사람을 더 쉽게 이해하기 위한 언어다.

성향지도 첫 화면 문구 초안:

```text
뉴앙 코드는 성격을 딱 잘라 나누기보다, 사람이 상황에 따라 여러 모습을 가질 수 있다는 점을 반영해 만들었어요.
성격 연구에서 널리 쓰이는 큰 관점들을 참고하고, 뉴앙의 질문과 AI 분석으로 관계와 일상에서 이해하기 쉬운 코드로 정리했습니다.
검사를 더 해볼수록 내 코드의 세부 신호가 조금씩 더 선명해져요.
```

### 피드

피드는 뉴앙 코드를 대화 소재로 만드는 장소다.

허용:

- `오늘 내 코드가 드러난 순간`
- `내 코드와 비슷한 사람들이 고른 선택`
- `코드 이름으로 보는 오늘의 질문`
- 검사 리포트 공유 preview
- 오늘의 질문 기반 글쓰기
- 밸런스 게임 기반 투표형 글쓰기
- 기본 소셜 피드 글쓰기, 반응, 댓글, 저장

금지:

- 내부 점수 공개
- 코드 우열 비교
- 위험 인물 판정
- 궁합 점수형 피드

피드 MVP 핵심:

- 피드는 인스타그램/스레드처럼 자연스럽게 글을 올리고 반응하는 소셜 공간이다.
- 다만 일반 커뮤니티가 아니라, 나를 이해하고 서로를 더 이해하는 성향 탐구 놀이터로 설계한다.
- 사용자는 자신의 검사 리포트를 피드에 공유할 수 있다.
- 리포트 공유 피드는 핵심 preview만 보여주고, 클릭하면 상세 리포트로 이동한다.
- 피드 작성 화면 상단에는 글의 형식을 고르는 선택지가 있다.
- `오늘의 질문`, `밸런스 게임`, `리포트 공유`, `일반 글`처럼 선택하되, 사용자가 직접 분류명을 입력하지 않는다.
- 사용자가 `오늘의 질문`을 누르면 서비스가 쉬운 질문을 제안하고, 사용자는 그 질문에 답하듯 글을 쓴다.
- 사용자가 `밸런스 게임`을 누르면 질문과 선택지가 제공되고, 사용자는 답을 선택해 업로드한다.

오늘의 질문 예시:

- `오늘 저녁 시간은 누구와 함께 보내시나요?`
- `요즘 나를 가장 편하게 해주는 루틴은 무엇인가요?`
- `오늘 누군가에게 고마웠던 순간이 있었나요?`
- `혼자 있고 싶은 날, 나는 어떤 신호를 보내나요?`

밸런스 게임 예시:

- `나 혼자 여행 간다면?` 선택지: `산`, `바다`
- `평소 내가 고르는 음식은?` 선택지: `짜장`, `짬뽕`
- `하루 쉬는 날 더 끌리는 건?` 선택지: `아무 계획 없이 쉬기`, `가고 싶던 곳 다녀오기`
- `내가 더 힘든 상황은?` 선택지: `말없이 멀어지는 사람`, `계속 확인하려는 사람`

민감 밸런스 게임 주의:

- `육체적 바람 vs 심리적 바람`처럼 강한 관계 주제는 높은 참여를 만들 수 있지만, 사용자 감정과 신고 리스크가 커질 수 있다.
- MVP에서는 너무 자극적인 질문보다 생활, 취향, 관계 거리감, 대화 방식 중심으로 시작한다.
- 관계·연애의 민감 주제는 운영 정책과 신고/숨김 UX가 정리된 뒤 열어야 한다.

투표형 피드 결과:

- 사용자가 답을 선택하면 전체 선택 수와 비율을 보여준다.
- 더 자세히 보고 싶으면 `뉴앙 코드별 통계 보기`로 이동한다.
- 통계 화면은 뉴앙 코드별 각 선택지 비율을 보여준다.
- 표본 수가 너무 적은 코드는 노출하지 않거나 `아직 충분한 응답이 없어요`로 표시한다.
- 개인의 선택은 다른 사람에게 직접 식별되지 않아야 한다.

### 1:1 비교

비교 리포트는 `같은 코드 자리`, `다른 코드 자리`, `서로 오해하기 쉬운 장면`, `편하게 맞출 수 있는 방식`을 중심으로 구성한다.

MVP 비교 기준:

- MVP의 핵심 비교 리포트는 서로의 현재 뉴앙 코드를 기준으로 만든다.
- 현재 뉴앙 코드는 정밀 코어와 누적 검사 증거를 바탕으로 계산된 대표 코드다.
- 단순히 `내 정밀 코어 결과 vs 상대 공개 프로필`처럼 표현하지 않는다.
- 공개 프로필은 비교를 시작하고 공개 범위를 확인하는 통로이며, 비교의 핵심 의미는 서로의 뉴앙 코드 조합에 있다.

후속 확장 비교 기준:

- 뉴앙 코드에 직접 영향을 주지 않는 상세검사나 유료검사는 별도 비교 리포트로 확장할 수 있다.
- 단, 두 사람이 모두 같은 검사를 수행했고, 둘 다 해당 결과의 비교 사용을 허용한 경우에만 비교한다.
- 한쪽만 수행한 검사는 비교하지 않는다.
- 서로 다른 검사끼리 억지로 비교하지 않는다.
- MVP에서는 유료 상세검사 기반 비교를 열지 않는다.

비교 리포트에서 필요한 변화:

- 모든 자리에서 같은 문장 구조를 반복하지 않는다.
- 각 코드 자리의 의미에 맞는 고유 문장 라이브러리를 만든다.
- 두 사람의 코드 글자가 같은지, 다른지, 점수 차이가 큰지, 두 모습이 비슷한 자리인지에 따라 문장을 달리한다.
- 공개되지 않은 정보는 추정하지 않는다.
- 결과는 관계 성공 예측이나 궁합 점수처럼 보이면 안 된다.

## 뉴앙 코드 설명 방식 옵션

### 옵션 A: 가벼운 코드 중심

내용:

- 전체 코드와 코드 이름만 강하게 보여준다.
- 코드 자리 설명은 접힌 상태로 둔다.

장점:

- 쉽고 빠르다.
- SNS 프로필처럼 보인다.

단점:

- 사용자가 각 글자의 의미를 배우는 속도가 느릴 수 있다.

### 옵션 B: 단계형 코드 학습

내용:

- 전체 코드를 보여준 뒤, 사용자가 원할 때 각 자리 설명을 하나씩 펼친다.
- 처음에는 한 문장, 더 보기에서 자세한 설명을 제공한다.

장점:

- 쉽지만 깊이도 있다.
- 코드가 MBTI처럼 대화 소재가 되기 좋다.

단점:

- 문장 설계와 UI 디테일이 필요하다.

### 옵션 C: 상세 분석형

내용:

- 코드 자리, 세부 신호, 변화 근거를 한 번에 자세히 보여준다.

장점:

- 신뢰감이 강하다.
- 정밀 리포트 느낌이 난다.

단점:

- 첫 사용자에게 무겁다.
- 모바일 첫 화면이 복잡해질 수 있다.

권장안:

- MVP 기본은 옵션 B다.
- 마이와 프로필 팝업은 옵션 A에 가깝게 간결하게 보여준다.
- 결과 리포트와 비교 리포트는 옵션 B를 기본으로 하되, 하단에 상세 분석을 제공한다.
- 상세검사나 유료 확장에서는 옵션 C를 검토한다.

## 비교 리포트로 이어지는 설계 원칙

비교 리포트는 단순히 `내 점수와 상대 점수`를 보여주는 화면이 아니다. 두 뉴앙 코드가 만났을 때 생기는 대화 리듬을 설명해야 한다.

비교 리포트 v1의 기본 섹션 후보:

1. 두 사람의 뉴앙 코드 요약
2. 같은 코드 자리
3. 다른 코드 자리
4. 가장 오해하기 쉬운 장면
5. 가장 편하게 맞는 장면
6. 대화 시작 문장
7. 관계 조율 팁
8. 공개 범위 안내

문장 생성 기준:

- 코드 자리별 고유 문장 라이브러리를 만든다.
- 동일 코드 자리와 다른 코드 자리를 구분한다.
- 차이값이 작으면 `비슷함`, 중간이면 `상황에 따라 다름`, 크면 `조율 필요`로 해석한다.
- 두 모습이 비슷하게 나타나는 자리는 단정하지 않는다.
- 각 자리별로 `일상 장면`, `오해 장면`, `상대에게 편한 말`, `내가 주의할 점`을 다르게 둔다.

## 추후 잘 맞는 성향 지도

사용자는 결국 `나와 잘 맞는 성향은 무엇인지`, `내가 좋아하는 사람과 잘 맞을 가능성이 있는지`를 궁금해할 수 있다. 뉴앙은 이 니즈를 무시하지 않되, MVP에서 바로 궁합 점수나 관계 성공 예측으로 제공하지 않는다.

추후 방향:

- 성향지도 탭에 `잘 맞는 코드 보기` 또는 `편하게 맞는 코드` 섹션을 추가할 수 있다.
- 제공 방식은 점수나 순위보다 `편하게 맞는 지점`, `오해가 생기기 쉬운 지점`, `대화를 시작하는 방식` 중심이어야 한다.
- 특정 코드가 무조건 좋은 상대라고 단정하지 않는다.
- 연애, 친구, 팀, 가족처럼 관계 맥락에 따라 잘 맞는 방식이 다를 수 있음을 설명한다.
- 충분한 데이터와 운영 정책이 쌓인 뒤, 뉴앙 코드별 상호작용 통계와 사용자 피드백을 참고해 확장한다.

MVP 금지:

- 궁합 점수
- 관계 성공 확률
- 특정 코드를 우월하거나 열등하게 보이게 하는 표
- 상대의 비공개 정보 추정

## 구현 전 승인 단계

Stage 1. 뉴앙 코드 이해 체계 분석 기획

- 이 문서 작성
- 사용자-facing 용어 결정
- `5축` 표현 폐기 원칙 수립
- 뉴앙 코드 설명 레이어 설계
- `FOAMT` invalid 예시 정리 대상 확인
- 코드 이름을 `모티프의 이름` 형태로 바꾸는 원칙 수립
- 피드 MVP의 리포트 공유, 오늘의 질문, 밸런스 게임, 코드별 통계 방향 수립

Stage 2. 뉴앙 코드 해석 사전 설계

- 다섯 코드 자리별 상세 해석
- 각 자리의 양방향 문장 라이브러리
- 10개 세부 신호의 사용자-facing 이름과 설명
- 32개 코드 이름의 조합 원리와 기본 해석

Stage 3. 비교 리포트 정보 구조 설계

- 비교 리포트 섹션 확정
- 같은 코드 자리와 다른 코드 자리 해석 규칙
- 오해 장면, 대화 가이드, 조율 가이드의 문장 구조
- 공개 범위와 안전 문구 확정
- MVP는 현재 뉴앙 코드 기반 비교에 집중하고, 상세검사 비교는 후속 확장으로 분리

Stage 4. 비교 리포트 생성 계약 설계

- payload 구조
- versioning
- 구버전 리포트 재생성 조건
- 테스트 fixture
- seed 데이터 보강

Stage 5. UI 용어 마이그레이션

- 화면의 `5축` 표현 제거
- `뉴앙 코드`, `코드 지도`, `세부 신호` 반영
- 마이, 결과 리포트, 공유 리포트, 비교 리포트, 성향지도 순서로 적용

Stage 6. 구현과 검증

- 계약 테스트
- 렌더 테스트
- 하네스 금지어 스캔
- 실제 데모 데이터 smoke
- 모바일 화면 확인

Stage 7. 피드 MVP 고도화 설계

- 리포트 공유 preview
- 오늘의 질문 작성 흐름
- 밸런스 게임 투표 모델
- 투표 결과와 뉴앙 코드별 통계 화면
- 표본 수 부족 시 비노출 기준
- 민감 질문 운영 기준

## Stage 1 승인 요청 포인트

다음 단계로 넘어가기 전에 승인받아야 할 결정:

1. 사용자-facing 표준 용어를 `뉴앙 코드`로 고정한다.
2. 사용자 화면에서 `5축`, `10축` 표현을 쓰지 않는다.
3. `코드 지도`, `코드 자리`, `세부 신호`를 새 설명 언어로 사용한다.
4. `경계` 대신 `두 모습이 비슷한 자리`처럼 누구나 이해하기 쉬운 표현을 사용한다.
5. 코드 이름은 `불꽃의 온기 탐험가`, `숲의 새길 개척가`처럼 `모티프의 이름` 형태를 기본으로 한다.
6. `FOAMT`는 invalid 예시로 보고 승인 후 테스트 fixture와 예전 예시에서 제거한다.
7. 뉴앙 코드 설명 방식은 옵션 B, 즉 단계형 코드 학습을 기본으로 한다.
8. 비교 리포트는 점수 차이표가 아니라 코드 조합 기반 관계 리듬 리포트로 재설계한다. MVP에서는 이 코드 조합을 서로의 현재 뉴앙 코드에서 가져온다.
9. MVP에서는 유료 상세검사 기반 비교를 열지 않고, 추후 두 사람이 모두 같은 검사를 수행한 경우에만 추가 비교를 검토한다.
10. 피드 MVP는 리포트 공유 preview, 오늘의 질문, 밸런스 게임, 뉴앙 코드별 통계를 핵심 기능 후보로 둔다.
