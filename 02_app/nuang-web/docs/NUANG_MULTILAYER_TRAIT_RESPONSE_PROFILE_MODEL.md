## 0. 제품 결정

뉴앙은 같은 상황에서 아래 세 가지를 하나의 문항이나 하나의 점수로 섞지 않는다.

```text
1. 사용자가 보고한 처음 드는 생각
2. 최근 비슷한 상황에서 실제 나타나는 반응
3. 사용자가 직접 밝힌 반응 이유
```

예를 들어 `원인과 해결책이 먼저 궁금하지만, 상대가 상할 수 있어 마음을 먼저 확인하는 사람`을 단순히 마음 우선 또는 해결 우선으로 분류하지 않는다.

```text
처음 드는 생각: 원인·해결
실제 나타나는 반응: 상대 마음 확인
직접 선택한 이유: 내 말이 상대에게 미칠 영향 고려
```

이 구조는 여섯 번째 성향 축이나 새로운 코드 글자가 아니다. 기존 5글자 코드는 검증된 facet 점수로 유지하고, 다층 프로필은 정밀 리포트의 해상도를 높이는 보조 해석으로 사용한다.

## 1. 세 층위의 의미 계약

| 내부 값                      | 고객에게 설명하는 말 | 측정 대상                                                        | 점수 역할                                       | 금지되는 해석               |
| ---------------------------- | -------------------- | ---------------------------------------------------------------- | ----------------------------------------------- | --------------------------- |
| `REPORTED_FIRST_ORIENTATION` | 처음 드는 생각       | 최근 비슷한 상황에서 사용자가 보고한 첫 관심·생각·감정·반응 충동 | 구성개념에 따라 핵심 trait 증거 가능            | 타고난 본성, 무의식의 진실  |
| `ENACTED_RESPONSE`           | 실제 나타나는 반응   | 최근 비슷한 상황에서 사용자가 보고한 말·행동 순서                | 구성개념에 따라 핵심 증거 또는 profile modifier | 연기, 가식, 능력·성과       |
| `SELF_REPORTED_RATIONALE`    | 내가 직접 고른 이유  | 사용자가 자신의 반응 이유로 선택한 설명                          | 비채점 맥락 정보                                | 객관적으로 증명된 동기·원인 |

두 층위 모두 자기보고다. `REPORTED_FIRST_ORIENTATION`은 실제 무의식이나 반응 속도를 직접 측정하지 않으며, `ENACTED_RESPONSE`도 관찰자가 확인한 객관적 행동 기록이 아니다. 질문과 도움말은 최근 비슷한 경험을 떠올려 답하도록 안내한다.

고객 표현의 `생각`은 완성된 문장이나 숙고한 판단만 뜻하지 않는다. 무엇이 먼저 눈에 들어오거나 궁금해지고 신경 쓰였는지를 함께 묻는 쉬운 이름이다.

## 2. 점수와 프로필을 분리한다

각 문항에는 층위와 별도로 점수 역할을 기록한다.

| `scoring_role`        | 의미                                       | 코드 영향                |
| --------------------- | ------------------------------------------ | ------------------------ |
| `CORE_TRAIT_EVIDENCE` | 승인된 facet 점수에 직접 사용할 후보       | 검증 후 가능             |
| `PROFILE_MODIFIER`    | 같은 trait가 행동으로 표현되는 방식을 설명 | 기본적으로 코드에 미반영 |
| `RESEARCH_ONLY`       | 경계·오염·가설 검증용                      | 불가                     |
| `NON_SCORING_CONTEXT` | 이유 선택처럼 사용자 설명을 보존           | 불가                     |

같은 `ENACTED_RESPONSE`라도 RO-RN처럼 행동 억제 자체가 구성개념이면 핵심 trait 증거가 될 수 있다. 반면 RO-EC에서 `마음을 먼저 확인하는 반응`은 처음 드는 생각과 구분하기 위한 profile modifier로 먼저 검증한다.

따라서 `행동 문항은 항상 부가 정보`, `생각 문항은 항상 핵심 점수`처럼 기계적으로 적용하지 않는다. M01 구성개념 정의가 역할을 결정한다.

## 3. 다층 조합 해석

검증 전 내부 작업명이며 고객에게 고정 유형처럼 노출하지 않는다.

| 처음 드는 생각        | 실제 나타나는 반응 | 내부 상태                             | 허용되는 고객 설명                                                          |
| --------------------- | ------------------ | ------------------------------------- | --------------------------------------------------------------------------- |
| 마음·안녕             | 마음 확인          | `ALIGNED_ATTENTIVE`                   | 마음이 먼저 보이고, 실제 반응도 마음을 확인하는 쪽에 가까워요.              |
| 원인·해결             | 해결 제시          | `ALIGNED_GOAL`                        | 해결할 부분이 먼저 보이고, 실제 반응도 해결을 꺼내는 쪽에 가까워요.         |
| 원인·해결             | 마음 확인          | `GOAL_ORIENTATION_ATTENTIVE_RESPONSE` | 해결할 부분이 먼저 보이지만, 실제로는 상대의 마음을 먼저 확인하는 편이에요. |
| 마음·안녕             | 해결 제시          | `ATTENTIVE_ORIENTATION_GOAL_RESPONSE` | 상대의 마음이 먼저 보이지만, 실제로는 필요한 해결을 먼저 꺼내는 편이에요.   |
| 한쪽 또는 양쪽이 경계 | 다양함             | `PROCESS_NEAR_BOUNDARY`               | 상황에 따라 먼저 보이는 것과 반응 순서가 달라질 수 있어요.                  |
| 유효 증거 부족        | 알 수 없음         | `PROCESS_INSUFFICIENT_EVIDENCE`       | 이 차이를 설명하려면 응답이 조금 더 필요해요.                               |

`GOAL_ORIENTATION_ATTENTIVE_RESPONSE`를 배려가 깊다거나 감정을 숨긴다고 해석하지 않는다. `ATTENTIVE_ORIENTATION_GOAL_RESPONSE`도 냉정하거나 문제 해결 능력이 높다고 해석하지 않는다.

## 4. 측정 설계

### 4.1 같은 상황에서 층위를 나누되 연속 제시하지 않는다

각 process pair는 같은 강도의 상황을 공유한다.

```text
process pair A
├─ 처음 드는 생각 문항: 무엇이 먼저 눈에 들어오거나 떠오르는가
└─ 실제 나타나는 반응 문항: 최근 비슷한 상황에서 무엇을 말하거나 했는가
```

두 문항은 같은 화면이나 바로 다음 순서에 놓지 않는다. 최소 세 문항 이상을 사이에 두거나 별도 블록에 배치해 사용자가 앞 답과 기계적으로 일치시키지 않게 한다.

완전히 같은 문장을 반복해 기억 효과가 커지면 인지 인터뷰에서 `동일 상황`과 `강도가 같은 평행 상황`을 비교한다.

### 4.2 상세 프로필 최소 증거

정밀검사에서 다음 조건을 모두 만족할 때만 다층 설명 후보를 만든다.

- 서로 다른 중간 강도 상황의 유효 process pair가 최소 3개
- 각 층위에 HIGH·LOW 방향 문항이 모두 포함
- 처음 드는 생각 점수와 실제 나타나는 반응 점수를 따로 계산할 수 있음
- 한 개 문항이 아니라 같은 차이가 최소 2개 상황에서 반복
- 두 층위 각각의 측정오차와 경계 상태가 허용 범위 안
- 응답 품질과 판단 어려움 비율이 사전 등록 기준을 통과

정확한 경계와 신뢰도 기준은 인지 인터뷰·파일럿 뒤 확정한다. 원점수 평균만으로 네 조합을 즉시 발급하지 않는다.

### 4.3 빠른 코어 제한

빠른 코어는 문항 수가 적으므로 다층 프로필을 확정하지 않는다.

- 한두 문항 차이는 `반응이 다르게 나타날 수 있다는 예비 신호`로만 내부 보존
- 고객에게 네 조합 이름을 단정적으로 노출하지 않음
- 정밀검사에서 최소 증거가 충족된 뒤 상세 설명 제공

## 5. 이유 확인 문항

### 5.1 왜 별도 문항인가

`상대가 상할 수 있어서 마음을 먼저 확인한다`를 행동만 보고 추론할 수 없다. 같은 행동도 관심, 예의, 갈등 회피, 정확한 문제 파악, 직업 규범 때문에 나타날 수 있다.

이유는 1~5 trait 점수에 섞지 않고 별도 적응형 probe에서 사용자가 직접 선택하게 한다.

### 5.2 노출 조건

운영 후보 조건:

1. 최소 3개 process pair가 유효하다.
2. 처음 드는 생각과 실제 나타나는 반응의 방향 차이가 최소 2개 상황에서 같은 방향으로 반복된다.
3. 검사 마지막에 한 번만 표시한다.
4. 건너뛰기와 `상황마다 달라요`를 허용한다.

인지 인터뷰·파일럿에서는 선택 편향을 확인하기 위해 일부 일치 프로필에도 무작위로 제시할 수 있다.

### 5.3 고객 문구

```text
방금 답한 내용을 조금 더 정확히 이해하고 싶어요.

해결할 부분이 먼저 보이지만 상대의 마음을 먼저 확인하는 데에는
어떤 이유가 가장 가까운가요?

□ 상대의 마음을 더 이해하고 싶어서
□ 내 말이 상대에게 미칠 영향을 고려해서
□ 문제를 정확히 이해한 뒤 말하고 싶어서
□ 대화가 원활하게 이어지도록 하려고
□ 상황과 사람에 따라 이유가 달라서
□ 답하고 싶지 않아요
```

이 선택지는 정답이 아니며 복수 선택 허용 여부는 인지 인터뷰에서 결정한다.

### 5.4 결과 사용

- 결과에 표시할 때 `뉴앙이 판단한 이유`가 아니라 `내가 직접 고른 이유`로 표시한다.
- 이유 응답은 코드·facet 점수·비교 점수에 사용하지 않는다.
- 공개 프로필·피드·1:1 비교에는 기본 비공개다.
- 응답하지 않아도 결과 해상도나 서비스 이용에서 불이익을 주지 않는다.

## 6. 데이터 계약

### 6.1 문항 메타데이터

```ts
type ResponseLayer = "REPORTED_FIRST_ORIENTATION" | "ENACTED_RESPONSE";

type ScoringRole = "CORE_TRAIT_EVIDENCE" | "PROFILE_MODIFIER" | "RESEARCH_ONLY";

type ProcessPairMetadata = {
  responseLayer: ResponseLayer;
  scoringRole: ScoringRole;
  processPairGroupId: string | null;
  profileDirection: "HIGH" | "LOW";
  probeTriggerRuleId: string | null;
};
```

콘텐츠 원장 필드:

- `response_layer`
- `scoring_role`
- `process_pair_group_id`
- `profile_direction`
- `probe_trigger_rule_id`
- `process_profile_model_version`

`keyed_direction`은 facet 채점 방향이고 `profile_direction`은 다층 비교에서 정규화할 방향이다. 둘이 같은 경우가 많아도 의미를 숨기기 위해 하나로 합치지 않는다.

### 6.2 필드 이름 전환 규칙

`AUTOMATIC_ORIENTATION`은 자기보고만으로 자동·무의식 과정을 측정한 것처럼 읽힐 수 있어 v0.2부터 사용하지 않는다.

```text
legacy: AUTOMATIC_ORIENTATION
current: REPORTED_FIRST_ORIENTATION
```

- 신규 콘텐츠와 신규 응답은 `REPORTED_FIRST_ORIENTATION`만 기록한다.
- 이전 후보 콘텐츠를 가져올 때만 legacy 값을 current 값으로 매핑한다.
- 원본의 `process_profile_model_version`은 보존해 어느 계약으로 수집됐는지 추적한다.
- 실제 DB enum·check constraint 변경은 측정 릴리스가 승인된 뒤 versioned migration으로 수행한다.
- UI 문구는 내부 enum을 그대로 번역하지 않고 `처음 드는 생각`, `실제 나타나는 반응`을 사용한다.

### 6.3 이유 probe 콘텐츠

별도 versioned 콘텐츠로 관리한다.

- `probe_id`
- `probe_version`
- `trigger_rule_id`
- `prompt_text`
- `option_set_id`
- `allow_multiple`
- `allow_skip`
- `content_status`
- `validation_status`

### 6.4 이유 응답

운영 DB migration 후보:

- `attempt_id`
- `probe_id` / `probe_version`
- `trigger_model_version`
- `selected_option_ids`
- `response_status: VALID | SKIPPED`
- `created_at`

상황 라벨·질문 원문을 응답 row에 복제하지 않는다. 사용자가 고른 이유를 자유 서술로 기본 수집하지 않아 불필요한 민감정보 입력을 줄인다.

## 7. 결과 리포트 계약

정밀 리포트의 RO 카드에서 충분한 증거가 있을 때만 다음 계층을 사용한다.

```text
처음 드는 생각
원인과 해결할 부분이 먼저 눈에 들어와요.

실제 나타나는 반응
상대에게 미칠 영향을 고려해 마음을 먼저 확인하는 편이에요.

내가 직접 고른 이유
내 말이 상대에게 미칠 영향을 고려해서
```

금지:

- `속마음은 차갑지만 겉으로 배려해요`
- `가식적으로 공감해요`
- `사회성이 좋아서 감정을 관리해요`
- `본성은 G지만 훈련으로 A가 됐어요`
- 사용자가 고르지 않은 동기를 자동 생성

다층 결과는 본성·가면을 나누는 기능이 아니라, 처음 드는 생각과 실제 나타나는 반응이 항상 같지는 않다는 사실을 설명하는 기능이다.

## 9. 사용자 승인 기록

2026-07-18 KST 사용자 승인으로 다음 방향을 확정했다.

1. 원인·결과가 먼저 궁금해도 상대에게 미칠 영향을 고려해 마음을 먼저 확인하는 사람을 구분할 수 있어야 한다.
2. 처음 드는 생각과 실제 나타나는 반응을 하나의 문항·점수로 합치지 않는다.
3. 상세한 차이는 정밀 리포트에서 설명하고 빠른 결과에서는 확정하지 않는다.
4. 반응 이유는 사용자가 직접 선택했을 때만 비채점 맥락으로 사용한다.
5. 이 구조는 새 코드 축이 아니라 기존 성향 해석의 해상도를 높이는 보조 프로필이다.
6. 고객 문구는 `처음 드는 생각 → 실제 나타나는 반응`으로 사용한다.
7. 대표 G/A와 RO-EC 관계 반응 흐름은 동일한 `원인·해결 ↔ 상대 마음` 의미축을 사용한다.
8. RO-RN은 G/A에 합산하지 않는 본인 전용 상세 신호이며 공개·공유·비교에서 기본 제외한다.
