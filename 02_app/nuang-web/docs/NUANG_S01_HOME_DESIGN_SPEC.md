# NUANG S01 Home Design Specification

작성일: 2026-07-17 KST  
상태: 디자인 v1 보존, 첫 진입 구조 변경에 따른 v2 재설계 대기  
기준: `NUANG_UI_UX_MASTER_DESIGN_BLUEPRINT.md`, `NUANG_APP_FUNCTIONAL_REQUIREMENTS.md`

## 0. 2026-07-17 변경 기록

첫 방문자는 홈으로 바로 진입하지 않고 `S00O 첫 실행 가이드 → 빠른 코어 → 간단 결과 리포트`를 먼저 경험한다. 따라서 이 문서의 `첫 방문·결과 없음` 대표 프레임은 온보딩을 건너뛰고 홈을 먼저 둘러본 예외 상태로만 유지한다.

홈 v2는 검사 후 돌아오는 개인화 허브로 재설계한다. 최근 결과, 오늘의 플레이, 코드별 익명 통계, 피드 대화, 코드 탐험을 카드 위계로 제공하며 세부 기준은 `NUANG_S00_FIRST_RUN_ONBOARDING_DESIGN_SPEC.md` 7장을 따른다.

## 1. 화면 목표

홈은 앱 메뉴판이 아니라 사용자의 현재 상태에 가장 적합한 다음 행동 하나를 제안하는 시작점이다.

온보딩을 마치거나 홈 먼저 둘러보기를 선택한 사용자는 다음을 이해해야 한다.

1. 뉴앙은 검사를 통해 나를 이해하는 서비스다.
2. 빠른 코어는 로그인 없이 약 3분 안에 시작할 수 있다.
3. 검사 후에는 5글자 코드와 생활 속 해석이 열린다.
4. 피드에서는 다른 사람의 생각을 읽을 수 있다.
5. 직접 응답과 원점수는 공유되지 않는다.

## 2. 성공 기준

- 사용자가 5초 안에 `빠른 코어 시작`을 찾는다.
- 사용자가 실행 후 20초 안에 첫 문항 화면으로 이동할 수 있다.
- primary action은 모든 상태에서 하나다.
- 탭이나 메뉴를 설명하는 반복 카드 목록이 없다.
- 로그인하지 않아도 검사를 시작하고 피드를 읽을 수 있다는 점을 이해한다.
- 공유되지 않는 정보의 범위를 화면 하단에서 짧게 확인할 수 있다.

## 3. 화면에서 하지 않는 일

- 로그인이나 계정 연결을 먼저 요구하지 않는다.
- 모든 기능으로 가는 바로가기 목록을 만들지 않는다.
- 지원되지 않는 알림, 메시지, 팔로잉 기능을 암시하지 않는다.
- 빠른 코어 결과를 1:1 비교 가능 상태로 표현하지 않는다.
- 결과가 없는 사용자에게 임의의 뉴앙 코드를 보여주지 않는다.
- 개인정보 안내를 긴 정책 설명으로 만들지 않는다.

## 4. 대표 프레임

- 상태: 온보딩 우회·결과 없음
- 기준 폭: 390px
- 검증 폭: 320px, 360px, 390px
- 레이아웃: 단일 세로 스크롤
- 하단 내비게이션: 홈, 피드, 검사, 성향지도, 마이
- 캐릭터 역할: 빠른 코어 시작을 돕는 첫 안내자

## 5. 첫 방문 정보 계층

### 5.1 상단

- 좌측: `NUANG`
- 우측 맥락: `오늘의 나`
- 알림, 메시지, 로그인 버튼은 노출하지 않는다.

### 5.2 첫 행동 영역

- eyebrow: `처음 오셨나요?`
- 제목: `오늘 나를 알아가는 3분`
- 설명: `부담 없는 20문항으로 첫 뉴앙 코드와 나의 특징을 확인해보세요.`
- primary CTA: `빠른 코어 시작`
- 보조 정보: `로그인 없이 시작 · 진행 내용 7일 보관`

사용자는 이 영역만 보고도 첫 가치를 시작할 수 있어야 한다.

### 5.3 서비스 이해 영역

제목은 `설명보다 한 번의 경험으로 이해해요`를 사용한다.

세 단계는 독립 기능 카드가 아니라 하나의 경험 순서로 표현한다.

1. 검사 — 내 성향 발견
2. 코드 — 5글자로 기억
3. 피드 — 다른 생각 탐색

### 5.4 피드 미리보기

- label: `오늘의 질문`
- 질문: `혼자 있고 싶은 날, 나는 어떤 신호를 보내나요?`
- 설명: `결과가 없어도 읽을 수 있고, 반응하거나 답할 때만 로그인이 필요해요.`
- 이동: `피드 미리보기`

피드는 첫 CTA와 경쟁하지 않도록 첫 행동 영역 아래에 둔다.

### 5.5 신뢰 안내

- 제목: `응답은 나만 볼 수 있어요`
- 설명: `문항 응답, 원점수, 민감한 정보는 공유·피드·비교에 사용하지 않아요.`

정책 전체를 설명하지 않고 사용자가 가장 먼저 걱정하는 정보만 확인시킨다.

## 6. 홈 상태별 계약

| 상태 | 제목 | primary action | secondary action | 핵심 보조 정보 |
|---|---|---|---|---|
| 첫 방문 | 오늘 나를 알아가는 3분 | 빠른 코어 시작 | 없음 | 로그인 없이 시작, 7일 이어보기 |
| 진행 중 검사 | 조금만 더 가면 첫 코드가 보여요 | N번 문항부터 이어하기 | 없음 | 응답 수, 전체 문항 수, 마지막 저장 시각 |
| 빠른 결과 | 첫 뉴앙 코드를 다시 만나보세요 | 결과 다시 보기 | 정밀 코어로 공식 코드 완성하기 | 빠른 결과는 예비 코드이며 비교 기준이 아님 |
| 정밀 결과 | 지금의 나를 다시 읽어볼까요? | 내 성향 다시 보기 | 오늘의 질문 보기 | 최근 결과와 선명해진 부분 |
| 로딩 | 화면 구조를 유지한 skeleton | 없음 | 없음 | `aria-busy=true`, 상태 추측 금지 |
| 오프라인 | 연결 없이도 첫 검사를 시작할 수 있어요 | 빠른 코어 시작 또는 로컬 이어하기 | 없음 | 피드만 일시 중단, 로컬 기능 유지 |

## 7. 상태 결정 우선순위

1. 진행 중 로컬 검사가 있으면 이어하기를 우선한다.
2. 방금 완료했지만 아직 충분히 읽지 않은 결과가 있으면 결과 다시 보기를 우선한다.
3. 빠른 코어만 있으면 정밀 코어 확장을 보조 행동으로 제공한다.
4. 정밀 코어가 있으면 내 성향과 최근 선명해진 부분을 우선한다.
5. 결과가 없으면 빠른 코어 시작을 제안한다.
6. 서버 피드 실패가 로컬 검사와 결과를 막아서는 안 된다.

## 8. 상호작용

### 8.1 Primary CTA

- 첫 방문 또는 오프라인: 빠른 코어 소개 화면으로 이동한다.
- 진행 중: 저장된 현재 문항으로 이동한다.
- 빠른 결과: 해당 결과 리포트를 다시 연다.
- 정밀 결과: 내 성향 상세를 연다.

### 8.2 Secondary CTA

- 빠른 결과: 정밀 코어 소개로 이동한다.
- 정밀 결과: 오늘의 질문이 있는 피드 문맥으로 이동한다.
- 같은 영역에서 secondary는 하나만 허용한다.

### 8.3 피드 미리보기

- 질문 preview를 누르면 해당 피드 항목 또는 피드 첫 위치로 이동한다.
- 비회원도 읽을 수 있다.
- 반응, 댓글, 투표 시점에만 로그인한다.

### 8.4 하단 내비게이션

- 현재 홈 탭은 텍스트와 아이콘 상태, `aria-current=page`로 표시한다.
- 모든 탭은 최소 44px 터치 영역을 가진다.
- 본문과 겹치지 않도록 safe area를 포함한다.

## 9. 시각 규칙

- 배경은 따뜻한 neutral을 기본으로 한다.
- 첫 행동 영역만 매우 옅은 motif surface로 구분한다.
- 카드 대신 full-width section과 얇은 경계를 사용한다.
- primary CTA 외에는 큰 채움 버튼을 사용하지 않는다.
- 캐릭터는 첫 행동 영역에서만 사용한다.
- gradient, glow, blur, 중첩 카드, 컬러 아이콘 박스를 사용하지 않는다.
- 하단 탭 라벨은 320px에서도 한 줄로 유지한다.

## 10. 접근성

- 화면의 h1은 상태별 핵심 제목 하나다.
- 각 섹션은 h2와 `aria-labelledby`로 구분한다.
- 로딩 중에는 `aria-busy=true`와 status 영역을 제공한다.
- 검사 진행률은 `role=progressbar`, min/max/now 값을 가진다.
- 아이콘은 의미가 중복되면 `aria-hidden=true`로 처리한다.
- 캐릭터가 순수 장식으로 확정되면 구현에서는 빈 alt를 사용한다.
- 키보드 focus 순서는 primary CTA → secondary CTA → 피드 preview → 하단 탭이다.
- `prefers-reduced-motion`에서는 진행률 전환 애니메이션을 끈다.

## 11. 개인정보와 데이터

홈에서 읽을 수 있는 데이터:

- 로컬 진행 중 검사 종류, 응답 수, 전체 문항 수, 마지막 저장 시각
- 최근 완료 결과의 코드, 코드 이름, 결과 route
- 공개 projection 기반 피드 preview

홈에서 절대 읽거나 표시하지 않는 데이터:

- 직접 문항 응답
- 원점수와 전체 점수 벡터
- OAuth provider 원문이나 token
- 이메일과 계정 식별 정보
- 민감 검사 결과
- 도움 연결 허브 이용 기록

## 12. 측정 이벤트

| 이벤트 | 발생 시점 | 필요한 속성 |
|---|---|---|
| `home_viewed` | 홈 표시 완료 | home_state, auth_state, online_state |
| `home_primary_clicked` | 상태별 primary 선택 | home_state, target |
| `home_secondary_clicked` | 보조 행동 선택 | home_state, target |
| `home_feed_preview_clicked` | 질문 preview 선택 | item_id, item_kind |
| `home_bottom_tab_clicked` | 하단 탭 선택 | from, to |

직접 응답, 원점수, 사용자 식별 정보는 분석 속성에 포함하지 않는다.

## 13. 디자인 검증 결과

- 첫 방문, 진행 중, 빠른 결과, 정밀 결과, 로딩, 오프라인 6개 상태 전환 확인
- 320px, 360px, 390px에서 수평 overflow 없음
- 320px에서 캐릭터와 제목 영역 겹침 없음
- 320px에서 하단 탭 라벨 잘림 없음
- CTA와 하단 내비게이션이 화면 폭 안에 유지됨
- 로딩 상태의 `aria-busy` 확인
- 진행 상태의 progressbar 값 갱신 확인
- 브라우저 console error와 warning 없음

## 14. 구현 acceptance criteria

- 첫 방문 화면에는 `빠른 코어 시작`만 primary로 보인다.
- 진행 중 attempt가 있으면 응답 수 기반 진행률과 이어하기가 보인다.
- 빠른 결과에는 비교 가능 표현이 없다.
- 정밀 결과 상태에서만 비교에 필요한 기반이 갖춰졌음을 다른 문맥에서 사용할 수 있다.
- 관심 없음 처리된 피드 항목은 preview에서 제외된다.
- 서버 피드 실패 시 seed fallback 또는 명시적인 오프라인 상태를 제공한다.
- 로컬 상태 로딩 전 잘못된 추천을 순간적으로 노출하지 않는다.
- 360px에서 텍스트, CTA, 하단 내비게이션이 겹치지 않는다.
- 직접 응답, 원점수, 민감 항목을 DOM이나 분석 payload에 포함하지 않는다.

## 15. 다음 단계

이 문서는 v1 기준으로 보존한다. 다음 작업은 `S00O-G01` 승인 후 `S00O-G02`이며, 네 장의 가이드와 첫 가치 여정이 확정되면 홈 v2를 다시 디자인한다. 앱 코드 반영은 디자인 화면 승인 순서가 끝난 뒤 별도 구현 단계에서 진행한다.
