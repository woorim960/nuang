## 0. 목표

검사 준비·응답 보관·결과 생성이 지연되거나 실패해도 답을 잃었다는 불안을 만들지 않고, 사용자가 현재 상태와 안전한 다음 행동을 즉시 이해하게 한다.

## 1. 상태 계약

| 상태           | 사용자에게 알릴 사실                                | 주 행동         | 보조 행동         | 진행 조건                                        |
| -------------- | --------------------------------------------------- | --------------- | ----------------- | ------------------------------------------------ |
| 질문 준비 지연 | 아직 답한 내용은 없고 질문 준비가 늦어지고 있음     | `다시 불러오기` | `검사 목록으로`   | 질문 release를 다시 확인한 뒤 첫 문항으로 이동   |
| 응답 보관 실패 | 선택은 화면에 남아 있지만 로컬 보관은 확인되지 않음 | `다시 시도`     | 이전 문항         | 보관 성공 전 `다음` 차단                         |
| 결과 생성 지연 | 완료 응답은 안전하게 보관됐고 결과 job이 진행 중임  | `홈으로 가기`   | `여기서 기다리기` | 홈의 진행 중 결과 카드 또는 현재 화면에서 재확인 |
| 결과 생성 실패 | 완료 응답은 안전하지만 결과 job이 실패함            | `다시 만들기`   | `홈으로 가기`     | 기존 완료 응답으로 결과 job만 재시도             |

## 2. 질문 준비 지연

- app bar는 `준비가 지연되고 있어요`, 본문은 `질문을 준비하는 데 / 시간이 걸리고 있어요`로 표시한다.
- 뉴앙 캐릭터·핵·성향 신호 motion은 작업이 계속되는 동안 유지한다.
- `다시 불러오기`를 누르면 `질문을 다시 / 불러오고 있어요`로 바뀌고 중복 요청을 막는다.
- 아직 응답이 없으므로 하단에는 `아직 답한 내용은 없어요.`를 표시한다.
- 장기 지연이 실패로 확정되면 같은 위치에서 오류 원인을 일반화해 안내하고 다시 불러오기를 유지한다.

## 3. 응답 보관 실패

- 선택한 응답·현재 문항·진행률·하단 dock 위치를 그대로 유지한다.
- overlay card는 `응답을 보관하지 못했어요`와 `다시 시도`를 제공한다.
- `다시 시도하기 전에는 다음 질문으로 넘어가지 않아요.`라고 차단 이유를 명확히 알린다.
- 로컬 보관 성공이 확인되기 전에는 `다음`을 비활성화한다.
- 성공하면 `응답을 보관했어요`를 확인한 뒤 overlay를 닫고 `다음`을 활성화한다.
- 메모리에만 남은 선택을 `기기에 임시 보관했어요`라고 표현하지 않는다.

## 4. 결과 생성 지연

- `답 보관 완료`를 먼저 보여 결과 지연과 데이터 손실을 구분한다.
- 제목은 `결과를 만드는 데 / 시간이 걸리고 있어요`다.
- 사용자를 화면에 가두지 않고 `홈으로 가기`를 주 행동으로 제공한다.
- `여기서 기다리기`를 선택하면 캐릭터·핵·성향 신호 motion을 유지한다.
- 홈 이동을 제공하려면 홈에 `진행 중 결과` 카드를 구현하고 같은 `assessment_attempt_id`와 결과 job을 조회해야 한다.
- 홈 카드가 구현되지 않은 release에서는 홈 이동 문구를 노출하지 않는다.

## 5. 결과 생성 실패

- `답 보관 완료`는 유지하고 제목을 `결과를 아직 / 만들지 못했어요`로 바꾼다.
- 작업이 멈춘 상태이므로 캐릭터·핵·성향 신호 반복 motion도 멈춘다.
- `다시 만들기`는 검사 재응답이 아니라 기존 완료 응답으로 결과 job만 재시도한다.
- 연속 탭과 재진입은 같은 idempotency key를 사용해 중복 결과를 만들지 않는다.
- 재시도 후에는 결과 생성 지연 상태로 돌아가고 완료되면 간단 결과로 자동 이동한다.

## 6. 구현 상태 필드

| 필드                       | 값 예시                                              | 역할                                 |
| -------------------------- | ---------------------------------------------------- | ------------------------------------ |
| `question_load_status`     | `pending · ready · delayed · failed`                 | 질문 준비 지연·재시도 판단           |
| `local_persist_status`     | `pending · saved · failed`                           | 다음 문항 차단 여부                  |
| `completion_status`        | `idle · submitting · completed · failed`             | 완료 응답 보관 여부                  |
| `result_generation_status` | `queued · processing · delayed · completed · failed` | 결과 화면·motion·행동 결정           |
| `result_job_id`            | opaque id                                            | 홈·현재 화면에서 같은 결과 작업 조회 |
| `completion_request_id`    | idempotency key                                      | 완료·결과 중복 생성 방지             |
| `result_retry_count`       | integer                                              | 운영 관측과 재시도 제한              |
| `last_error_code`          | internal enum                                        | 사용자 문구와 분리된 진단 정보       |

필드와 저장 위치는 `RBL-07` versioned migration에서 확정한다. 내부 오류 코드·stack·route 이름은 고객 화면에 노출하지 않는다.

## 7. 접근성·반응형

- 지연 상태는 `aria-live=polite`, 응답 보관 실패 overlay는 `role=alert`를 사용한다.
- 차단 여부는 색이 아니라 비활성화된 `다음`과 설명 문구로 함께 전달한다.
- 결과 생성 실패에서는 motion이 멈춰도 제목·주 행동으로 상태를 이해할 수 있어야 한다.
- 390px light와 320px dark에서 네 상태 모두 수평 overflow 없이 주 행동이 보인다.
- 응답 보관 실패 overlay는 240ms 전환 후 phone shell 안에 완전히 들어오며 본문을 재배치하지 않는다.

## 8. 이번 검토 범위

1. 네 복구 상태의 문구와 행동 우선순위
2. 응답 보관 실패의 blocking 원칙
3. 결과 생성 지연에서 안전한 홈 복귀
4. 결과 생성 실패에서 검사 재응답 없는 재시도
5. 지연에서는 motion 유지, 실패에서는 motion 정지

## 9. 사용자 승인 기록

`UIX-01`을 완료하고 `UIX-02`의 S04-R 빠른 결과 리포트 화면으로 이동한다.
