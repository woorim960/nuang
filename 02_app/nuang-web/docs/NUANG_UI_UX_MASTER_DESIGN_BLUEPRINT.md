# NUANG UI/UX Master Design Blueprint

작성일: 2026-07-17 KST  
기준 문서: `NUANG_APP_FUNCTIONAL_REQUIREMENTS.md`  
목적: 기능 요구사항을 고객 만족, 첫 사용 이해도, 신뢰, 흥미, 체류, 재방문을 높이는 실제 화면 설계 순서로 변환한다.

## 0. 최종 방향 한 문장

뉴앙은 `짧은 캐릭터 가이드로 이해하고 → 3분 안에 나를 읽고 → 5글자 코드로 기억하고 → 실제 모습으로 이해하고 → 다른 사람의 생각과 선택을 살펴보며 다시 돌아오는 성향 기반 SNS`로 설계한다.

첫 경험의 목표는 앱 전체를 설명하는 것이 아니다. 네 장 이내의 뉴앙 캐릭터 가이드로 서비스 가치를 빠르게 이해하고, 로그인 없이 첫 문항에 답하고, 결과에서 “나를 꽤 잘 설명한다”는 감정을 느끼며, 공개 범위가 안전하다는 확신을 얻고, 자연스럽게 개인화 홈과 피드의 다음 콘텐츠로 이동하게 만드는 것이다.

## 1. 요구사항에서 도출한 제품 설계 결론

| 로그인 없이 검사 시작과 로컬 결과 확인  | 강제 회원가입 없이 네 장 이내의 첫 실행 가이드 후 빠른 코어를 시작한다. 로그인은 저장·공유·작성·비교 시점에 요청한다.                    | 첫 실행 로그인 벽, 긴 기능 투어, 로그인 장점 장문 설명         |

## 2. 현재 구현과 교차 점검한 우선 개선 위험

이 항목은 기능을 바꾸자는 제안이 아니라, 새 화면 설계에서 반드시 바로잡을 기대 불일치다.

1. 홈의 여러 메뉴 바로가기는 사용자의 `다음 행동 하나`보다 앱 구조를 더 강하게 보여준다. 새 홈은 상태별 주 행동 하나와 맥락 콘텐츠만 남긴다.
2. 현재 피드의 `팔로잉`과 메시지 아이콘은 요구사항에 없는 팔로우·DM 기능을 기대하게 만든다. 기능이 실제로 열리기 전에는 노출하지 않는다.
3. 빠른 코어는 1:1 비교 기준이 될 수 없으므로, 빠른 결과 보유자에게 `비교 가능`으로 보이는 표현을 금지한다.
4. 피드 리포트 preview와 공개 공유 화면에서는 숫자 점수나 원점수처럼 해석될 수 있는 값을 노출하지 않는다. 시각 요약도 공개 projection만 사용한다.
5. 결과 리포트의 숫자는 사용자가 우열이나 정확도로 읽기 쉽다. 본인 결과에서도 생활 언어와 상대적 방향을 먼저 보여주고, 숫자는 꼭 필요한 경우에만 낮은 위계로 제한한다.
6. 성향지도는 정적 코드 목록이 아니라 `32개 코드 세계 탐색 → 코드 상세 → 내 코드와의 연결`이 가능해야 한다.
7. 피드 저장 기능은 회수 경로가 있어야 의미가 완성된다. `저장한 글` 화면은 기능 요구사항 확장 승인을 거쳐 P1 후보로 둔다.
8. 루트 화면과 피드 화면의 폭, 상단바, 배경, 여백 리듬이 다르다. 공통 앱 셸을 먼저 확정한 뒤 화면별 예외를 최소화한다.

## 3. 뉴앙이 만들어야 하는 다섯 가지 고객 감정

1. **즉시성**: “회원가입 없이 바로 해볼 수 있네.”
2. **몰입감**: “한 문항씩이라 부담 없고, 어디까지 했는지 명확하다.”
3. **공감**: “결과가 나를 단정하지 않으면서도 생활 속 내 모습을 잘 설명한다.”
4. **안심**: “무엇이 공개되고 무엇이 절대 공개되지 않는지 이해된다.”
5. **호기심**: “나와 비슷한 코드의 사람들은 어떻게 생각할까? 다음 검사도 해보고 싶다.”

시각적 화려함보다 이 다섯 감정의 연속성이 고객 만족도를 좌우한다.

## 4. 핵심 사용자와 해야 할 일

### 4.1 처음 온 탐색형 사용자

- 목적: 짧은 시간 안에 재미있는 자기이해 결과를 얻고 싶다.
- 불안: 회원가입, 긴 검사, 개인정보 수집, 뻔한 결과.
- 성공 조건: 실행 후 20초 안에 첫 문항에 도달하고, 빠른 코어를 중단 없이 완료한다.

### 4.2 자기이해 반복형 사용자

- 목적: 이전 결과를 다시 보고, 다른 생활 주제로 이해를 구체화하고 싶다.
- 불안: 결과가 매번 흔들리거나 과학적으로 과장되어 보이는 것.
- 성공 조건: 홈에서 이어할 행동을 바로 찾고, 최근 선명해진 부분을 이해한다.

### 4.3 사회적 호기심 사용자

- 목적: 다른 사람이 어떤 선택을 하는지 보고 가볍게 공감하고 싶다.
- 불안: 공격적 커뮤니티, 과도한 자기노출, 사람을 유형으로 낙인찍는 문화.
- 성공 조건: 피드에서 안전한 질문에 반응하고, 익명 집계와 공개 범위를 이해한다.

### 4.4 개인정보 민감 사용자

- 목적: 서비스는 써보고 싶지만 응답과 민감 정보는 노출하고 싶지 않다.
- 불안: 문항 응답·원점수·계정 정보가 공유나 비교에 쓰이는 것.
- 성공 조건: 필요한 순간에 짧고 정확한 공개 범위 안내를 보고 스스로 선택한다.

## 5. 전체 경험 루프

```text
첫 실행 캐릭터 가이드
→ 로그인 없이 빠른 코어 시작
→ 한 문항씩 답변, 자동 저장
→ 예비 코드와 생활 장면 중심 결과
→ 정밀 코어 또는 개인화 홈으로 선택 확장
→ 홈의 오늘의 플레이·코드 탐험·피드 맥락
→ 피드에서 읽기·반응·저장
→ 마이에서 내 코드와 리포트 재확인
→ 무료 주제 검사로 해석 구체화
→ 다음 방문 홈에서 가장 적절한 행동 하나 추천
```

이 루프에서 로그인은 출발점이 아니라 사용자의 의도가 서버 권한을 필요로 하는 순간에만 등장한다.

## 6. 정보 구조

### 6.1 하단 내비게이션

| 탭       | 사용자의 질문                         | 주 역할                                        | 제외                     |
| -------- | ------------------------------------- | ---------------------------------------------- | ------------------------ |
| 홈       | 지금 무엇을 하면 좋지?                | 상태 기반 다음 행동 하나, 최근 결과, 피드 맥락 | 모든 메뉴 바로가기 목록  |
| 피드     | 다른 사람들은 어떻게 느끼고 선택하지? | 읽기, 반응, 댓글, 저장, 질문, 투표             | 검사·설정·비교 전용 메뉴 |
| 검사     | 무엇을 더 알아볼 수 있지?             | 코어, 무료 주제, 연구소, 도움 허브 진입        | 피드 복제, 리포트 관리   |
| 코드지도 | 32개 뉴앙 코드는 어떻게 다르지?       | 코드 세계 탐색, 코드 상세, 내 코드 연결        | 개인 결과 전체 리포트    |
| 마이     | 내 결과와 권리를 어디서 관리하지?     | 대표 코드, 내 리포트, 계정, 공개 범위, 데이터  | 피드·검사 홈 복제        |

### 6.2 맥락에서만 시작하는 기능

- 결과 공유: 결과 리포트의 공유 행동에서 시작한다.
- 피드 리포트 공유: 결과 리포트의 `피드에 공유`에서 시작한다.
- 1:1 비교: 상대 공개 프로필 팝업의 `나와 비교하기`에서 시작한다.
- 로그인: 저장, 공유, 글쓰기, 댓글, 반응, 투표, 비교처럼 권한이 필요한 순간에 시작한다.
- 도움 연결: 검사 허브의 안전한 보조 경로와 관련 안내에서 시작한다.

### 6.3 홈 상태 결정 규칙

우선순위가 가장 높은 항목 하나만 주 행동으로 노출한다.

1. 로컬에 진행 중 검사가 있으면 `이어하기`.
2. 방금 완료한 결과를 아직 충분히 보지 않았다면 `결과 계속 보기`.
3. 빠른 코어만 있고 정밀 코어가 없다면 `공식 코드 완성하기`.
4. 결과가 있고 오늘의 질문이 있으면 `오늘의 질문 보기/답하기`.
5. 결과가 전혀 없으면 `빠른 코어 시작`.

로딩 중에는 선택을 추측하지 않고 같은 영역 크기의 skeleton을 보여준다. 오프라인이어도 로컬 검사와 로컬 결과는 열 수 있고, 서버 기능만 재시도 상태를 표시한다.

## 7. 뉴앙 고유 디자인 언어

### 7.1 디자인 콘셉트: `차분한 관찰, 선명한 발견`

- 기본 화면은 따뜻한 무채색 배경, 얇은 경계, 넉넉한 여백으로 신뢰를 만든다.
- 발견해야 할 코드, 문장, 행동에만 선명한 색을 쓴다.
- 종이 설문처럼 정적이고 평면적인 화면을 그대로 옮기지 않는다. 레이어, 전환, 직접 조작 피드백을 사용해 모바일 앱다운 깊이와 반응성을 만든다.
- 화면을 카드로 과도하게 포장하기보다, 핵심 정보에만 역할이 분명한 카드와 인터랙티브 표면을 사용한다.
- 캐릭터는 장식이 아니라 감정의 초점, 결과의 주인공, 빈 상태 안내 중 하나의 역할만 맡는다.

### 7.2 브랜드 시그니처

1. **코드 워드마크**: 5글자 코드 전체를 큰 활자로 보여주고, 설명 중인 자리만 motif 색과 굵기로 강조한다.
2. **성향 라인**: 진행, 코드 자리, 투표 비율, 세부 신호에 공통으로 쓰는 얇은 선형 표현이다. 숫자 경쟁처럼 보이지 않게 라벨과 문장을 먼저 둔다.
3. **생활 장면 문장**: 추상 형용사보다 `모임 뒤`, `마음이 흔들릴 때`, `계획이 바뀔 때`처럼 사용자가 기억을 떠올릴 수 있는 장면을 쓴다.
4. **네 가지 motif**: 불꽃, 햇살, 물결, 숲은 코드 세계의 구분과 캐릭터에만 사용하고, 일반 버튼 색으로 남발하지 않는다.

### 7.3 시각 시스템 기준

- 모바일 기준 폭: 360, 390, 430px. 웹 최대 콘텐츠 폭은 520px를 기본으로 검증한다.
- 터치 영역: 최소 44×44px, 핵심 버튼은 48~52px 높이를 기본으로 한다.
- radius: 온보딩에서 승인된 부드러운 인상을 이어간다. 주 버튼은 14~~16px, 주요 컨텐츠 표면은 16~~20px, 반복 목록은 12~16px를 기본 후보로 검증한다. 원형은 아바타와 아이콘 버튼에만 쓴다.
- 색상: warm ivory canvas + pale lavender surface/deck + NUANG purple primary action을 기본으로 하고, motif 4색은 코드·비교·결과 표현에만 제한한다.
- 타이포: 큰 코드, 화면 제목, 섹션 제목, 본문, 보조 문구의 5단계로 제한한다. 긴 설명은 2~3문장 이상 연속하지 않는다.
- 아이콘: lucide 계열의 단일 선 스타일. 아이콘을 반복적으로 컬러 박스에 넣지 않는다.
- 그림자: 카드 구분은 색 차이와 얇은 경계를 우선하고, 모달·sheet·고정 내비게이션·직접 드래그하는 표면에만 약하게 사용한다.
- 모션: 기본 상태 전환은 160~~240ms, 화면·sheet 전환은 240~~320ms 범위에서 검증한다. 선택 피드백, 진행 변화, 결과 reveal, 직접 조작의 인과를 설명할 때 사용하며 `prefers-reduced-motion`을 존중한다.
- safe area: 하단 내비게이션, 검사 다음 버튼, bottom sheet 모두 기기 safe area를 포함한다.

### 7.4 감각적이되 신뢰를 지키는 동적 경험 원칙

뉴앙의 동적 요소는 사용자를 흥분시키기 위한 장식이 아니라 `내 행동이 반영되었다`, `어디까지 왔다`, `무엇이 새롭게 드러났다`를 즉시 이해시키는 피드백이어야 한다.

#### 적극적으로 사용하는 요소

- 응답 선택 시 카드의 색·경계·크기가 부드럽게 반응하고, 다음 행동이 활성화되는 연결 피드백
- 좌우 스와이프, bottom sheet, 카드 펼치기처럼 모바일에서 익숙한 직접 조작
- 문항 진행에 따라 자연스럽게 채워지는 segment와 짧은 숫자 전환
- 결과에서 코드 다섯 글자와 생활 장면이 정보 위계 순서대로 나타나는 staged reveal
- 홈과 피드에서 카드의 pressed·saved·voted 상태를 명확히 보여주는 micro-interaction
- 캐릭터의 시선·호흡·작은 제스처처럼 집중을 방해하지 않는 저빈도 ambient motion
- 사용자의 직접 행동이 성공했을 때만 제공하는 절제된 촉각·시각 피드백

#### 신뢰를 위해 제한하는 요소

- 응답을 특정 선택으로 유도하는 크기 변화, 색상 편향, 캐릭터 반응, 정답처럼 보이는 효과
- 검사 중 streak, 점수, 콤보, 경쟁, 보상 상자처럼 답변을 왜곡할 수 있는 게임화
- 채점 정확성을 과장하는 가짜 분석 파형, 의미 없는 긴 로딩, 무작위 숫자 증가
- 결과 공개 전 과도한 confetti, roulette, slot-machine 효과
- 한 화면에서 반복되는 bounce, glow, parallax, 자동 재생 장식
- 정보 읽기를 늦추거나 멀미를 유발하는 장거리 이동과 강한 탄성

#### 화면 유형별 강도

| 화면 유형      | 동적 강도 | 허용되는 표현                                            |
| -------------- | --------- | -------------------------------------------------------- |
| 온보딩·홈·피드 | 중간      | swipe, 카드 전환, 저빈도 캐릭터 motion, 저장·투표 피드백 |
| 검사 진행      | 낮음~중간 | 선택·진행·저장 상태만 즉시 반응, 응답 중립성 유지        |
| 채점 전환      | 낮음      | 실제 처리 상태와 짧은 브랜드 motion, 가짜 분석 연출 금지 |
| 결과 리포트    | 중간      | 정보 위계에 따른 reveal, 코드 탐색, 접고 펼치기          |
| 공유·비교·설정 | 낮음      | 권한·범위·성공·실패를 설명하는 기능적 전환               |

모든 핵심 정보와 행동은 애니메이션이 없어도 이해되고 완료되어야 한다. reduced motion에서는 이동을 fade·즉시 전환으로 대체하며, 기능과 정보 순서는 동일하게 유지한다.

### 7.5 문구 원칙

- 명령보다 동행: `정확히 분석합니다`보다 `현재까지 반복된 모습을 함께 살펴봐요`.
- 단정보다 현재성: `당신은`보다 `지금까지는 이런 모습이 자주 보였어요`.
- 기술보다 효용: `로컬 저장`보다 `이 기기에서 7일 동안 이어볼 수 있어요`.
- 불안 조장 금지: 정밀 코어를 `더 정확한 진짜 결과`로 판매하지 않는다.
- 오류는 다음 행동 포함: `실패했습니다`에서 끝내지 않고 `응답은 기기에 남아 있어요. 다시 시도`처럼 안내한다.

## 8. 신뢰를 만드는 정보 공개 설계

### 8.1 사용자가 이해해야 하는 세 층

1. **항상 비공개**: 직접 응답, 원점수, 전체 점수 벡터, 계정 식별 정보, 민감 검사, 도움 허브 이용 기록.
2. **기본 공개 가능**: 프로필, 대표 코드, 코드 이름, 코드 지도와 세부 신호의 제한된 요약.
3. **사용자가 행동할 때 생성**: 30일 공유 주소, 피드 리포트 preview, 공개 프로필 기반 비교.

### 8.2 안내 위치

- 검사 시작: `로그인 없이 시작·자동 저장`만 짧게 안내한다.
- 결과 공유 직전: 공유되는 항목과 공유되지 않는 항목을 2줄로 확인한다.
- 피드 작성 직전: 공개 범위와 moderation 상태를 자연어로 안내한다.
- 비교 직전: 상대가 공개한 정보만 사용하며 비공개 내용은 추정하지 않는다고 안내한다.
- 데이터 삭제 직전: 함께 삭제되는 공유·비교 기록을 명확히 알린다.

신뢰 안내를 모든 화면에 반복하면 오히려 불안감을 키우므로, 의사결정 시점에만 보여준다.

## 10. 화면을 하나씩 만드는 최종 순서

아래 S ID는 요구사항 추적을 위한 안정적인 화면 식별자다. 온보딩 승인 후 실제 작업 순서는 약속 이행과 수직 여정을 기준으로 재배치했으며, 최신 순서는 17장을 기준으로 한다. 각 S ID는 한 번의 독립적인 디자인·검토 단위로 유지한다.

### 10.0 문항 설계 후 복귀 오버라이드

[문항 설계 완료 후 UI/UX 복귀 계획](./NUANG_ITEM_DESIGN_TO_UI_UX_RETURN_PLAN.md)을 현재 실제 작업 순서로 사용한다. `RBL-01A~E`의 전 코어 문항 내부 설계와 통합 감사가 끝나면, 아래 전체 목록을 처음부터 다시 시작하지 않고 상황 라벨 설계로 중단됐던 `S03-R 검사 runner`에서 복귀한다.

복귀 순서:

```text
S03-R → S04-R → S05-R → S06-R → S07-R → S08-R
```

UI 설계는 M04~M07 전체 완료를 기다리지 않는다. 실제 동작 prototype을 M05 인지 인터뷰와 M06 파일럿에 사용하고, 최종 scoring·code·report copy만 승인된 measurement release에서 production binding한다.

## 11. 화면 하나를 디자인하는 고정 작업 절차

각 화면은 아래 순서로만 진행한다. 같은 화면의 상태는 한 작업 단위 안에서 함께 확정한다.

1. **화면 brief**: 대상 사용자, 진입 맥락, 사용자의 질문, 주 행동 하나, 성공 기준을 한 페이지로 작성한다.
2. **요구사항 추적**: 원문 요구사항, 금지 항목, 개인정보 항목, 필요한 API·데이터 상태를 연결한다.
3. **콘텐츠 계층**: 사용자가 3초, 10초, 30초에 각각 이해해야 할 내용을 정한다.
4. **저해상도 wireframe**: 360px 기준으로 primary action과 스크롤 흐름을 검증한다.
5. **상태 wireframe**: 로딩, 빈 상태, 오류, 오프라인, 인증, 만료 등 해당 화면의 상태를 추가한다.
6. **시각 디자인 v1**: 390px 기준 high fidelity 화면을 만든다.
7. **크기 검증**: 360px, 430px, 520px에서 줄바꿈, safe area, 고정 요소 겹침을 검증한다.
8. **상호작용 명세**: 탭, 스크롤, 키보드, 뒤로가기, optimistic update, 실패 복구, 모션을 정의한다.
9. **접근성 명세**: landmark, heading, aria label/live, focus order, 대비, reduced motion을 정의한다.
10. **문구 검수**: 내부 영어 키, 금지 용어, 진단·우열·과장 표현, 이모지를 검사한다.
11. **프로토타입 테스트**: 최소 5명의 과업 테스트 또는 동등한 내부 테스트로 이해도와 막힘을 확인한다.
12. **승인 snapshot**: 화면 버전, 결정 이유, 미해결 항목, 사용 컴포넌트를 기록하고 다음 화면으로 이동한다.

## 12. 화면별 필수 산출물

각 화면 작업이 끝날 때 아래가 모두 있어야 한다.

- 360px과 390px high-fidelity 시안
- 430px 또는 520px 확장 검증본
- 기본, 로딩, 빈 상태, 오류, 권한 상태 시안
- 클릭·스크롤·뒤로가기·고정 영역 동작 명세
- 최종 사용자-facing copy
- 사용 토큰과 공통 컴포넌트 목록
- 데이터 필드와 개인정보 공개 등급
- 접근성 메모
- 이벤트 측정 정의
- 개발 acceptance criteria
- 구현 후 브라우저 smoke 경로

## 13. 화면 완료 기준

다음 조건을 모두 통과해야 다음 화면으로 넘어간다.

### 이해도

- 처음 보는 사용자가 5초 안에 화면의 목적을 말할 수 있다.
- 사용자가 주 행동을 별도 설명 없이 찾을 수 있다.
- 같은 수준의 primary action이 둘 이상 경쟁하지 않는다.

### 신뢰

- 공개·저장·삭제의 결과를 행동 전에 이해할 수 있다.
- 직접 응답, 원점수, 계정 식별 정보, 민감 항목이 금지 표면에 없다.
- 결과와 비교 문구가 진단·순위·운명·관계 성공 예측처럼 보이지 않는다.

### 사용성

- 360px에서 텍스트와 버튼이 겹치거나 잘리지 않는다.
- 주요 터치 영역은 최소 44px이며 키보드와 화면 읽기 도구로 조작할 수 있다.
- 뒤로가기와 이탈 후 복귀가 데이터 손실 없이 예측 가능하다.
- 로딩·오류·오프라인 상태가 빈 화면이 아니며 다음 행동을 제공한다.

### 브랜드

- 캐릭터는 한 역할만 맡는다.
- 과한 gradient, glow, blur, 중첩 카드, 컬러 아이콘 박스, 반복 태그가 없다.
- 뉴앙 코드, motif, 생활 장면 언어가 일관되게 사용된다.
- 정적인 종이 설문처럼 보이지 않으며, 선택·진행·전환에 앱다운 반응이 있다.
- 모든 모션은 기능적 이유를 설명할 수 있고 reduced motion에서도 같은 과업을 완료할 수 있다.
- 검사 중 동적 표현이 특정 응답을 유도하거나 결과의 정확성을 과장하지 않는다.

## 14. 측정 계획

초기 수치는 실제 baseline을 얻은 뒤 조정한다. 먼저 아래 funnel과 정성 지표를 심는다.

### 첫 가치

- 앱 실행 → 빠른 코어 소개 도달률
- 실행 → 첫 응답까지 걸린 시간
- 빠른 코어 시작률과 완료율
- 중단 후 7일 이내 이어하기 성공률

### 결과 신뢰

- 결과 리포트 핵심 구간 도달률
- `이 결과가 나를 단정한다고 느꼈는가` 역문항
- 코드 이름과 생활 장면 회상률
- 공개/비공개 항목 이해도

### 체류와 재방문

- 피드 세션당 유의미한 읽기 수
- 반응·댓글·저장·투표 전환율
- 저장 후 재열람률
- 결과 리포트 7일 내 재방문률
- 무료 주제 검사 후 대표 결과 재열람률

### 만족

- 핵심 과업 성공률
- 과업 후 5점 만족도
- 신뢰도와 재미를 분리한 평가
- 앱을 다시 열 이유를 사용자가 자신의 말로 설명할 수 있는지

## 15. 구현 전환 절차

디자인 전체가 끝난 뒤 한 번에 코드를 바꾸지 않는다. 승인된 화면 순서대로 작은 수직 단위로 전환한다.

1. 화면의 acceptance criteria와 기존 route·API 계약을 대조한다.
2. 공통 토큰과 컴포넌트를 먼저 추가하되, 아직 승인되지 않은 화면까지 일괄 변경하지 않는다.
3. 한 화면과 그 상태를 구현한다.
4. unit/component test로 상태와 금지 문구를 검증한다.
5. 360px 모바일과 520px 웹에서 시각 비교한다.
6. 키보드, 화면 읽기, reduced motion, safe area를 확인한다.
7. 아래 검증을 실행한다.

```bash
npm run harness:check
npm run typecheck
npm run lint
npm run test
```

8. 큰 UI 단계가 끝날 때 `npm run qa:mvp`와 브라우저 smoke를 실행한다.
9. 승인 snapshot과 실제 구현 차이를 기록한 뒤 다음 화면으로 이동한다.

### 17.3 화면 하나씩 진행하는 방법

1. 먼저 W1을 하나의 기반 승인 단위로 완료한다. 이 단계에서 화면을 여러 개 만들지 않는다.
2. 다음 디자인 화면은 `S03 빠른 코어 진행`으로 확정한다.
3. S03 내에서 `첫 문항 → 응답 선택 → 이전·다음 → 판단하기 어려움 → 마지막 문항 → 저장 상태`를 한 번에 설계하고 승인한다.
4. 승인된 S03을 코드로 구현하고 시각 비교·접근성·저장 smoke를 통과한 뒤 S04로 이동한다.
5. 각 화면에서 발견된 공통 요소만 F08에 추가한다. 미리 모든 컴포넌트를 만들지 않는다.

### 17.4 시뮬레이션과 실제 구현의 기능 계약

- 시뮬레이션은 시각 방향과 핵심 상태를 빠르게 검토하기 위해 일부 주변 행동이 동작하지 않을 수 있다.
- 실제 구현에서는 화면에 노출한 모든 버튼·링크·선택지는 실제 기능, 명시적인 준비 중 상태, 또는 안전한 unavailable 처리 중 하나를 가져야 한다. 아무 반응 없는 컨트롤은 출시하지 않는다.
- S03 문항은 작성형 `contextLabel`과 질문 `text`를 분리한다. 상황 라벨은 작은 뉴앙 퍼플 텍스트로 질문 위에 표시하며, 내부 축명이나 채점 방향에서 자동 생성하지 않는다. 라벨과 질문은 같은 release 콘텐츠로 버전 관리하고 오프라인 이어하기에서도 동일하게 복원한다.
- 구현 시 `contextLabel`은 DB 문항 콘텐츠의 `context_label` 컬럼으로 관리한다. 현재 DB에는 canonical 문항 테이블이 없으므로 `assessment_response`에 복제하지 않고 버전별 content release/item 테이블을 추가한다. 세부 순서는 `NUANG_S03_CONTEXT_LABEL_DB_MIGRATION_PLAN.md`를 따른다.
- S03의 `판단하기 어려워요`는 장식 문구가 아니라 1~5 응답과 동등한 선택 상태다. 선택 즉시 자동 저장되고 다음 문항으로 이동할 수 있으며, 이전 이동·7일 이내 복귀 시 선택 상태가 복원되어야 한다.
- 자동 저장은 기본 동작이지 상시 노출할 콘텐츠가 아니다. 정상 상태의 `자동 저장` 표시는 제거하고, 저장 지연·실패·복구 필요 또는 검사 이탈 확인처럼 사용자의 판단이 필요한 순간에만 상태를 안내한다.
- 비로그인 코어 검사 진행 내용은 현재 브라우저·기기의 IndexedDB `nuang-local/assessmentAttempts`에 저장한다. 다른 기기나 브라우저로 자동 이동하지 않으며, 진행 중 결과는 7일 보관한다. 이 사실은 검사 화면에 반복하지 않고 시작·이탈·데이터 안내에서만 설명한다.
- 검사 진행률은 문항마다 끊어진 segment가 아니라 하나의 연속형 progress bar로 표현한다. 빠른 코어 약 20문항과 정밀 코어 약 60문항에 동일한 패턴을 사용하고, 상단의 `현재 문항 / 전체 문항` 숫자로 정확한 위치와 접근성 값을 보완한다.
- S03의 이전·다음 행동은 엄지 도달 영역의 하단 dock에 배치한다. dock은 safe area를 포함하고, 콘텐츠·키보드와 겹치지 않으며, 스크롤 중에도 다음 행동을 잃지 않게 한다.
- `S03`, `S05`, route·컴포넌트 이름 같은 내부 식별자는 고객 문구에 사용하지 않는다. 완료 전환이 실제로 지연될 때는 `답변을 바탕으로 첫 성향을 정리하고 있어요.`처럼 고객이 이해할 수 있는 처리 상태만 표시한다.
- 홈은 이번 Foundation 검토에서 정보 구조 방향만 확인한다. 카드 위계·개인화·게임 요소·피드 연결은 W3의 S01 전용 기획에서 다시 설계한다.
