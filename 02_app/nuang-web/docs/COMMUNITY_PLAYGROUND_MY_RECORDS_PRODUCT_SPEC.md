# 오늘의 성향 놀이터 · 내 기록 상세 기획

문서 상태: 승인 검토안
대상: MVP 커뮤니티와 향후 마이 프로필 개편
원칙: 가벼운 선택은 재미있게 보여주되 공식 성향 검사처럼 해석하지 않는다.

## 1. 현재 구현 진단

- 피드의 `오늘의 성향 놀이터`에서 투표와 코드별 관점을 볼 수 있다.
- `내 기록`은 `/feed/perspectives`로 연결되지만 실제 계정 데이터를 읽지 않고 고정 예시 문구를 보여준다.
- 따라서 현재 화면의 `자주 고른 관점`, `최근 달라진 관점`, `지난달 대비`는 실제 사용자 근거가 없다.
- 실제 투표는 계정, 질문, 선택지, 선택 당시 뉴앙 코드와 성향명 단위로 DB에 저장된다.
- 현재 데이터만으로는 최신 선택 목록은 만들 수 있지만 선택 변화나 주간 추세를 신뢰성 있게 설명할 수 없다.

결론: 고정 예시 화면을 마이에 노출하면 안 된다. 실제 기록 조회 기능을 만든 뒤 피드와 마이에서 같은 화면으로 연결해야 한다.

## 2. 제품 목표

### 사용자 목표

1. 오늘 질문에 빠르게 참여한다.
2. 전체와 여러 뉴앙 코드의 관점 차이를 피드 안에서 바로 즐긴다.
3. 과거에 내가 어떤 답을 골랐는지 다시 본다.
4. 진행 중인 질문은 실수를 바로잡을 수 있다.
5. 기록이 공식 성향 검사 결과를 바꾸는지 혼동하지 않는다.

### 서비스 목표

- 커뮤니티 피드 체류와 반복 방문을 자연스럽게 높인다.
- 성향 기반 커뮤니티라는 뉴앙의 차별점을 반복해서 경험시킨다.
- 사용자의 개인 선택을 과도하게 해석하거나 공개하지 않는다.
- 향후 마이 프로필 개편과 중복 화면 없이 연결한다.

### 하지 않을 것

- 한두 번의 밸런스 게임으로 성향을 단정하지 않는다.
- `당신은 이런 사람`과 같은 진단형 문장을 만들지 않는다.
- 개인의 투표 선택을 상대 프로필, 검색, 공유 카드에 공개하지 않는다.
- 재투표 전후로 전체 퍼센트가 얼마나 변했는지 알려주지 않는다.
- 충분한 데이터가 없는데 주간 변화나 성향 변화를 만들어내지 않는다.

## 3. 용어 정의

- **오늘의 성향 놀이터**: 뉴앙이 제공하는 가벼운 밸런스 게임과 관점 탐색 피드.
- **내 기록**: 사용자가 참여한 질문과 현재 선택을 본인만 다시 보는 개인 활동 기록.
- **코드별 관점**: 최소 공개 인원을 충족한 뉴앙 코드 집단의 익명 선택 비율.
- **선택 당시 코드**: 투표 또는 재투표가 저장된 시점의 사용자 뉴앙 코드 스냅샷.
- **공식 검사 결과**: 정밀·빠른 코어 검사 응답으로 계산된 결과. 성향 놀이터 기록과 분리한다.

## 4. 전체 사용자 흐름

### 피드에서 참여

1. 추천 피드에서 `오늘의 성향 놀이터`를 본다.
2. 두 선택지 중 하나를 고른다.
3. 로그인하지 않았다면 로그인 후 원래 질문으로 돌아온다.
4. 저장이 완료되면 전체 결과와 코드별 관점이 한 번 자동으로 펼쳐진다.
5. 사용자는 `전체`, `내 코드`, `참여 코드`를 바꿔 보며 차이를 살핀다.
6. 관점 영역을 접고 피드를 계속 본다.
7. `내 기록`을 누르면 피드 스크롤 위치를 기억한 채 개인 기록으로 이동한다.

### 다시 방문

1. 이전에 참여한 질문은 선택 결과를 보여준다.
2. 코드별 관점은 기본적으로 접혀 있어 피드가 지나치게 길어지지 않는다.
3. 필요할 때 펼쳐서 마지막으로 보던 코드 관점을 확인한다.
4. 진행 중인 질문에는 `투표 다시 하기`를 제공한다.

### 재투표

1. `투표 다시 하기`를 누른다.
2. 현재 답이 선택된 편집 상태가 열린다.
3. 다른 답을 고른다.
4. `이 선택으로 바꾸기`를 직접 누른다.
5. 기존 답과 같으면 완료 버튼은 비활성화한다.
6. 저장 후 `선택을 바꿨어요`를 보여주고 관점 영역은 접힌 상태로 돌아간다.
7. 네트워크 오류가 나면 선택 상태를 유지하고 다시 시도할 수 있다.

## 5. 화면별 상세 설계

### 5.1 피드 · 오늘의 성향 놀이터

#### 투표 전

- 뉴앙 메인 캐릭터, `오늘의 성향 놀이터`, 질문 주제를 표시한다.
- 질문과 선택지만 우선 노출한다.
- 코드별 관점과 내 기록 해석은 노출하지 않는다.
- 안내 문구: `가볍게 고른 선택은 공식 검사 결과를 바꾸지 않아요.`

#### 투표 직후

- 선택한 답을 연한 브랜드 색으로 표시한다.
- 전체 선택 비율을 바로 보여준다.
- 코드별 관점을 한 번 자동으로 펼친다.
- 좋아요, 댓글, 공유, 저장은 일반 피드 게시물과 같은 형태를 유지한다.

#### 코드별 관점 접힘

- 한 줄 헤더만 남긴다.
- 구성: `코드별 관점 보기` / 총 참여 수 / 펼침 아이콘 / `투표 다시 하기`.
- 피드 카드의 높이를 줄여 다음 게시물로 자연스럽게 이동하게 한다.

#### 코드별 관점 펼침

- `전체`, `내 코드`, `참여 코드 N` 탭을 제공한다.
- 동일 코드 참여자가 5명 미만이면 내 코드 탭을 비활성화한다.
- 비활성 문구: `같은 코드의 참여자가 5명 이상 모이면 공개돼요.`
- 공개 기준 미달 코드는 참여 코드 목록에서도 숨긴다.
- 결과 문장은 관찰형으로만 작성한다.
  - 허용: `ENAKQ 참여자는 A를 더 많이 골랐어요.`
  - 금지: `ENAKQ는 원래 A를 좋아해요.`

### 5.2 내 기록 메인

권장 정식 경로: `/my/playground`

기존 `/feed/perspectives`는 다음 중 하나로 유지한다.

- `/my/playground?from=feed`로 리다이렉트
- 동일 컴포넌트를 사용하되 뒤로가기 목적지만 피드로 설정

화면 구성:

1. 앱바: 뒤로가기, `성향 놀이터 기록`.
2. 개인정보 안내: `이 기록은 나만 볼 수 있어요.`
3. 요약: 총 참여 수와 가장 최근 참여일만 제공.
4. 주제 필터: 전체, 관계, 대화, 취향, 일상 등 실제 질문 데이터에 있는 주제만 표시.
5. 최신순 기록 목록.
6. 페이지 추가 로딩.
7. 빈 상태와 피드 이동 버튼.

기록 한 건의 구성:

- 질문 주제와 참여 날짜
- 질문 문장
- `내가 고른 답`
- `선택 당시 코드`
- 진행 중 또는 마감 상태
- 진행 중일 때 `투표 다시 하기`
- `현재 결과 보기` 펼침 기능

한 화면에서 기록을 펼치게 하고 별도 상세 화면은 MVP에서 만들지 않는다. 이동 단계를 줄이고 피드와 같은 읽기 흐름을 유지하기 위해서다.

### 5.3 향후 마이 프로필 연결

마이 화면은 상대 프로필과 같은 기본 위계를 사용한다.

1. 프로필 이미지, 이름, 뉴앙 코드와 역할명
2. 내 소개
3. 핵심 행동: 프로필 편집, 공유 범위 확인
4. 활동 요약
5. 내 게시물

`성향 놀이터 기록`은 공개 프로필 영역이 아닌 **나의 활동**에 배치한다.

- 메뉴명: `성향 놀이터 기록`
- 보조 문구: `내가 참여한 질문과 선택 모아보기`
- 최근 선택 한 건을 미리 보여줄 수 있지만 상대에게는 노출하지 않는다.
- 피드와 마이 모두 같은 `/my/playground` 화면을 연다.
- 마이에서 들어오면 뒤로가기는 `/my`, 피드에서 들어오면 `/feed`와 기존 스크롤 위치로 돌아간다.

## 6. 상태 설계

필수 화면 상태:

- 로그인 필요
- 초기 로딩
- 기록 없음
- 기록 목록
- 필터 결과 없음
- 다음 페이지 로딩
- 네트워크 오류와 재시도
- 마감된 질문
- 운영 중단 또는 삭제된 질문
- 재투표 가능
- 재투표 저장 중
- 재투표 성공
- 재투표 실패

운영 중단 질문은 원문을 계속 노출하지 않고 `더 이상 볼 수 없는 질문`으로 표시한다. 사용자의 참여 시각과 기록 존재 여부는 유지할 수 있다.

## 7. 데이터 및 API 설계

### 현재 활용 가능한 데이터

`feed.feed_poll_vote`에 다음 값이 이미 있다.

- vote ID
- poll ID
- option ID
- account ID
- 선택 당시 뉴앙 코드와 성향명
- 생성 시각
- 삭제 시각

### 추가 필드 제안

`feed.feed_poll_vote`

- `updated_at`: 마지막으로 선택을 변경한 시각
- `change_count`: 변경 횟수

`feed.feed_poll`

- `topic_key`: 관계, 대화, 취향 등 질문 주제
- `published_at`: 노출 시작 시각
- `closes_at`: 투표 마감 시각
- `status`: draft, scheduled, active, closed, removed

MVP에서는 선택 변경 이력을 별도 행으로 장기 보관하지 않고 최종 선택만 유지한다. 분석형 변화 기능을 승인할 때에만 별도 이벤트 로그의 필요성과 보관 기간을 다시 검토한다.

### 조회 계약

제안 API:

`GET /api/community/playground-records?topic=all&cursor=...`

응답 필드:

- `voteId`
- `pollId`
- 질문과 주제
- 선택지 ID와 문구
- `createdAt`, `updatedAt`
- 선택 당시 코드와 성향명
- `pollStatus`
- `canRevote`
- `changeCount`
- 현재 전체 선택 비율
- 공개 가능한 코드별 관점
- 다음 페이지 커서

서버는 요청 파라미터로 account ID를 받지 않는다. 로그인 세션에서 현재 계정을 확인하고 모든 조회에 `account_id = 현재 계정`을 강제한다.

### 쓰기 계약

- 기존 `vote_poll` 액션을 최초 투표와 재투표에 함께 사용한다.
- 활성 질문과 해당 질문의 선택지인지 서버에서 확인한다.
- 재투표 시 기존 활성 행만 갱신한다.
- 동일 선택이면 DB 쓰기를 수행하지 않는다.
- 중복 요청에도 동일한 최종 선택 한 건만 남아야 한다.

## 8. 개인정보와 신뢰 기준

- 내 기록은 항상 본인만 조회한다.
- 공개 프로필 설정과 별도 정책으로 운영한다.
- 개인 선택은 익명 집계에만 반영한다.
- 코드별 관점은 코드당 최소 5명 이상일 때만 공개한다.
- 재투표 후 5명 미만이 된 코드 집계는 즉시 숨긴다.
- 선택 변경 전후의 퍼센트 차이는 공개하지 않는다.
- 계정 삭제 시 투표도 함께 삭제한다.
- 공식 검사와 커뮤니티 놀이 기록을 UI와 문구에서 명확히 구분한다.
- 정확한 안내 문구: `내 선택은 개인 단위로 공개되지 않고 익명 집계에만 반영돼요.`

## 9. 재미와 재방문 설계

MVP에서 사용할 장치:

- 투표 직후 결과와 코드별 관점을 즉시 보여주는 보상
- 전체와 내 코드의 차이를 직접 탐색하는 재미
- 과거 질문과 내 선택을 다시 보는 회고
- 매일 새로운 질문이 있다는 기대

초기에 넣지 않을 장치:

- 연속 출석을 끊었을 때 불안을 주는 강한 스트릭
- 경쟁 순위
- 성향 우열처럼 보이는 배지
- 충분한 근거가 없는 AI 성향 해석
- 과도한 푸시 알림

## 10. 운영 설계

`오늘의`라는 이름을 유지하려면 매일 별도 poll을 발행해야 한다. 같은 poll의 질문 문구만 바꾸면 과거 투표가 새 질문 결과에 섞이므로 금지한다.

필요 운영 기능:

- 질문 초안 작성
- 선택지 검수
- 주제 지정
- 발행 예약
- 자동 마감
- 신고 또는 오류 시 운영 중단
- 최소 집계 인원 확인

초기에는 관리 도구 없이 검수된 seed 또는 DB 스크립트로 운영할 수 있다. 질문 수가 늘면 관리자 화면을 별도 기획한다.

## 11. 성공 지표

핵심 지표:

- 놀이터 노출 대비 투표 완료율
- 투표 후 코드별 관점 펼침률
- 투표 후 피드 추가 체류 시간
- 7일 내 놀이터 재참여율
- 내 기록 진입률
- 재투표 오류율

보호 지표:

- 투표 직후 이탈률
- 반복 재투표 비율
- 코드별 관점 관련 신고율
- 공식 검사 결과로 오해했다는 고객 문의
- 5명 미만 집계 노출 사고 0건

## 12. 개발 우선순위와 승인 범위

### 현재 승인되어 구현하는 범위

- 코드별 관점 접기·펼치기
- 확인 단계를 포함한 투표 다시 하기
- 기존 투표 행을 새 선택으로 갱신

### 추가 승인이 필요한 개발

#### A. 실제 내 기록 MVP 전환 — 최우선 권장

- 고정 예시 제거
- 계정 전용 기록 조회 API
- 최신순 기록과 빈 상태
- 피드와 마이의 공통 화면 연결
- `/my/playground` 정식 경로

#### B. 날짜별 질문 운영 구조 — A와 함께 권장

- 질문 주제, 발행, 마감 필드
- 매일 별도 poll 발행
- 마감 질문 재투표 차단

#### C. 기록 관리

- 개별 투표 취소
- 전체 기록 지우기
- 통계 반영 방식과 확인 문구

#### D. 분석형 요약 — 후순위

- 충분한 참여 수 기준 정의
- 자주 고른 선택과 주제 요약
- 최근 변화 설명
- 심리 검사로 오해하지 않도록 검증된 문구 규칙

#### E. 재방문 확장 — 후순위

- 알림 수신 설정
- 부담 없는 주간 회고
- 새 질문 알림

권장 승인 순서는 `A + B → C → D → E`다. A와 B 없이 현재 예시 화면을 마이에 연결하거나 분석 문구를 늘리면 신뢰를 해칠 가능성이 높다.
