# NUANG App Functional Requirements

작성일: 2026-07-11 KST

상태: 현재 완료된 MVP 기능을 기준으로 작성한 통합 기능 요구사항 명세서.

이 문서는 NUANG 앱을 다시 설계하거나 이어서 개발할 때 참조할 수 있는 단일 기능 명세다. 특정 화면 배치나 메뉴 구성을 확정하는 문서가 아니다. 즉, `홈에는 반드시 A, B, C가 있어야 한다`처럼 섹션 배치를 단정하지 않는다. 대신 앱이 제공해야 하는 기능, 사용자 행동, 데이터 경계, 개인정보 원칙, 상태 처리, API/route 참고 표면을 정리한다. 새로운 AI 또는 개발자는 이 문서를 바탕으로 하단 탭바, 메뉴, 페이지 구조, 컴포넌트 배치를 다시 설계할 수 있다.

## 1. 문서 사용 원칙

이 문서는 기능 요구사항을 고정한다.

- 어떤 기능이 존재해야 하는가.
- 사용자가 그 기능으로 무엇을 할 수 있어야 하는가.
- 어떤 데이터가 저장되거나 공개될 수 있는가.
- 어떤 데이터는 절대 공개, 공유, 비교, 피드에 쓰면 안 되는가.
- 로그인, 동의, 공개 범위, 삭제, noindex, QA 조건은 무엇인가.

이 문서는 정보 구조를 고정하지 않는다.

- 특정 기능이 반드시 특정 메뉴의 특정 순서에 있어야 한다고 단정하지 않는다.
- 하단 탭바의 최종 라벨, 순서, grouping은 이 문서의 기능 도메인을 바탕으로 별도 설계할 수 있다.
- 단, 어떤 메뉴 구조를 선택하더라도 사용자가 핵심 기능에 접근할 수 있어야 하고, 같은 이동 경로를 카드 목록으로 반복해서 보여주면 안 된다.

현재 구현의 대표 접근 도메인은 다음과 같다.

- 시작점과 오늘 이어갈 행동
- 성향 기반 피드
- 검사와 결과 생성
- 뉴앙 코드와 성향지도 탐색
- 내 성향, 내 리포트, 설정
- 결과 공유와 공개 요약
- 공개 프로필 기반 1:1 비교
- 도움 연결과 정책 안내

## 2. 서비스 정의

NUANG은 검사가 중심인 성향 기반 SNS다. 사용자는 가볍게 검사를 시작하고, 뉴앙 코드와 결과 리포트를 통해 자신을 이해하며, 공개 범위 안에서 피드와 비교 기능을 사용할 수 있다.

서비스가 만들어야 하는 핵심 경험은 다음과 같다.

- 사용자가 로그인 없이도 검사를 쉽게 시작한다.
- 검사 진행은 짧고 명확하며 자동 저장된다.
- 결과는 예쁘고 믿을 만하되, 사람을 고정하거나 낙인찍지 않는다.
- 뉴앙 코드는 사용자가 기억하고 대화할 수 있는 5글자 대표 성향 코드로 쓰인다.
- 피드는 단순 커뮤니티가 아니라 성향 탐구와 체류를 만드는 SNS 화면이다.
- 공유와 비교는 사용자가 공개한 범위 안에서만 작동한다.
- 직접 응답, 원점수, 계정 식별 정보, 민감 항목, 도움 허브 이용 맥락은 공개, 공유, 비교, 피드에 포함하지 않는다.

## 3. 공통 내비게이션 요구사항

앱은 모바일 우선으로 설계한다. 주요 기능 도메인은 사용자가 반복 방문해도 길을 잃지 않게 접근 가능해야 한다.

내비게이션 요구사항:

- 주요 기능 도메인은 한두 번의 명확한 행동으로 접근 가능해야 한다.
- 하단 탭바를 사용할 경우 각 탭은 서로 다른 기능 목적을 가져야 한다.
- 하단 탭바와 같은 최상위 이동 수단을 화면 내부에서 카드 목록으로 반복하지 않는다.
- 현재 위치는 접근성 속성 또는 시각 상태로 명확히 표시한다.
- 모바일 safe area를 고려해 하단 고정 내비게이션이 콘텐츠와 겹치지 않아야 한다.
- `함께`는 현재 제품 방향에서 폐기된 최상위 탭이다. 기존 `/together` 계열 접근은 현재 제품 동선으로 redirect하거나 noindex/unavailable 상태로 처리한다.
- `피드와 커뮤니티`처럼 피드와 커뮤니티를 병기하지 않는다. 사용자-facing 명칭은 `피드`를 사용한다.
- 공개 코드 입력, 공개 코드 발급, 공개 코드 복사 기능은 만들지 않는다.

하단 탭바나 주요 메뉴를 다시 구성할 때 고려할 기능 도메인:

- 시작/요약/이어가기 도메인
- 피드 읽기/반응/작성 도메인
- 검사 시작/진행/검사 카탈로그 도메인
- 뉴앙 코드와 코드 지도 탐색 도메인
- 내 성향, 내 리포트, 계정/공개 범위/데이터 관리 도메인

## 4. 공통 UI/UX 요구사항

전반 UI는 SNS 앱처럼 직접적이고 반응 가능해야 한다.

필수 요구사항:

- 모바일 우선 레이아웃을 기본으로 한다.
- Tailwind 기반 스타일을 사용한다.
- lucide-react 등 검증된 아이콘 시스템을 사용한다.
- 반복 항목, 모달, 도구 표면에만 카드 형태를 사용한다.
- 페이지 섹션은 얇은 경계선, 충분한 여백, 읽기 쉬운 타이포그래피를 중심으로 구성한다.
- 한 화면에서 즉시 선택해야 하는 primary action은 하나를 기본값으로 한다.
- 사용자가 매번 긴 설명을 읽지 않아도 상태와 행동을 이해할 수 있어야 한다.
- 텍스트는 360px 모바일 폭에서도 겹치지 않아야 한다.
- 버튼, 고정 내비게이션, 입력 컨트롤은 안정적인 높이와 터치 영역을 가져야 한다.
- 캐릭터 이미지는 감정적 초점으로 쓰되, 화면마다 하나의 분명한 역할만 맡긴다.

금지 요구사항:

- 이모지 또는 데스크탑 이모티콘을 사용자-facing source에 넣지 않는다.
- 과한 gradient, glow, blur, orb 장식을 쓰지 않는다.
- 카드 안의 카드를 중첩하지 않는다.
- 컬러 박스 안 아이콘을 반복해 관리툴 기능 목록처럼 보이게 하지 않는다.
- 둥근 사각형 태그를 반복해 AI 데모처럼 보이게 하지 않는다.
- 사용자 화면에서 내부 영어 축약어를 노출하지 않는다. 예외는 뉴앙 코드뿐이다.
- 사용자 화면에서 `5축`, `10축`, `5개 영역` 표현을 쓰지 않는다. `코드 지도`, `코드 자리`, `세부 신호`를 사용한다.

## 5. 계정, 로그인, 동의 요구사항

로그인은 모든 사용 흐름의 출발점이 아니다. 검사는 로그인 없이 시작할 수 있고, 계정 저장, 공유, 피드 write, 비교 생성처럼 서버 권한이 필요한 순간에 로그인하도록 한다.

기능 요구사항:

- 사용자는 로그인 없이 빠른 코어, 정밀 코어, 무료 주제 검사, 별난 성향 연구소를 시작할 수 있어야 한다.
- 사용자는 로그인 없이 로컬 결과를 볼 수 있어야 한다.
- 계정 결과 저장, 공유 주소 생성, 피드 작성, 댓글, 반응, 저장, 투표, 1:1 비교 생성/조회/삭제, 계정 결과 삭제는 인증된 사용자만 수행한다.
- 로그인 전 필수 동의는 이용약관과 개인정보 처리방침 동의다.
- 선택 동의는 분석, 마케팅으로 분리한다.
- 동의 draft는 인증 시작 전에 localStorage에 보관할 수 있다.
- 인증 후에는 provider 표시명과 provider label을 보여줄 수 있다.
- 로그아웃 기능을 제공한다.

지원 provider 요구사항:

- Kakao와 Google은 Supabase OAuth provider로 연결할 수 있다.
- Naver는 Supabase 기본 provider가 아니므로 MVP에서는 실제 redirect를 열지 않는다. custom OAuth/OIDC bridge 검증 전까지는 준비 상태로 취급한다.
- provider 원문 응답, access token, refresh token, raw birthdate, raw mobile, raw CI는 사용자 데이터 payload로 저장하거나 노출하지 않는다.

OAuth callback 요구사항:

- callback은 authorization code가 없으면 실패 상태로 안전하게 되돌린다.
- Supabase 환경 변수가 없으면 계정 기능을 닫힌 상태로 안내한다.
- `next` 이동 값은 내부 path만 허용한다.
- `//example.com` 같은 protocol-relative 또는 외부 redirect는 `/my` 같은 안전한 기본 경로로 대체한다.

## 6. 시작/요약/이어가기 기능 요구사항

앱은 사용자가 현재 할 수 있는 다음 행동을 빠르게 알 수 있어야 한다. 이 기능은 현재 구현에서 `/home`이 담당하지만, 새 IA에서는 다른 이름이나 구조로 재배치할 수 있다.

기능 요구사항:

- 로컬에 진행 중인 검사가 있으면 이어갈 수 있게 해야 한다.
- 로컬에 완료 결과가 있으면 결과 리포트 또는 내 성향으로 다시 갈 수 있게 해야 한다.
- 진행 중 검사, 최근 결과, 첫 검사 시작 중 가장 적절한 하나의 다음 행동을 우선 제안해야 한다.
- 피드 미리보기는 전체 피드와 같은 server read model을 사용해야 한다.
- 피드 미리보기는 관심 없음 처리된 항목을 제외해야 한다.
- 질문형 피드 콘텐츠가 있을 경우 오늘 답해볼 수 있는 질문으로 강조할 수 있다.
- 공유와 비교에서 직접 응답, 원점수, 민감 항목이 노출되지 않는다는 신뢰 안내를 제공해야 한다.
- 시작 화면은 모든 주요 메뉴로 이동하는 반복 카드 목록이 되어서는 안 된다.

상태 요구사항:

- 로컬 상태를 불러오는 동안 로딩 상태를 보여준다.
- 진행 중 검사는 응답 수와 전체 문항 수를 기반으로 진행률을 보여줄 수 있다.
- 완료 결과는 뉴앙 코드, 코드 이름, 리포트 재열기 동선을 제공할 수 있다.
- 결과가 없으면 빠른 코어 또는 적절한 첫 검사 시작을 제안할 수 있다.

## 7. 피드 기능 요구사항

피드는 NUANG의 체류 경험을 만드는 핵심 SNS 기능이다. 피드는 읽기, 반응, 댓글, 저장, 오늘의 질문, 밸런스 게임, 리포트 공유 preview, 뉴앙 코드별 통계를 제공한다.

### 7.1 피드 읽기

기능 요구사항:

- 피드는 단일 스트림 기반으로 콘텐츠를 보여준다.
- 피드에는 공식 seed 카드와 서버에 저장된 게시글이 함께 표시될 수 있다.
- Supabase 서버 read가 불가능하면 seed fallback을 제공한다.
- 로그인한 사용자는 본인의 게시글과 공개 게시글을 볼 수 있다.
- `not_interested` 처리한 게시글 또는 seed 카드는 해당 사용자 피드에서 제외된다.
- 각 피드 항목은 작성자, 시간, 제목 또는 질문, 본문, 반응 수, 댓글 수, 저장 상태를 표현할 수 있어야 한다.
- 피드 항목 작성자 프로필을 눌러 공개 프로필 팝업을 열 수 있어야 한다.
- 공개 프로필 팝업에서는 상대 프로필 이미지, 표시 이름, 대표 성향, 공개 범위 안내, 나와 비교하기 행동을 제공할 수 있다.

피드 항목 종류:

- 일반 글
- 오늘의 기분
- 오늘의 질문 답변
- 성향 카드
- 성향지도 노트
- 밸런스 게임 투표 글
- 검사 리포트 공유 preview
- 공식 주제 카드

### 7.2 피드 작성

피드 작성은 인증된 사용자만 가능하다.

작성 형식 요구사항:

- 사용자가 직접 고르는 피드 작성 형식은 `글`, `오늘의 질문`, `밸런스 게임` 세 가지다.
- `리포트 공유`는 피드 composer의 선택지가 아니다. 리포트 화면의 `피드에 공유` 행동에서 시작한다.
- `글`은 자유 본문을 작성한다.
- `오늘의 질문`은 서비스가 제공한 질문 template에 답하는 방식이다.
- `밸런스 게임`은 서비스가 제공한 질문과 선택지를 고르고, 선택적으로 짧은 설명을 덧붙이는 방식이다.
- 비로그인 사용자가 게시하려 하면 로그인 화면으로 이동해야 한다.

데이터 요구사항:

- 모든 feed write 요청은 `/api/feed`의 discriminated action 계약을 통과해야 한다.
- 게시글 본문은 최대 800자다.
- 댓글 본문은 최대 400자다.
- 게시글 source는 `daily_mood`, `daily_question`, `trait_card`, `map_reflection`, `free_text`, `balance_game`, `report_share` 중 하나다.
- 게시글 visibility는 `public`, `profile_public`, `private_draft` 중 하나다.
- 기본 공개 게시글은 `public`을 사용한다.
- 새 게시글은 `pending_review` 상태로 저장된다.
- public projection payload는 서버가 생성한다. 클라이언트가 보낸 공개 요약을 그대로 신뢰하지 않는다.
- 직접 응답, 원점수, 민감 검사 결과, 도움 허브 이용 기록은 피드 payload에 포함하지 않는다.

### 7.3 피드 상호작용

지원 action:

- `create_comment`
- `react`
- `bookmark`
- `remove_reaction`
- `remove_bookmark`
- `not_interested`
- `vote_poll`

반응 요구사항:

- 반응 종류는 `like`, `same`, `curious`, `support`를 지원한다.
- 사용자의 현재 좋아요/저장 상태를 초기 UI에 반영한다.
- 좋아요와 저장은 soft-delete 기반 toggle로 동작한다.
- 댓글 작성 후 read model이 갱신되어 최근 댓글 preview가 반영되어야 한다.
- 공식 seed 카드에도 반응, 댓글 preview, 저장 상태를 반영할 수 있어야 한다.
- 관심 없음은 개인화 신호로 저장하고 이후 내 피드와 홈 preview에서 제외해야 한다.

### 7.4 밸런스 게임과 투표

기능 요구사항:

- 밸런스 게임 게시글은 질문과 선택지를 포함한다.
- 사용자는 로그인 후 하나의 투표에 하나의 선택만 할 수 있다.
- 비로그인 사용자가 투표하려 하면 로그인 화면으로 이동해야 한다.
- MVP에서는 투표 재선택을 허용하지 않는다.
- 투표 후 전체 선택 수와 비율을 볼 수 있어야 한다.
- `뉴앙 코드별 통계 보기`를 제공할 수 있다.

통계 요구사항:

- 전체 통계는 선택지별 비율과 투표 수를 보여준다.
- 코드별 통계는 뉴앙 코드, 코드 이름, 선택지별 비율, 응답 수를 보여준다.
- 0명이 투표한 뉴앙 코드는 목록에서 숨긴다.
- 코드별 통계에서 누가 투표했는지 알 수 있는 계정 식별자, 사용자 목록, 프로필 링크를 노출하지 않는다.
- 투표 당시 사용자의 뉴앙 코드와 코드 이름만 익명 집계용 snapshot으로 저장할 수 있다.
- 원점수, 직접 응답, 세부 신호 원본은 투표 payload에 저장하지 않는다.

### 7.5 리포트 공유 preview

기능 요구사항:

- 사용자는 결과 리포트에서 피드 공유를 시작할 수 있다.
- 피드에 공유된 리포트는 공개 가능한 preview만 보여준다.
- preview를 누르면 피드 공유 리포트 화면으로 이동한다.
- 피드 공유 리포트 화면은 noindex다.
- 피드 공유는 일반 30일 만료 공유 주소와 수명 정책이 다르다. 피드 공유 preview는 해당 피드 글이 삭제되기 전까지 유지될 수 있다.

preview에 포함 가능한 항목:

- 뉴앙 코드
- 코드 이름
- 공개 프로필 이미지 또는 캐릭터 이미지
- 코드 지도 요약
- 검사 이름
- 완료일
- 리포트 보기 동선

preview 금지 항목:

- 직접 응답
- 원점수
- 내부 채점 payload
- 민감 검사 결과
- 도움 요청 기록
- OAuth 정보
- 이메일

## 8. 검사 기능 요구사항

검사는 NUANG의 중심 기능이다. 피드와 비교는 검사 결과를 더 자주 다시 보게 만드는 확장 경험이다.

### 8.1 코어 검사

코어 검사는 빠른 코어와 정밀 코어로 나뉜다.

빠른 코어 요구사항:

- 무료로 제공한다.
- 약 20문항 내외로 구성한다.
- 로그인 없이 시작할 수 있다.
- 예비 뉴앙 코드와 결과 요약을 제공한다.
- 정밀 코어로 자연스럽게 확장할 수 있다.
- 1:1 비교의 기준으로 사용하지 않는다.

정밀 코어 요구사항:

- 영구 무료로 제공한다.
- 약 60문항 내외로 구성한다.
- 공식 대표 성향 코드, 코드 이름, 코드 지도, 세부 신호를 제공한다.
- 공개 프로필과 1:1 비교의 기준으로 사용할 수 있다.
- 결제벽 뒤에 두면 안 된다.

진행 요구사항:

- 한 문항씩 집중해서 응답하게 한다.
- 응답 선택지는 1~5 scale과 `판단하기 어렵다`를 제공한다.
- 이전/다음 이동을 제공한다.
- 진행률과 현재 문항 번호를 보여준다.
- 응답과 현재 위치는 자동 저장된다.
- 진행 중 로컬 검사는 7일 동안 이어갈 수 있다.
- 완료 로컬 결과는 30일 동안 다시 열 수 있다.
- 정밀 코어 시작 시 빠른 코어 응답 중 재사용 가능한 문항 응답을 초기값으로 활용할 수 있다.

### 8.2 무료 주제 검사

무료 주제 검사는 체류, 재미, 자기이해, 성향 구체화, 피드 소재를 만드는 핵심 콘텐츠다.

기능 요구사항:

- 무료로 제공한다.
- 짧은 문항으로 구성한다.
- 로그인 없이 시작할 수 있다.
- 결과는 먼저 로컬에 저장한다.
- 로그인 상태에서는 서버 동기화 queue를 통해 저장을 시도한다.
- 오프라인 또는 서버 실패 시 나중에 다시 동기화할 수 있어야 한다.
- URL 재방문 시 로컬에 없으면 서버 저장 결과를 복원할 수 있다.
- 결과는 주제별 해석, 신뢰 기준, 세부 신호를 보여준다.
- 내부 영어 채점 키는 사용자 화면에 노출하지 않는다.
- 공개 요약만 피드에 공유할 수 있다.
- 직접 응답과 관찰 payload는 피드 요청에 섞이지 않아야 한다.

현재 열린 무료 주제 검사 범위:

- 관계: 대화 온도, 사과 방식, 거리감 리듬, 갈등 후 회복
- 일상: 충전 루틴, 집중 전환, 정리 방식
- 감정: 기분 전환 방식, 서운함 표현, 위로 받는 방식
- 사회적 에너지: 모임 후 회복, 새 사람 대하는 방식, 팀플 역할
- 취향형 성향: 계획 여행과 즉흥 여행 등
- 성장 영역은 확장 후보로 남아 있다.

성향 증거 요구사항:

- 무료 주제 검사는 기본적으로 `free_topic` source weight 0.35의 성향 보강 증거로 취급한다.
- 승인된 축·세부 신호 매핑과 충분한 유효 가중치가 쌓이면 현재 대표 성향을 더 구체화할 수 있다.
- 무료 주제 검사 하나로 공식 대표 코드를 즉시 흔들면 안 된다.
- 1:1 비교에는 기본적으로 무료 주제 검사 신호를 사용하지 않는다.

### 8.3 별난 성향 연구소

별난 성향 연구소는 가벼운 생활형 검사 영역이다.

기능 요구사항:

- 무료로 제공한다.
- 로그인 없이 시작할 수 있다.
- 결과는 로컬에 저장한다.
- 기본적으로 서버 전송을 하지 않는다.
- 기본적으로 성향지도, 대표 성향 코드, 1:1 비교 기준에 반영하지 않는다.
- 결과는 프로필형 재미 결과, 장점, 살펴볼 점, 관계 팁, 작은 실험, 결과 분포를 제공한다.
- 결과는 삭제할 수 있다.
- 민감도는 S1 또는 S2로 표시할 수 있다.
- 진단, 치료, 위험 판정, 관계 성공 예측, 채용·금융 판단에 사용할 수 없다고 안내해야 한다.

### 8.4 도움 연결 허브

도움 연결 허브는 검사가 아니다. 위기, 치료, 민감 주제는 점수화하지 않고 도움 정보로 연결한다.

기능 요구사항:

- 긴급 상황에서 전화 행동을 바로 제공한다.
- 도움 자원 목록과 공식 안내 링크를 제공한다.
- 이 화면에서 본 항목은 계정, 결과, 성향지도, 비교 리포트에 저장하지 않는다.
- 자살, 자해, 우울, ADHD, 중독, 트라우마, 폭력, 약물 등은 재미형 검사로 열지 않는다.
- 성적 지향은 질병이나 치료 대상으로 다루지 않는다.
- 임상 선별 도구를 열려면 별도 승인, 출처, 목적, 한계, 비저장 기본값, 긴급 안내, 전문가 상담 권고가 필요하다.

## 9. 뉴앙 코드와 성향지도 요구사항

뉴앙 코드는 사용자가 기억하고 대화할 수 있는 5글자 대표 성향 코드다.

코드 규칙:

- 1자리: S 또는 T
- 2자리: C 또는 V
- 3자리: F 또는 O
- 4자리: A 또는 D
- 5자리: E 또는 P
- 유효 코드는 32개다.
- `FOAMT` 같은 임의 5글자 조합은 유효 코드가 아니다.

코드 자리 사용자-facing 이름:

- 에너지가 시작되는 방식
- 마음이 흔들릴 때의 반응
- 일상을 굴리는 방식
- 관계를 맞추는 방식
- 생각이 넓어지는 방식

Motif 규칙:

- TV: 불꽃
- TC: 햇살
- SV: 물결
- SC: 숲

코드 이름 요구사항:

- 사용자-facing 이름은 `모티프의 이름` 형태를 기본으로 한다.
- 예: `불꽃의 온기 탐험가`, `물결의 새길 개척가`, `숲의 편안한 조율가`
- 코드와 이름은 마이, 결과 리포트, 공유 리포트, 프로필 팝업, 1:1 비교 리포트, 성향지도, 피드 preview에서 일관되게 표시한다.

성향지도 기능 요구사항:

- 32개 뉴앙 코드를 탐색할 수 있어야 한다.
- 코드는 motif 계열과 코드 이름으로 이해할 수 있어야 한다.
- 개인 결과가 없더라도 뉴앙 코드 세계를 둘러볼 수 있어야 한다.
- 개인 결과가 있는 경우 내 코드와 코드 지도, 세부 신호로 연결할 수 있다.
- 내부 domain/facet key는 화면에 그대로 노출하지 않는다.
- `5축`, `10축` 대신 `코드 지도`, `코드 자리`, `세부 신호`를 사용한다.

## 10. 결과 리포트 요구사항

결과는 저장 위치와 무관하게 사용자에게 하나의 `내 결과`로 느껴져야 한다.

공통 결과 요구사항:

- 결과 화면 제목은 `결과 리포트`를 사용한다.
- 목록 제목은 `내 결과` 또는 `내 리포트`처럼 저장 위치를 드러내지 않는 표현을 사용한다.
- `계정 결과`, `로컬 결과`, `기기 사본`, `계정 저장` 같은 구현 위치 명칭을 사용자-facing CTA나 상태로 쓰지 않는다.
- 결과는 뉴앙 코드, 코드 이름, 캐릭터/프로필 이미지, 코드 지도, 세부 신호, 영역별 요약, 읽는 기준을 제공할 수 있다.
- 점수는 순위, 능력, 우열, 진단으로 표현하면 안 된다.
- 경계 표현 대신 `두 모습이 비슷한 자리`, `균형 구간`처럼 비낙인 문구를 사용한다.
- 결과 이미지를 생성하거나 기기 공유 기능을 제공할 수 있다.
- 결과 삭제는 하나의 행동으로 제공한다.

로컬 결과 요구사항:

- 로컬 결과는 로그인 없이 열 수 있어야 한다.
- 완료 로컬 결과는 30일 보관한다.
- 로컬 결과는 계정 동의와 로그인 상태가 있으면 서버 결과로 claim할 수 있다.
- claim은 idempotent해야 한다. 이미 claim된 로컬 결과는 기존 서버 결과를 복원한다.
- 사용자는 로컬 결과를 삭제할 수 있어야 한다.

계정 결과 요구사항:

- 인증된 사용자는 본인의 계정 결과를 읽을 수 있어야 한다.
- 계정 결과는 결과 리포트 route에서 다시 열 수 있어야 한다.
- 삭제 시 해당 결과, 공유 링크, 공개 프로필 스냅샷, 그 결과를 참조한 비교 기록을 함께 정리한다.
- 계정 결과 삭제는 소유자 검증 후 처리한다.

무료 주제 결과 요구사항:

- 주제 검사 결과는 주제명, 요약 headline, 평균 선명도, 세부 해석, 결과 읽는 기준을 제공한다.
- 결과는 로컬 우선으로 불러오고, 필요하면 서버 저장 결과에서 복원한다.
- 피드 공유는 공개 가능한 요약만 사용한다.

별난 성향 연구소 결과 요구사항:

- 결과는 생활형 재미 프로필로 제공한다.
- 코어 성향지도와 5글자 코드를 바꾸지 않는다고 안내한다.
- 결과 삭제를 제공한다.

## 11. 내 성향, 내 리포트, 설정 요구사항

개인 영역은 내 대표 성향, 내 리포트, 공개 범위, 계정, 데이터 권리를 안정적으로 다시 여는 공간이다. 새 IA에서 이 기능들이 하나의 메뉴에 있든 여러 화면에 나뉘든 아래 기능은 유지해야 한다.

내 성향 요구사항:

- 최신 계정 결과 또는 로컬 완료 결과를 바탕으로 대표 뉴앙 코드와 코드 이름을 보여준다.
- 로컬 결과가 있으면 로컬 결과를 우선 사용할 수 있다.
- 로컬 결과가 없으면 계정 저장 결과 요약으로 대표 성향과 코드 지도를 복원할 수 있다.
- 코드 지도와 세부 신호를 보여준다.
- 무료 주제 검사 결과가 있으면 최근 더 선명해진 부분으로 보여줄 수 있다.
- 결과가 없으면 검사를 완료하면 이 기능이 열린다고 안내한다.

내 리포트 요구사항:

- 코어 검사 결과, 계정 결과, 진행 중 검사, 별난 성향 연구소 결과, 1:1 비교 리포트를 한 목록으로 모을 수 있다.
- 같은 결과의 로컬/계정 중복은 한 행으로 합친다.
- 저장 위치는 사용자-facing 정보로 강조하지 않는다.
- 진행 중 검사는 이어가기 동선을 제공한다.
- 완료 결과는 다시 열기 동선을 제공한다.
- 1:1 비교 리포트는 다시 열기와 삭제를 지원한다.
- 데이터 내보내기를 제공한다.
- 삭제는 결과 종류별로 적절한 권한과 연쇄 삭제 정책을 따라야 한다.

설정 요구사항:

- 계정 연결 상태를 보여준다.
- 공개 범위 설정 preview를 제공한다.
- 데이터 관리로 이동할 수 있어야 한다.
- 알림 설정은 준비 중 상태로 둘 수 있다.
- 일반 기능은 전연령으로 이용할 수 있고, 19세 이상 전용 기능은 해당 기능을 열 때 별도 확인한다고 안내한다.

공개 범위 요구사항:

- 기본 공개 항목: 기본 프로필, 대표 성향, 코드 지도 요약, 세부 신호 요약
- 기본 비공개 항목: 빠른 코어 예비 결과, 별난 성향 연구소 결과, 인구통계 등
- 항상 비공개 항목: 직접 문항 응답, 원점수와 전체 점수 벡터, 민감 검사, 도움 연결 이용 기록, 계정 식별 정보
- 공개 범위 저장 기능이 열리기 전에는 기본값 preview만 보여주고 저장 완료처럼 보이게 하지 않는다.

## 12. 공유 기능 요구사항

공유는 공개 가능한 요약만 전달하는 기능이다.

일반 공유 주소 요구사항:

- 사용자는 자신의 결과 리포트에서 공유 주소를 만들 수 있다.
- 공유 주소는 최대 30일 동안 열 수 있다.
- 공유 주소는 token 원문을 DB에 저장하지 않고 hash로 검증해야 한다.
- 공유 화면은 noindex다.
- 공유 화면은 뉴앙 코드, 코드 이름, 캐릭터/프로필 이미지, 코드 지도 요약, 제한된 코드 자리 요약, 같은 검사 시작 CTA를 제공할 수 있다.
- 공유 화면에서 1:1 비교를 바로 생성하지 않는다.
- 공유 주소가 만료, 철회, 삭제, 비활성 상태이면 unavailable 화면을 보여준다.

공유 금지 항목:

- 직접 응답
- 원점수
- OAuth 정보
- 이메일
- 계정 식별 정보
- 민감 검사 결과
- 도움 연결 이용 맥락

공유 철회 요구사항:

- 서버 API는 공유 링크 철회를 지원할 수 있다.
- MVP 사용자 화면에서는 별도 공유 링크 관리 메뉴를 노출하지 않는 방향이다.
- 철회 기능을 노출할 경우 리포트 상세의 보조 행동으로만 재검토한다.

피드 공유와 일반 공유의 차이:

- 일반 공유 주소는 30일 자동 만료다.
- 피드에 공유된 리포트 preview는 피드 글이 삭제되기 전까지 유지될 수 있다.
- 두 공유 방식 모두 공개 요약만 사용한다.

## 13. 공개 프로필과 1:1 비교 요구사항

1:1 비교는 링크나 공개 코드를 직접 입력해서 시작하지 않는다. 상대 프로필을 클릭해 열린 공개 프로필 팝업에서 명시적으로 `나와 비교하기`를 선택할 때 시작한다.

공개 프로필 요구사항:

- 공개 프로필은 상대 표시 이름, 프로필 이미지, 대표 뉴앙 코드, 코드 이름, 공개 범위 안내를 보여준다.
- 프로필 이미지는 공개 프로필 이미지다. 캐릭터 이미지를 fallback으로 쓸 수 있다.
- 공개 프로필에는 직접 응답, 원점수, 계정 식별 정보, 민감 항목을 넣지 않는다.
- 공개 프로필 snapshot은 policy version을 가진다.

1:1 비교 생성 요구사항:

- 사용자는 로그인해야 한다.
- 내 정밀 코어 결과가 있어야 한다.
- 상대 공개 프로필 snapshot이 active 상태여야 한다.
- 공개 범위 policy version이 일치해야 한다.
- 비교에 사용할 공개 항목이 비어 있으면 비교를 만들지 않는다.
- 상대의 비공개 항목을 추정하지 않는다.
- 생성된 비교 리포트는 내 리포트에서 다시 열 수 있다.
- 비교 생성과 조회는 접근 감사와 소유자 검증을 전제로 한다.

1:1 비교 리포트 요구사항:

- 서로의 현재 뉴앙 코드, 코드 이름, 프로필 이미지, 표시 이름을 중심으로 보여준다.
- 같은 코드 자리와 다른 코드 자리를 관계 리듬으로 설명한다.
- 오해가 생기기 쉬운 장면, 대화 질문, 조율 가이드를 제공한다.
- 궁합 점수, 순위, 관계 성공 확률, 위험 판정, 상대 진단을 제공하지 않는다.
- 직접 응답, 원점수, private inference, 민감 검사 결과를 사용하지 않는다.
- 공개 범위가 바뀌거나 결과가 삭제되면 stale, disabled, deleted 상태를 처리한다.

상세검사 비교 확장 조건:

- MVP에서는 상세검사 또는 유료검사 기반 비교를 열지 않는다.
- 추후 열더라도 두 사람이 모두 같은 검사를 수행하고, 둘 다 해당 결과의 비교 사용을 허용한 경우에만 비교할 수 있다.
- 한쪽만 수행한 검사는 비교하지 않는다.
- 서로 다른 검사끼리 억지로 비교하지 않는다.

## 14. 공개 코드 폐기 요구사항

공개 코드는 사용자 경험에서 폐기된 기능이다.

금지 기능:

- 공개 코드 발급 UI
- 공개 코드 입력 UI
- 공개 코드 복사
- `/p/[code]`처럼 코드를 주인공으로 하는 사용자 route
- `publicProfileCode`를 받는 비교 생성 API
- 공개 코드를 공유 링크나 프로필 링크처럼 설명하는 문구

대체 구조:

- 결과 공유는 공유 token으로 처리한다.
- 피드와 공유 리포트에서 공개 프로필을 만난다.
- 1:1 비교는 공개 프로필 팝업에서 시작한다.
- 내부 구현은 `publicSnapshotId` 같은 불투명 ID를 사용할 수 있지만, 사용자에게 입력시키거나 표시하지 않는다.

## 15. 정책, 관리자, 출시 게이트 요구사항

정책 화면 요구사항:

- 이용약관과 개인정보 처리방침 route를 제공한다.
- 최종 법률 문서가 승인되기 전에는 준비 문서 상태로 표시하고 noindex 처리한다.
- 정책 문서는 계정 저장, 공유 링크, 공개 비교, 피드 글쓰기 조건을 다루어야 한다.
- 개인정보 문서는 직접 응답, 원점수, 민감 주제, 도움 허브 맥락의 처리 기준을 다루어야 한다.

관리자/운영 준비 요구사항:

- 관리자 화면은 noindex다.
- 관리자 화면은 QA 게이트와 운영 준비 상태를 보여준다.
- 관리자 화면에서 직접 응답을 조회하면 안 된다.
- 공개 MVP 상태는 법률/개인정보/계정 서버/운영 QA 게이트가 열리기 전까지 NO-GO로 취급한다.
- moderation queue readiness, safety action contract 같은 운영 준비 항목을 확인할 수 있어야 한다.

출시 차단 조건:

- 고객 공개용 코어 문항·채점·코드·리포트 문구의 `measurement_release_id` 승인
- runtime 코드 체계와 승인된 `E/I · R/N · G/A · K/M · C/Q` release의 일치
- provisional 또는 legacy 결과의 공개 프로필·공유·피드·비교 전파 차단
- 최종 이용약관 승인
- 최종 개인정보 처리방침 승인
- OAuth provider 실제 연결 확인
- Supabase RLS와 권한 검증
- 공유 주소 noindex, 만료, 철회, 삭제 검증
- 공개 비교 접근 상태 재검증
- 피드 moderation 운영 기준
- 직접 응답/원점수/민감 항목 비노출 검증

## 16. 데이터 저장과 개인정보 요구사항

로컬 저장:

- 코어 검사 진행 중 attempt는 IndexedDB에 저장한다.
- 진행 중 코어 검사는 7일 동안 이어갈 수 있다.
- 완료 코어 결과는 30일 동안 다시 열 수 있다.
- 무료 주제 검사 결과는 로컬 우선으로 저장하고 서버 동기화 queue를 가질 수 있다.
- 별난 성향 연구소 결과는 로컬 저장을 기본으로 한다.
- 사용자는 로컬 데이터 내보내기와 삭제를 할 수 있어야 한다.

서버 저장:

- 계정 연결 후 결과 claim을 통해 assessment attempt, score snapshot, result report를 저장할 수 있다.
- 무료 주제 검사 결과는 summary/evidence-only 계약으로 저장한다.
- 피드 글, 댓글, 반응, 저장, 관심 없음, 투표는 계정 기반으로 저장한다.
- 공유 주소는 token hash, 만료, 상태를 저장한다.
- 공개 비교 리포트는 viewer 소유권과 target snapshot 상태를 재검증해야 한다.

절대 공개 금지:

- 직접 문항 응답
- 원점수와 전체 score vector
- OAuth token과 provider raw response
- 이메일과 계정 식별 정보
- 민감 검사 결과
- 도움 연결 이용 기록
- 성적 지향을 질병이나 치료 대상으로 다루는 정보
- 자살, 자해, 약물, 폭력, 정신질환 등을 재미형 검사나 피드 소재로 점수화한 정보

## 17. API 요구사항 요약

현재 구현 기준 주요 API 표면:

- `GET /api/account-results`: 인증 사용자 계정 결과와 비교 리포트 목록 조회
- `DELETE /api/account-results`: 결과 또는 비교에 연결된 데이터 삭제
- `GET /api/claim-result`: 로컬 결과의 계정 저장 상태 조회
- `POST /api/claim-result`: 로컬 결과를 계정 결과로 claim
- `GET /api/feed`: 피드 read payload 조회
- `POST /api/feed`: 피드 write action 처리
- `GET /api/free-topic-results`: 무료 주제 검사 서버 저장 결과 조회
- `POST /api/free-topic-results`: 무료 주제 검사 summary/evidence-only 저장
- `POST /api/profile-visibility`: 공개 범위 저장. 현재 저장 DB write가 완전히 열리기 전에는 closed-state를 반환할 수 있다.
- `POST /api/public-comparisons`: 공개 프로필 기반 1:1 비교 생성
- `POST /api/public-comparison-report`: 비교 리포트 조회
- `DELETE /api/public-comparison-report`: 비교 리포트 삭제
- `POST /api/share-links`: 공유 주소 생성
- `POST /api/revoke-share`: 공유 주소 철회

API 공통 요구사항:

- 모든 mutation은 zod schema 또는 동등한 구조 검증을 통과해야 한다.
- 인증이 필요한 mutation은 인증 실패 시 401을 반환해야 한다.
- Supabase 환경이 없으면 임의 성공처럼 보이지 않고 closed-state 응답을 반환해야 한다.
- 사용자-facing error는 다음 행동을 안내해야 한다.
- 클라이언트 payload에 공개 요약을 맡기지 말고 서버에서 projection을 생성해야 한다.
- 개인정보 금지 항목이 API response에 섞이지 않도록 테스트해야 한다.

## 20. 새 IA/메뉴 설계 시 사용 방법

새 AI 또는 개발자는 아래 순서로 메뉴와 하단 탭바를 설계한다.

1. 이 문서의 기능 도메인을 사용자 주요 행동으로 묶는다.
2. 각 묶음이 서로 다른 목적을 갖는지 확인한다.
3. 하단 탭바가 필요하면 4~5개 이하의 주요 도메인만 노출한다.
4. 나머지 기능은 해당 도메인 내부의 명확한 하위 화면이나 맥락 행동으로 둔다.
5. 시작 화면은 모든 탭으로 가는 카드 모음이 아니라, 사용자의 현재 상태에 따른 다음 행동을 돕는 역할로 설계한다.
6. 피드는 읽기와 반응의 체류 리듬을 우선한다.
7. 검사는 시작과 이어하기를 가장 쉽게 만든다.
8. 개인 영역은 내 코드, 리포트, 공개 범위, 데이터 권리를 안정적으로 다시 여는 역할을 맡긴다.
9. 공유와 비교는 독립 메뉴로 과하게 드러내기보다 결과, 피드, 공개 프로필 맥락에서 자연스럽게 시작되게 한다.
10. 어떤 구조에서도 개인정보 금지 항목과 비낙인 원칙을 약화하지 않는다.
