# NUANG Product Canon

작성일: 2026-07-09 KST

이 문서는 뉴앙 MVP 개발의 제품 기준 원본이다. 구현 단계에서 기억이나 대화 맥락이 아니라 이 문서를 기준으로 화면, 기능, 문구, 데이터 공개 범위를 판단한다.

## 2026-07-17 측정·제품 재기준화 오버라이드

[NUANG_MVP_MEASUREMENT_PRODUCT_REBASELINE.md](./NUANG_MVP_MEASUREMENT_PRODUCT_REBASELINE.md)가 고객 공개 MVP의 현재 실행 순서를 고정한다.

- 현재 runtime의 `S/T · C/V · F/O · D/A · P/E`와 `TVOAE` 예시는 구현된 legacy v0.1 감사 대상이다.
- 고객 공개 MVP의 뉴앙 코드 작업 범례는 `E/I · R/N · G/A · K/M · C/Q`다.
- 전 5영역 최소 검증과 scoring/copy release 전에는 새 코드·DB·공유 URL을 운영에 반영하지 않는다.
- legacy runtime은 내부 UX·회귀 테스트에는 사용할 수 있지만 고객에게 검증된 대표 성향처럼 공개하지 않는다.
- 홈·지도·공유·피드·비교의 최종 데이터와 문구는 승인된 `measurement_release_id`에만 연결한다.
- OE 전문가 검토 키트는 통합 M04의 실행 템플릿이며, OE 단독 결과로 전체 코드를 승인하지 않는다.

이 절은 아래의 legacy 구현 설명보다 우선한다.

## 고객-facing 검사 용어

고객 화면에서는 속도나 구현 구조보다 사용자가 지금 어떤 검사를 했고 무엇을 더 할 수 있는지 바로 이해하는 표현을 사용한다.

| 내부 명칭 | 고객 화면 명칭 |
| --- | --- |
| `QUICK`, quick assessment | 첫 성향 검사 |
| quick result | 첫 성향 결과 |
| provisional code | 예비 코드, 문맥에서는 방금 확인한 코드 |
| precision/full assessment | 정밀 성향 검사 |
| representative code | 대표 코드, 문맥에서는 내 대표 코드 |

- `빠른 결과`는 결과 생성 속도로 오해되므로 고객 화면에서 사용하지 않는다.
- 내부 DB·API·analytics의 `QUICK`, `quickAttemptId`는 기술 명칭으로 유지할 수 있다.
- 사용자-facing tag는 `첫 성향 결과 · 예비 코드`를 사용한다.
- 문맥이 분명하면 `방금 확인한 성향`, `방금 본 검사 결과`처럼 더 자연스러운 문장을 우선한다.

## 서비스 정의

뉴앙은 검사가 중심인 성향 기반 SNS다. 사용자는 빠르게 성향을 확인하고, 성향지도와 결과 리포트를 통해 자신을 이해하며, 공개 범위 안에서 다른 사람과 비교하거나 성향 기반 피드를 볼 수 있다.

서비스의 핵심 감정은 다음 순서로 만든다.

1. 쉽게 시작한다.
2. 검사 과정이 가볍고 재미있다.
3. 결과가 예쁘고 믿을 만하다.
4. 내 성향이 생활과 관계에서 어떻게 보이는지 이해한다.
5. 공개 범위가 분명해서 불안하지 않다.
6. 다른 사람과 비교하거나 피드를 볼 때도 진단, 순위, 낙인처럼 보이지 않는다.

서비스의 체류 경험은 `/feed`에서 확정한 인스타그램/스레드형 모바일 SNS 리듬을 기준으로 한다. 사용자는 설명을 읽는 앱이 아니라, 자연스럽게 스크롤하고 반응하고 다음 콘텐츠로 이어지는 앱으로 느껴야 한다. 단, 체류 시간을 늘리기 위해 민감 정보 보호, 비교 안전선, 진단/치료 오인 방지 원칙을 희석하지 않는다.

## MVP 하단 메뉴 계약

하단 메뉴는 MVP의 최상위 정보 구조다. 각 메뉴는 서로 다른 목적을 가져야 하며 같은 이동 경로를 반복해서 보여주지 않는다.

| 메뉴     | 역할                                                      | 포함                                                                | 제외                                         |
| -------- | --------------------------------------------------------- | ------------------------------------------------------------------- | -------------------------------------------- |
| 홈       | 오늘 시작점, 핵심 진행 상태, 피드 미리보기, 다음 행동 1개 | 오늘의 질문, 내 성향 요약, 검사 이어하기, 피드 preview              | 전체 피드, 모든 탭 이동 카드                 |
| 피드     | 성향 기반 SNS 체류 화면                                   | 추천순 피드, 오늘의 질문, 성향 기반 게시글, 반응과 댓글             | 검사 진행, 리포트 관리, 설정                 |
| 검사     | 코어 검사, 무료 주제 검사, 별난 성향 연구소 진입          | 빠른 코어, 정밀 코어, 무료 주제 검사, 재미형 연구소, 도움 허브 연결 | 피드, 1:1 비교 결과, 그룹 리포트 목록        |
| 성향지도 | 뉴앙 코드와 32개 성향을 탐색하는 정보 화면                | 32개 뉴앙 코드 탐색, 코드 이름, motif, 내 코드 연결                 | 검사 목록 반복, 피드, 커뮤니티               |
| 마이     | 내 뉴앙 코드, 리포트, 설정                                | 뉴앙 코드, 코드 이름, 캐릭터, 내 리포트, 설정                       | 피드 탐색, 검사 홈 복제, 공유 링크 관리 메뉴 |

## 결과 경험 계약

사용자에게 결과는 저장 위치와 무관하게 하나의 `내 결과`다.

- 화면 제목은 `결과 리포트`, 목록 제목은 `내 결과`를 사용한다.
- `계정 결과`, `로컬 결과`, `기기 사본`, `계정 저장`처럼 구현 위치를 드러내는 명칭을 사용자 화면에 표시하지 않는다.
- 로그인된 사용자의 결과 연결과 복원은 서비스가 백그라운드에서 처리한다.
- 사용자는 결과마다 하나의 `삭제`만 선택한다.
- 삭제하면 해당 결과, 공유 링크, 공개 프로필 스냅샷, 그 결과를 참조하는 비교 기록을 함께 정리한다.
- 데이터 내려받기는 `내 데이터 내려받기`로 표현하며 저장 위치를 설명하지 않는다.
- 개인정보 처리방침처럼 법적 고지가 필요한 문서에는 실제 처리·보관 방식을 정확히 밝힌다.

## 결과 공유 계약

공유할 공개 요약이 있는 모든 완료 리포트는 상단의 단일 `공유` 진입을 제공한다. 공유를 누르면 카카오톡, 링크 복사, 다른 앱으로, 피드에 공유를 고르는 action sheet가 열린다. 응답 근거가 부족해 코드와 공개 요약이 아직 없는 상태에는 공유를 노출하지 않는다. 공유는 1:1 비교 입력 링크가 아니다.

카카오톡, 링크 복사, 다른 앱으로는 모두 `내 결과 리포트`를 다른 사람이 열 수 있는 동일한 30일 자동 만료 공개 리포트 URL을 사용한다.

피드에 공유된 리포트 preview와 연결은 일반 링크 복사 공유와 수명이 다르다. 피드 공유는 작성자가 해당 피드 글을 삭제하기 전까지 유지한다.

사용자 흐름:

1. 사용자가 자신의 결과 리포트에서 공유를 누른다.
2. 서비스가 공개될 예비 코드·요약과 제외되는 개인정보를 먼저 보여준다.
3. 사용자가 카카오톡, 링크 복사, 다른 앱으로, 피드에 공유 중 목적에 맞는 방법을 고른다.
4. 외부 전송 방법은 같은 30일 공개 리포트 URL을 만들거나 기존 유효 URL을 재사용한다.
5. 받은 사람은 웹뷰 또는 브라우저에서 공개 가능한 요약만 담긴 공유용 결과 리포트를 연다.
6. 받은 사람은 `나도 같은 검사 해보기`로 같은 검사에 진입할 수 있다.

공유용 결과 리포트 원칙:

- 공유 링크 화면은 상대의 결과를 보여주는 공개 리포트다.
- 직접 응답, 원점수, OAuth 정보, 이메일, 민감 검사 결과는 공유 링크에 포함하지 않는다.
- 첫 성향 결과 공유에는 `예비 결과` 표시를 유지하고, 본인 전용 세부 신호와 과정 코드는 제외한다.
- 공유 화면은 noindex 대상이다.
- 공유 화면에서 1:1 비교를 바로 생성하지 않는다. 비교는 상대 프로필 팝업의 명시적 `나와 비교하기` 행동에서만 시작한다.
- 공유 주소는 30일 뒤 자동 만료한다.
- 사용자가 별도 `공유 링크 관리` 메뉴를 이해하거나 관리할 필요가 없게 한다.
- MVP에서는 공유 주소 철회 UI를 노출하지 않는다. 운영상 필요하면 리포트 상세의 보조 메뉴로만 재검토한다.
- 로컬 결과에서 인증이 필요하면 사용자가 고른 `pending_share_method`와 `return_to_result_id`를 보존하고 로그인 뒤 같은 공유 흐름을 이어간다.
- 클립보드·기기 공유·앱 전환 실패는 생성된 주소 표시와 링크 복사 대안으로 복구한다.

## 피드 정의

피드는 커뮤니티의 별도 병기 명칭이 아니다. 사용자-facing 문구에서는 `피드`만 사용한다.
피드 MVP 상호작용 설계의 원본은 `docs/NUANG_FEED_MVP_INTERACTION_DESIGN.md`다.

MVP 방향:

- 선택안: B안.
- `/feed`는 별도 페이지로 둔다.
- 하단 메뉴에 넣는다.
- 홈에는 피드 섹션 preview만 둔다.
- 홈에는 피드 preview를 두고, 전체 탐색은 하단 `피드` 탭에서 한다.

초기 피드 내용:

- 글
- 검사 리포트 공유 preview
- 오늘의 질문 기반 글쓰기
- 밸런스 게임 기반 투표 글
- 내 뉴앙 코드 기반 짧은 글
- 공식 주제 카드

피드 MVP 핵심:

- 피드는 기본 소셜 글쓰기, 반응, 댓글, 저장을 제공한다.
- 사용자는 검사 리포트를 피드에 공유할 수 있고, 피드에는 핵심 정보 preview만 노출한다.
- 리포트 preview를 누르면 상세 리포트로 이동한다.
- 피드 작성 화면 상단에는 `글`, `오늘의 질문`, `밸런스 게임`만 둔다. 리포트 공유는 피드 작성기에서 고르지 않고 각 리포트 화면의 `피드에 공유`에서 시작한다.
- `오늘의 질문`은 사용자가 직접 분류를 입력하는 기능이 아니라, 서비스가 쉬운 질문을 제안하고 사용자가 답하는 방식이다.
- `밸런스 게임`은 질문과 선택지를 제공하고, 사용자가 답을 선택해 업로드한다.
- 다른 사용자는 밸런스 게임 피드에서 답을 선택할 수 있고, 선택 후 전체 비율을 본다.
- `뉴앙 코드별 통계 보기`에서는 코드별 선택 비율을 보여줄 수 있다.
- 코드별 통계에서는 한 명 이상 투표한 코드는 비율을 보여주고, 0명인 코드만 숨긴다.

궁극 방향:

- 내 성향과 공개 범위, 상호작용 기록을 기준으로 추천순 노출한다.
- 뉴앙 코드별 피드 반응과 투표 통계를 활용해 더 재미있는 성향 탐구 경험을 만든다.
- 피드가 비교 기능이나 그룹 검사 기능을 침범하지 않는다.
- 피드는 뉴앙의 체류 경험 기준 화면이다. 다른 탭의 UI도 피드에서 검증한 여백, 타이포, 경계선, 즉시 반응 가능한 액션 리듬을 우선 참고한다.

금지:

- `피드와 커뮤니티`처럼 둘을 같은 레벨로 병기하지 않는다.
- `함께` 탭을 피드 탐색 장소로 쓰지 않는다.
- 피드에서 진단, 치료, 위험 인물 판정, 궁합 점수, 관계 성공 예측을 제공하지 않는다.

## 비교와 그룹 정의

`함께` 탭은 제거한다. 1:1 비교는 별도 탭에서 시작하지 않고, 피드, 공유 리포트, 프로필 화면 등에서 상대 프로필을 클릭해 열린 팝업에서 `나와 비교하기`를 누를 때 생성한다.

MVP 방향:

- 1:1 비교는 링크를 직접 입력해서 시작하지 않는다.
- 1:1 비교는 상대의 매번 동의 요청이 아니라, 상대가 미리 설정한 공개 범위 안에서 바로 비교한다.
- MVP의 핵심 비교 리포트는 서로의 현재 뉴앙 코드를 기준으로 상세하게 제공한다.
- 현재 뉴앙 코드는 정밀 코어와 누적된 승인 검사 증거를 바탕으로 계산한 대표 코드다.
- 상세검사나 유료검사 기반 추가 비교는 추후 제공하되, 두 사람이 모두 같은 검사를 수행하고 비교 사용을 허용한 경우에만 제공한다.
- MVP에서는 유료 상세검사 기반 비교를 열지 않는다.
- 민감 항목은 기본 비공개이며 비교에 사용하지 않는다.
- 생성된 1:1 비교 리포트는 `마이 > 내 리포트`에서 다시 확인한다.
- 그룹 만들기는 추후 `검사` 탭에서 열 유료 기능이다.
- 그룹 리포트도 생성 후 `마이 > 내 리포트`에서 확인한다.
- 그룹 초대, 결제, 참여 권한, 리포트 범위는 해당 기능 개발 단계에서 별도 승인 후 설계한다.

상대 프로필 팝업 구조:

1. 상대 프로필 이미지, 표시 이름과 대표 성향
2. 공개된 뉴앙 코드 요약
3. 공개 범위 안내
4. `나와 비교하기`
5. 닫기

프로필 이미지는 사용자가 설정한 공개 프로필 이미지다. 캐릭터 이미지나 임시 성향 이미지를 쓰는 경우에도 별도 장식 이미지가 아니라 프로필 이미지로 노출한다.

`나와 비교하기`를 누른 뒤:

1. 내 정밀 코어 결과가 있는지 확인한다.
2. 상대 프로필의 현재 공개 범위를 다시 확인한다.
3. 서로 공개된 항목만 비교한다.
4. 비교 리포트를 생성한다.
5. 생성된 리포트로 이동한다.

금지:

- 피드 preview, 일반 게시글, 커뮤니티 글쓰기 composer를 함께 탭에 넣지 않는다.
- 상대를 점수화하거나 순위화하지 않는다.
- 상대의 비공개 항목을 추정하지 않는다.
- 사용자가 링크나 코드를 직접 입력해서 1:1 비교를 시작하게 하지 않는다.

## 공개 코드 폐기 원칙

공개 코드는 사용자 경험에서 전면 폐기한다. 사용자는 어떤 화면에서도 공개 코드를 발급받거나, 복사하거나, 입력하거나, 코드 형식을 이해할 필요가 없다.

폐기 이유:

- 사용자가 코드를 입력해야 하는 흐름은 번거롭고 재미가 없다.
- `공개 코드`, `성향 코드`, `공유 링크`가 서로 혼동된다.
- 피드, 공유 리포트, 프로필 팝업만으로 자연스럽게 상대 프로필을 만날 수 있다.
- 코드 입력 UI는 성향 기반 SNS의 가벼운 체류 경험과 맞지 않는다.

공개 코드 없는 대안 구조:

- 결과 공유는 공유 링크 token으로 처리한다.
- 공유받은 사람은 공유용 결과 리포트를 열고, 원하면 같은 검사를 시작한다.
- 1:1 비교는 피드, 공유 리포트, 프로필 화면에서 상대 프로필을 클릭해 열린 팝업에서 시작한다.
- 앱 내부 구현은 사용자가 클릭한 공개 프로필의 `publicSnapshotId` 같은 불투명 ID를 사용한다.
- 불투명 ID는 화면에 표시하지 않고, 사용자가 직접 입력하는 대상도 아니다.

금지:

- 공개 코드 발급 UI
- 공개 코드 입력 UI
- `/p/[code]`처럼 코드를 주인공으로 하는 사용자 route
- `publicProfileCode`를 받는 비교 생성 API
- 공개 코드를 공유 링크나 프로필 링크처럼 설명하는 문구

DB 정리:

- 기존 개발 DB에 이미 공개 코드 테이블이나 컬럼이 적용되어 있을 수 있으므로, 실제 drop SQL 실행은 별도 승인 후 진행한다.
- 새 코드와 새 UX는 공개 코드 테이블에 의존하지 않는다.

## 구현 전 승인 계약

새 기능을 구현하기 전에는 먼저 사용자에게 아래 항목을 정리해 승인받는다.

1. 기획 의도
2. 사용자 경험 흐름
3. 화면 구조
4. API와 DB 구현 흐름
5. 삭제하거나 닫을 기존 기능
6. 개인정보와 공개 범위 영향
7. 테스트와 smoke 범위

사용자가 승인하기 전에는 해당 기능의 구현, DB write 오픈, 화면 노출을 진행하지 않는다.

## 검사와 안전 영역

뉴앙의 일반 검사는 성향 이해와 관계 커뮤니케이션을 돕는 서비스다. 치료, 진단, 병명 판정, 위기 평가를 대신하지 않는다.

검사 제품 구조의 원본은 `docs/NUANG_ASSESSMENT_ARCHITECTURE.md`다.
누적 검사 기반 성향 갱신 알고리즘의 원본은 `docs/NUANG_DYNAMIC_TRAIT_EVIDENCE_MODEL.md`다.

검사 원칙:

- 정밀 코어는 MVP뿐 아니라 앞으로도 무료로 운영한다.
- 정밀 코어는 공식 대표 성향 코드, 성향지도, 공개 프로필, 1:1 비교의 기준이다.
- 빠른 코어는 무료 예비 검사이며 1:1 비교 기준으로 쓰지 않는다.
- 무료 주제 검사는 체류와 재미, 피드 소재를 만드는 핵심 콘텐츠다.
- 무료 주제 검사는 대표 성향 코드를 즉시 흔들지 않지만, 승인된 축·세부 성향 매핑과 충분한 유효 가중치가 쌓이면 현재 대표 코드를 갱신할 수 있다.
- 무료 주제 검사는 사용자가 검사를 더 할수록 세부 해석, 현재 대표 성향, 추천 품질을 더 구체화한다.
- 결제 여부는 성향 정확도 가중치가 아니다. 무료와 유료는 같은 기준을 사용하며, 가중치는 문항 수, 구성개념 직접성, 응답 품질, 승인 상태, 최근성, 반복 할인으로 결정한다.
- 별난 성향 연구소는 무료 주제 검사 중 유별나고 가벼운 실험 영역이다.
- 상세검사는 후속 확장 영역이며 결제, 리포트 범위, 노출 방식은 별도 승인 후 결정한다.
- 그룹 검사는 이번 단계에서 설계하지 않고, 추후 별도 기획한다.

허용:

- 빠른 코어
- 정밀 코어
- 무료 주제 검사
- 재미형 별난 성향 연구소
- 생활형 자기 이해 문항
- 도움 허브와 공식 자원 연결
- 민감 주제에 대한 정보형 안내

제한:

- 자살, 자해, 우울, ADHD, 중독, 트라우마, 폭력, 약물 등은 재미형 검사로 열지 않는다.
- 임상 선별 도구를 쓰려면 출처, 목적, 한계, 긴급 안내, 비저장 기본값, 전문가 상담 권고를 별도 게이트로 둔다.
- 성적 지향은 치료 대상처럼 다루지 않는다.

## 캐릭터와 뉴앙 코드

뉴앙은 32개 성향 프로필을 가진다. 5종 캐릭터는 32개 성향을 줄이는 것이 아니라, 32개 프로필을 시각적으로 묶는 motif system이다.
구현된 legacy v0.1의 감사 원본은 `docs/NUANG_CODE_INTERPRETATION_DICTIONARY.md`다. 고객 공개 코드의 설계 원본은 `docs/NUANG_FINAL_CODE_SEMANTIC_AUDIT.md`와 `docs/NUANG_CORE_MEASUREMENT_VALIDATION_PLAN.md`다.

뉴앙 코드 원칙:

- 뉴앙 코드는 5글자 대표 성향 코드다. 고객 공개 MVP 작업 범례는 `E/I · R/N · G/A · K/M · C/Q`이며, `TVOAE`는 legacy v0.1 fixture다.
- 뉴앙 코드는 사용자가 기억하고 소통할 수 있는 핵심 단위로 강조한다.
- 성향 이름은 뉴앙 코드 아래에 바로 표시한다. 예: `불꽃의 온기 탐험가`.
- 코드 이름은 `모티프의 이름` 형태를 기본으로 한다. 예: `물결의 새길 개척가`, `숲의 편안한 조율가`.
- `32개 성향 중` 같은 수식어는 사용자 화면에 붙이지 않는다.
- 코드와 이름은 마이, 결과 리포트, 공유 리포트, 프로필 팝업에서 일관되게 표시한다.
- 사용자 화면에 노출되는 영어 알파벳 표현은 뉴앙 코드뿐이다. `RO-EC`, `SE-AI` 같은 내부 채점 키는 리포트, 피드, 프로필, 설정 화면에 노출하지 않는다.
- 사용자 화면에서는 `5축`, `10축`이라는 표현을 쓰지 않는다. 오각형 그래프나 세부 막대가 필요할 때도 `코드 지도`, `코드 자리`, `세부 신호`로 설명한다.
- 무료 주제 검사와 세부 리포트는 내부 키 대신 `상대 마음 살피기`, `먼저 말 꺼내기`처럼 누구나 이해할 수 있는 한국어 신호명과 해석 문구를 사용한다.

현재 motif:

- 불꽃
- 햇살
- 물결
- 숲
- 별빛 또는 후속 확정 motif

캐릭터 적용 원칙:

- 5종은 대표 motif와 서비스 정체성에 사용한다.
- 32개 프로필은 motif, 표정, 자세, 소품, 색상 accent로 확장한다.
- 캐릭터 이미지는 필요한 경우 생성 이미지를 사용하고, CSS로 동물/캐릭터 형태를 직접 그리지 않는다.

## 인증과 연령

소셜 로그인은 Google, Kakao, Naver를 목표로 한다. OAuth에서 생년월일을 안정적으로 가져온다고 가정하지 않는다.

원칙:

- 뉴앙의 일반 진입과 로그인은 전연령 이용 가능 흐름으로 둔다.
- 일반 로그인에서 만 14세 이상 체크와 연령대 선택을 요구하지 않는다.
- 19세 이상 전용 기능이 생길 때만 해당 기능 진입 직전에 성인 인증을 요구한다.
- MVP에는 19세 이상 전용 기능을 열지 않는다.
- 생년월일은 provider별 제공 가능성과 사용자 동의가 다르다.
- 통계에는 가능하면 정확한 생년월일보다 별도 동의 기반의 선택형 연령대 값을 우선 사용한다.
- 진입 UX를 복잡하게 만들지 않되, 필수 동의는 명확하게 분리한다.

## 문구 원칙

- 사용자에게 개발 용어를 보이지 않는다.
- `커뮤니티`와 `피드`를 중복 표기하지 않는다.
- `공개`가 검색 노출처럼 들릴 때는 noindex와 공개 범위를 함께 설명한다.
- 결과 문구는 낙인, 진단, 순위, 궁합 점수처럼 읽히지 않게 쓴다.
